# HashMap 底层原理

## 一、先说结论

JDK 8 之后 HashMap 底层结构是：

```text
数组 + 链表 + 红黑树
```

核心流程：

```text
根据 key 的 hash 定位数组下标
-> 如果桶为空，直接插入
-> 如果桶不为空，比较 key
-> 相同则覆盖
-> 不同则插入链表或红黑树
-> 达到阈值后扩容
```

HashMap 不是线程安全的，并发场景应使用 `ConcurrentHashMap`。

## 二、核心字段

重要概念：

- table：数组。
- capacity：容量，始终是 2 的幂。
- loadFactor：负载因子，默认 0.75。
- threshold：扩容阈值，capacity * loadFactor。
- size：键值对数量。

默认负载因子 0.75 是空间和时间的折中。

## 三、为什么容量是 2 的幂

定位下标：

```java
(n - 1) & hash
```

当 n 是 2 的幂时，`n - 1` 的二进制低位全是 1，可以高效取模。

相比 `%`：

- 位运算更快。
- 扩容迁移更简单。

## 四、hash 扰动

HashMap 会对 key 的 hashCode 做扰动：

```java
h ^ (h >>> 16)
```

目的：

```text
让 hashCode 的高位也参与低位下标计算，减少碰撞。
```

因为数组下标主要由 hash 低位决定。

## 五、put 流程

简化流程：

```text
1. 如果 table 为空，先初始化。
2. 计算 key 的 hash。
3. 根据 (n - 1) & hash 找桶下标。
4. 桶为空，直接放新节点。
5. 桶不为空，先比较第一个节点。
6. key 相同，覆盖 value。
7. key 不同，如果是树节点，走红黑树插入。
8. 否则遍历链表，找到相同 key 覆盖，找不到就尾插。
9. 链表长度达到阈值，尝试树化。
10. size 超过 threshold，扩容。
```

## 六、链表什么时候转红黑树

条件：

```text
链表长度 >= 8
数组容量 >= 64
```

如果链表长度达到 8 但数组容量小于 64，优先扩容而不是树化。

原因：

```text
容量小导致的冲突，扩容更划算；容量已经足够大还冲突严重，才树化。
```

退化条件：

```text
红黑树节点数量太少时会退化回链表。
```

## 七、扩容机制

HashMap 每次扩容容量翻倍。

JDK 8 优化点：

```text
节点扩容后要么留在原位置，要么移动到 原位置 + oldCap。
```

判断依据：

```java
(hash & oldCap) == 0
```

好处：

- 不需要重新完整计算 hash。
- 链表拆分更高效。

## 八、为什么 HashMap 线程不安全

并发问题：

- put 时数据覆盖。
- size 统计不准。
- 扩容时数据丢失。
- 遍历时结构被修改抛 ConcurrentModificationException。

JDK 7 中并发扩容还可能产生链表环，JDK 8 改为尾插降低了这个问题，但 HashMap 仍然不是线程安全容器。

## 九、HashMap、Hashtable、ConcurrentHashMap

| 容器 | 线程安全 | 特点 |
|---|---|---|
| HashMap | 否 | 性能高，单线程使用 |
| Hashtable | 是 | 方法级 synchronized，老旧 |
| ConcurrentHashMap | 是 | 并发性能好，推荐并发场景 |

## 十、常见面试追问

### 1. 为什么链表长度到 8 才树化

这是基于概率和性能的折中。正常 hash 分布下链表长度达到 8 的概率很低。红黑树节点更占内存，链表短时遍历更快。

### 2. 为什么数组容量小于 64 时不树化

此时冲突更可能来自容量太小，扩容能更有效降低冲突；树化成本更高。

### 3. HashMap 可以存 null 吗

可以。HashMap 允许一个 null key，多个 null value。null key 通常放在 0 号桶。

## 十一、生产经验

- 已知数据量时指定初始容量，减少扩容。
- key 必须正确实现 equals 和 hashCode。
- 不要使用可变对象作为 key。
- 并发场景不要用 HashMap。
- 高频大 Map 要关注内存占用和扩容成本。
