# 微服务治理核心问题

## 一、先说结论

微服务治理解决的是服务拆分后的复杂性：

```text
服务怎么发现、配置怎么管理、调用怎么稳定、故障怎么隔离、链路怎么观测、版本怎么发布。
```

资深开发不能只会写 Controller 和 Feign，要能解释一次微服务调用背后的完整链路。

## 二、一次微服务调用链路

典型链路：

```text
客户端
-> 网关
-> 鉴权 / 限流 / 路由
-> 服务 A
-> 注册中心发现服务 B
-> 负载均衡选择实例
-> RPC/HTTP 调用服务 B
-> 超时 / 重试 / 熔断
-> 链路追踪记录
-> 指标和日志上报
```

## 三、注册中心

注册中心保存：

- 服务名。
- 实例 IP 和端口。
- 健康状态。
- 元数据。

常见组件：

- Nacos。
- Eureka。
- Zookeeper。
- Consul。

核心机制：

```text
服务启动时注册。
服务定期心跳。
注册中心剔除异常实例。
消费者拉取或订阅服务列表。
```

## 四、配置中心

配置中心管理：

- 数据库连接配置。
- 开关配置。
- 限流阈值。
- 线程池参数。
- 灰度配置。

关键能力：

- 动态刷新。
- 环境隔离。
- 权限控制。
- 配置版本。
- 回滚。
- 监听推送。

生产建议：

```text
动态配置变更必须有审计、灰度和回滚，不能随便全量推送。
```

## 五、网关

网关常见职责：

- 路由。
- 鉴权。
- 限流。
- 跨域。
- 协议转换。
- 灰度发布。
- 统一日志。

不要把复杂业务逻辑写进网关。

网关应该处理横切能力，不应该成为业务大泥球。

## 六、负载均衡

常见策略：

- 轮询。
- 随机。
- 权重。
- 最少连接。
- 一致性哈希。
- 基于响应时间。

生产中要考虑：

- 实例预热。
- 异常实例摘除。
- 同机房优先。
- 灰度权重。

## 七、超时和重试

### 1. 超时

每个远程调用都必须设置超时。

没有超时的远程调用是线上事故高发点。

### 2. 重试

重试可以提升成功率，但也可能放大流量。

重试必须满足：

- 接口幂等。
- 次数有限。
- 有退避策略。
- 不对长耗时写接口盲目重试。

面试表达：

```text
超时是止损，重试是补偿，但重试必须建立在幂等基础上。
```

## 八、熔断降级

当下游异常时，调用方要能快速失败或返回兜底结果。

否则会出现：

```text
下游慢 -> 调用线程堆积 -> 上游线程池打满 -> 整条链路雪崩
```

常见保护：

- 熔断。
- 限流。
- 线程池隔离。
- 舱壁隔离。
- 降级返回。

## 九、链路追踪

链路追踪解决：

```text
一个请求经过多个服务后，如何定位慢在哪里、错在哪里。
```

核心概念：

- traceId：整条请求链路 ID。
- spanId：链路中的一次调用。
- baggage：跨服务透传上下文。

常见系统：

- SkyWalking。
- Zipkin。
- Jaeger。
- OpenTelemetry。

生产建议：

- 日志必须打印 traceId。
- 网关生成或透传 traceId。
- MQ 异步链路也要传递 trace 信息。

## 十、灰度发布

灰度常见维度：

- 用户 ID。
- 租户。
- 地域。
- 请求头。
- 版本号。
- 流量比例。

关键能力：

- 小流量验证。
- 快速回滚。
- 指标观测。
- 新老版本兼容。

注意：

```text
灰度不仅是流量路由，还包括数据库结构、消息格式、缓存 key、接口协议的兼容。
```

## 十一、服务版本兼容

微服务发布不是同时完成的，所以接口要兼容一段时间。

原则：

- 只加字段，不随意删字段。
- 新字段给默认值。
- 枚举扩展要兼容未知值。
- MQ 消息体要考虑新老消费者。
- 数据库变更遵循 expand -> migrate -> contract。

## 十二、生产经验

- 服务拆分前先明确边界，不要为了微服务而微服务。
- 每个远程调用必须有超时、限流、熔断策略。
- 重试必须配合幂等。
- 网关只做通用能力，不写复杂业务。
- 链路追踪、指标、日志必须同时建设。
- 配置中心变更必须可审计、可灰度、可回滚。

## 十三、常见面试追问

### 1. 注册中心挂了服务还能调用吗

如果消费者本地有服务列表缓存，短时间内通常还能调用已有实例。但新实例发现、故障实例摘除、配置变更可能受影响。

### 2. 重试为什么可能导致雪崩

下游已经慢或故障时，上游大量重试会放大请求量，让下游更慢，最终拖垮整条链路。

### 3. 微服务拆分最重要的是什么

业务边界和数据边界。技术组件能解决调用问题，但解决不了错误拆分带来的复杂度。
