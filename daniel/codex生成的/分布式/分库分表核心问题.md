# 分库分表核心问题

## 一、先说结论

分库分表是为了解决单库单表在数据量、并发、存储、维护上的瓶颈，但它会把复杂度从数据库转移到应用和中间件。

核心问题不是“怎么分”，而是：

```text
分片键怎么选、跨分片怎么查、事务怎么处理、扩容怎么迁移、全局 ID 怎么生成。
```

## 二、什么时候需要分库分表

常见信号：

- 单表数据量过大，索引层级变高，查询变慢。
- 单库连接数、CPU、IO 打满。
- 写入压力太大。
- 历史数据和热点数据混在一起。
- DDL、备份、归档成本过高。

不要过早分库分表。优先尝试：

- 索引优化。
- SQL 优化。
- 读写分离。
- 缓存。
- 归档冷热数据。
- 垂直拆分。

## 三、垂直拆分和水平拆分

### 1. 垂直拆分

按业务边界拆：

```text
用户库、订单库、支付库、库存库
```

优点：

- 业务边界清晰。
- 降低单库复杂度。

缺点：

- 跨库 join 消失。
- 分布式事务问题出现。

### 2. 水平拆分

同一张表按规则拆成多张表或多个库。

例如订单表：

```text
order_00
order_01
...
order_63
```

优点：

- 分摊单表数据量和写压力。

缺点：

- 路由、查询、分页、扩容复杂。

## 四、分片键如何选择

好的分片键应该满足：

- 查询高频。
- 区分度高。
- 数据分布均匀。
- 业务稳定，不轻易变化。
- 能减少跨分片查询。

常见分片键：

- user_id。
- order_id。
- tenant_id。
- merchant_id。

订单系统常见取舍：

```text
按 user_id 分片，用户查订单方便，但商家查订单可能跨片。
按 order_id 分片，订单详情方便，但用户订单列表可能跨片。
```

很多系统会配合异构索引表：

```text
user_order_index(user_id, order_id, shard_no)
```

## 五、分片算法

### 1. 取模

```text
shard = user_id % N
```

优点：

- 简单。
- 分布较均匀。

缺点：

- 扩容迁移成本高。

### 2. 范围分片

```text
user_id 1-1000w -> shard_1
user_id 1000w-2000w -> shard_2
```

优点：

- 易归档。
- 范围查询方便。

缺点：

- 容易热点。

### 3. 一致性哈希

减少扩容迁移量，但实现和运维复杂。

### 4. 基因法

把分片信息编码进 ID，例如订单 ID 里带 user_id hash 后的分片基因。

适合：

- 通过订单 ID 反推出分片位置。

## 六、分布式 ID

分库分表后不能依赖单表自增主键。

常见方案：

### 1. 雪花算法

结构大致：

```text
时间戳 + 机器 ID + 序列号
```

优点：

- 趋势递增。
- 本地生成，性能高。

缺点：

- 依赖时钟。
- 机器 ID 管理复杂。

### 2. 号段模式

从数据库批量获取一段 ID，本地递增使用。

优点：

- 性能好。
- 趋势递增。
- 对时钟不敏感。

缺点：

- 依赖发号中心。

### 3. Redis INCR

简单高性能。

缺点：

- Redis 可用性和持久化风险要处理。

## 七、跨分片查询问题

### 1. 能带分片键

最理想，直接路由到单分片。

```sql
select * from order where user_id = ? and order_id = ?;
```

### 2. 不带分片键

可能需要广播查询所有分片，再聚合结果。

风险：

- 慢。
- 放大数据库压力。
- 分页排序复杂。

解决：

- 建异构索引表。
- 搜索引擎。
- 数据仓库。
- 限制查询条件。
- 后台任务异步查询。

## 八、分页和排序问题

跨分片分页不能简单对每个分片 `limit offset, size`。

因为全局排序需要合并多个分片结果。

常见方案：

- 禁止深分页。
- 基于游标翻页。
- 先查各分片 topN，再在应用层归并。
- 使用 ES 等搜索系统。
- 构建宽表或索引表。

## 九、分布式事务问题

分库分表后，一次业务可能写多个库。

解决思路：

- 尽量让同一业务聚合落在同一分片。
- 使用最终一致方案。
- TCC / Saga / 本地消息表。
- 避免跨分片强事务。

## 十、扩容迁移

常见方案：

### 1. 停机迁移

简单但影响业务。

### 2. 双写迁移

流程：

```text
1. 新老库双写。
2. 全量迁移历史数据。
3. 增量校验补偿。
4. 读流量灰度切新库。
5. 停止老库写入。
```

难点：

- 双写一致性。
- 数据校验。
- 回滚方案。

### 3. 使用中间件

例如 ShardingSphere 等，降低应用层复杂度，但仍要理解底层问题。

## 十一、生产经验

- 分库分表前先确认增长趋势和瓶颈来源。
- 分片键选择比中间件选择更重要。
- 分片数量要预估未来容量，不要频繁变更。
- 查询接口必须围绕分片键设计。
- 后台管理类查询不要影响核心交易库。
- 分库分表后必须建设数据校验、补偿、迁移工具。

## 十二、常见面试追问

### 1. 分库分表后如何 join

尽量避免跨库 join。通过字段冗余、宽表、异构索引、应用层组装、搜索引擎等方式解决。

### 2. 分片键选错了怎么办

轻则建索引表和异构查询链路，重则重新分片迁移。分片键选错是高成本架构错误。

### 3. 如何避免热点分片

选择分布均匀的分片键，避免按时间范围直接分热点数据，必要时引入 hash、热点拆分或单独热点表。
