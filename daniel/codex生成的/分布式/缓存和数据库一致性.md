# 缓存和数据库一致性

## 一、先说结论

缓存和数据库很难做到强一致，大多数互联网业务追求的是最终一致。

最常见模式是 Cache Aside Pattern：

```text
读：先读缓存，缓存没有再读数据库，回填缓存。
写：先更新数据库，再删除缓存。
```

为什么通常不是更新缓存？

```text
因为缓存数据可能是复杂聚合结果，更新逻辑容易出错，删除缓存让下一次读重新加载更简单可靠。
```

## 二、读流程

```text
1. 查询缓存。
2. 命中，直接返回。
3. 未命中，查询数据库。
4. 数据库查到后写入缓存。
5. 返回结果。
```

注意：

- 缓存值要设置过期时间。
- 空值也可以短暂缓存，防穿透。
- 热点 key 要考虑并发回源。

## 三、写流程为什么推荐先更新数据库再删除缓存

推荐：

```text
更新数据库
-> 删除缓存
```

原因：

- 数据库是事实源。
- 删除缓存比更新缓存更稳。
- 下一次读会从数据库加载新值。

## 四、为什么不推荐先删除缓存再更新数据库

风险流程：

```text
1. 线程 A 删除缓存。
2. 线程 B 查询缓存未命中。
3. 线程 B 查询数据库，读到旧值。
4. 线程 B 把旧值写回缓存。
5. 线程 A 更新数据库为新值。
```

结果：

```text
缓存里长期是旧值，直到过期。
```

## 五、先更新数据库再删除缓存也有问题吗

有。

风险流程：

```text
1. 线程 A 更新数据库成功。
2. 删除缓存失败。
3. 缓存仍是旧值。
```

解决思路：

- 删除失败重试。
- 写入删除缓存任务表。
- MQ 异步重试。
- 监听 binlog 删除缓存。
- 缓存设置合理过期时间兜底。

## 六、延迟双删

流程：

```text
1. 删除缓存。
2. 更新数据库。
3. sleep 一小段时间。
4. 再删除缓存。
```

适用场景：

- 读写并发较高。
- 希望清理并发读期间写入的旧缓存。

局限：

- sleep 时间不好设置。
- 会阻塞业务线程。
- 不能保证绝对一致。
- 工程上不如可靠异步删除优雅。

## 七、binlog + Canal 方案

流程：

```text
1. 业务只更新数据库。
2. Canal 监听 MySQL binlog。
3. 解析变更事件。
4. 根据表和主键删除或更新缓存。
```

优点：

- 缓存失效和业务代码解耦。
- 适合多服务共享数据变更。
- 能统一处理缓存删除。

缺点：

- 链路更长。
- binlog 延迟会导致短暂不一致。
- Canal/MQ/消费者都要保证可用和可观测。

## 八、缓存穿透、击穿、雪崩

### 1. 缓存穿透

查询不存在的数据，缓存和数据库都没有，请求每次都打到数据库。

解决：

- 缓存空值，设置短 TTL。
- 布隆过滤器。
- 参数校验。

### 2. 缓存击穿

热点 key 过期，大量请求同时回源数据库。

解决：

- 热点 key 不过期或逻辑过期。
- 互斥锁重建缓存。
- 后台异步刷新。

### 3. 缓存雪崩

大量 key 同时过期，或 Redis 故障，请求集中打到数据库。

解决：

- TTL 加随机值。
- 多级缓存。
- 限流降级。
- Redis 高可用。
- 核心数据预热。

## 九、热点 key 处理

方案：

- 本地缓存 + Redis。
- 热点 key 副本拆分。
- 逻辑过期异步刷新。
- 请求合并。
- 限流保护。
- 提前预热。

逻辑过期示例：

```text
缓存永不过期，但 value 里存 expireTime。
读到过期数据时，先返回旧值，再异步刷新。
```

适合读多写少、允许短暂旧数据的场景。

## 十、生产经验

- 不要为了“强一致”滥用分布式锁包住缓存和数据库。
- 数据库必须是最终事实源。
- 缓存删除失败要有重试和告警。
- 关键数据可以短 TTL + 主动删除双保险。
- 聚合缓存最好有版本号或数据更新时间，便于排查旧数据。
- 多级缓存要考虑本地缓存失效通知。

## 十一、常见面试追问

### 1. 为什么删除缓存而不是更新缓存

更新缓存可能涉及复杂聚合逻辑，容易和数据库更新不一致。删除缓存让下一次读取从数据库重建，简单可靠。

### 2. 先更新数据库再删除缓存一定安全吗

不是。删除缓存失败仍会旧读，所以需要重试、MQ、binlog 监听和 TTL 兜底。

### 3. 如何做到强一致

严格强一致通常要牺牲性能和可用性，可以让读写都走数据库事务和锁，不使用缓存。但大多数高并发业务选择最终一致。
