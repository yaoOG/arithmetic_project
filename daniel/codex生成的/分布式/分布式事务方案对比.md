# 分布式事务方案对比

## 一、先说结论

分布式事务的本质是：

```text
一次业务操作涉及多个本地事务，如何保证它们最终一致。
```

没有一种方案适合所有场景。资深开发要能根据一致性要求、性能、复杂度和业务补偿能力做取舍。

常见方案：

- 2PC / XA。
- TCC。
- Saga。
- 本地消息表。
- 事务消息。
- 最大努力通知。

## 二、2PC / XA

两阶段提交：

```text
Prepare 阶段：协调者询问所有参与者能否提交。
Commit 阶段：所有参与者都准备成功后统一提交，否则回滚。
```

优点：

- 一致性强。
- 对业务侵入相对小。

缺点：

- 性能差。
- 锁持有时间长。
- 协调者单点问题。
- 不适合高并发互联网长链路业务。

适合：

- 强一致要求高。
- 参与方少。
- 调用链短。
- 性能要求不极端。

## 三、TCC

TCC 分为：

```text
Try：预留资源。
Confirm：确认提交。
Cancel：取消释放资源。
```

示例：扣库存

```text
Try：冻结库存。
Confirm：扣减冻结库存。
Cancel：释放冻结库存。
```

优点：

- 一致性较强。
- 业务可控。
- 性能比 XA 更适合互联网场景。

缺点：

- 业务侵入大。
- 每个参与方都要实现 Try/Confirm/Cancel。
- 要处理空回滚、幂等、悬挂。

### TCC 三个核心问题

#### 1. 空回滚

Cancel 到达时 Try 还没执行。

处理：

```text
Cancel 需要能识别 Try 是否执行过；没执行过也要返回成功。
```

#### 2. 幂等

Confirm/Cancel 可能重复调用。

处理：

```text
事务状态表 + 唯一事务号。
```

#### 3. 悬挂

Cancel 先于 Try 执行，之后 Try 又到达。

处理：

```text
Try 执行前检查事务状态，如果已经 Cancel，则拒绝 Try。
```

## 四、Saga

Saga 把长事务拆成多个本地事务，每个本地事务都有对应补偿动作。

流程：

```text
T1 -> T2 -> T3
如果 T3 失败：
C2 -> C1
```

优点：

- 适合长流程。
- 不长时间锁资源。
- 扩展性好。

缺点：

- 只能最终一致。
- 补偿逻辑复杂。
- 中间状态对用户可见。

适合：

- 订单履约。
- 旅游预订。
- 审批流程。
- 跨多个服务的长事务。

## 五、本地消息表

核心思想：

```text
业务数据和消息记录在同一个本地事务中提交。
后台任务可靠投递消息。
消费者幂等处理。
```

流程：

```text
1. 开启本地事务。
2. 写业务表。
3. 写消息表，状态为待发送。
4. 提交本地事务。
5. 后台任务扫描待发送消息。
6. 发送 MQ。
7. 发送成功后更新消息状态。
8. 消费者幂等消费。
```

优点：

- 简单可靠。
- 不依赖 XA。
- 适合最终一致。

缺点：

- 有延迟。
- 需要消息表扫描和补偿。
- 消费者必须幂等。

适合：

- 下单后发积分。
- 支付成功后通知履约。
- 订单状态同步。

## 六、事务消息

以 RocketMQ 为例，核心是：

```text
半消息 + 本地事务 + 二次确认 + 事务回查
```

流程：

```text
1. Producer 发送半消息。
2. Broker 存储半消息，但 Consumer 不可见。
3. Producer 执行本地事务。
4. 本地事务成功，提交消息；失败，回滚消息。
5. 如果 Producer 失联，Broker 发起事务回查。
```

适合：

- 本地事务成功后需要可靠触发下游。
- 业务能接受最终一致。

注意：

```text
事务消息保证的是本地事务和消息发送的一致性，不保证消费者业务一定成功。
消费者仍然要幂等和重试。
```

## 七、最大努力通知

发送方尽最大努力通知接收方，失败后重试一定次数。

适合：

- 支付结果通知。
- 物流状态通知。
- 对账可修复场景。

特点：

- 不保证实时一致。
- 依赖接收方幂等。
- 通常配合对账系统。

## 八、方案选择

| 场景 | 推荐方案 |
|---|---|
| 强一致、参与方少 | XA/2PC |
| 资源预留明确 | TCC |
| 长流程、多步骤 | Saga |
| 本地事务后发消息 | 本地消息表 / 事务消息 |
| 第三方通知 | 最大努力通知 + 对账 |
| 高并发核心链路 | 最终一致优先 |

## 九、生产经验

- 不要一上来就追求强一致，先判断业务是否能接受最终一致。
- 最终一致系统必须有对账和补偿。
- 消费者幂等是分布式事务闭环的一部分。
- TCC 的复杂度很高，不要低估空回滚、悬挂、幂等。
- 事务边界要短，不要把远程调用放进数据库长事务。

## 十、常见面试追问

### 1. RocketMQ 事务消息解决了什么

解决本地事务和消息发送的一致性。它不能保证消费者处理一定成功，所以消费者仍要重试、幂等和补偿。

### 2. 本地消息表和事务消息怎么选

本地消息表更通用、可控，但需要自己做扫描和投递状态。事务消息减少了本地消息投递复杂度，但依赖 MQ 能力。

### 3. 分布式事务最终靠什么兜底

靠幂等、重试、补偿、对账、人工处理通道共同兜底。
