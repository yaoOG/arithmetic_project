# 接口幂等设计

## 一、先说结论

幂等是指同一个请求被执行一次和执行多次，最终业务结果一致。

资深开发要把幂等和三个概念区分开：

- 幂等：多次执行结果一致。
- 防重：防止重复提交。
- 并发控制：防止多个请求同时修改同一份资源造成竞争。

典型必须做幂等的场景：

- 支付回调。
- 订单创建。
- 库存扣减。
- 优惠券领取。
- MQ 消费。
- 第三方接口重试。
- 用户重复点击提交。

一句话：

```text
只要请求可能因为超时、重试、重复投递、用户重复点击而多次到达，就要考虑幂等。
```

## 二、为什么会重复

常见来源：

- 前端重复点击。
- 网关超时重试。
- RPC 框架重试。
- MQ 至少一次投递。
- 第三方回调多次通知。
- 用户刷新页面。
- 服务处理成功但响应丢失。

最典型的是：

```text
服务端已经扣款成功，但响应超时，调用方认为失败，于是重试。
```

如果没有幂等，就可能重复扣款、重复发货、重复发券。

## 三、常见幂等方案

### 1. 唯一索引

利用数据库唯一约束兜底。

适合：

- 创建订单。
- 支付流水。
- 消费记录。
- 用户领取优惠券。

示例：

```sql
create unique index uk_order_no on t_order(order_no);
```

优点：

- 强一致。
- 简单可靠。
- 数据库天然兜底。

缺点：

- 依赖数据库。
- 高并发下冲突会造成异常和锁竞争。

面试表达：

```text
唯一索引是幂等设计里最可靠的最后一道防线，尤其适合防止重复插入。
```

### 2. Token 防重

流程：

```text
1. 客户端进入表单页，先向服务端申请 token。
2. 服务端把 token 存 Redis，并返回给客户端。
3. 客户端提交时携带 token。
4. 服务端原子校验并删除 token。
5. 删除成功才执行业务。
```

适合：

- 表单重复提交。
- 下单按钮重复点击。

Redis 处理必须原子化，通常用 Lua：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 3. 状态机

利用业务状态流转保证幂等。

例如订单状态：

```text
待支付 -> 已支付 -> 已发货 -> 已完成
```

支付回调时：

```sql
update t_order
set status = 'PAID'
where order_no = ?
  and status = 'WAIT_PAY';
```

如果重复回调，第二次更新影响行数为 0，业务自然幂等。

优点：

- 贴合业务。
- 可读性强。
- 能防止非法状态跳转。

### 4. 去重表

为请求或消息建立处理记录。

```sql
create table t_request_log (
    request_id varchar(64) primary key,
    status varchar(16),
    create_time datetime
);
```

处理流程：

```text
1. 先插入 request_id。
2. 插入成功，执行业务。
3. 插入失败，说明处理过，直接返回历史结果或忽略。
```

适合：

- MQ 消费幂等。
- 第三方回调。
- 跨系统请求。

### 5. Redis SET NX

```text
SET key value NX EX seconds
```

适合：

- 短时间防重。
- 接口频控。
- 非核心强一致场景。

注意：

- Redis 幂等通常不能作为金融级最终防线。
- 关键业务仍建议数据库唯一索引兜底。

### 6. 乐观锁

使用 version 字段：

```sql
update account
set balance = balance - 100,
    version = version + 1
where id = ?
  and version = ?;
```

适合：

- 防止并发覆盖。
- 库存扣减。
- 账户余额更新。

## 四、支付回调幂等设计

核心原则：

```text
支付平台可能多次通知，业务系统必须做到重复通知不重复入账。
```

推荐设计：

```text
1. 校验签名。
2. 根据支付流水号查询本地支付单。
3. 如果已成功，直接返回成功。
4. 如果未成功，使用状态机更新：INIT -> SUCCESS。
5. 更新成功后，触发订单后续流程。
6. 支付流水号建立唯一索引。
```

关键 SQL：

```sql
update t_pay_order
set status = 'SUCCESS',
    pay_time = now()
where pay_no = ?
  and status = 'INIT';
```

如果影响行数是 1，说明本次完成状态推进。  
如果影响行数是 0，说明已经处理过或状态不允许变更。

## 五、MQ 消费幂等设计

MQ 通常保证至少一次投递，不保证只投递一次。

消费者必须幂等。

常见方案：

```text
消息业务唯一键 + 消费记录表唯一索引
```

流程：

```text
1. 开启本地事务。
2. 插入消费记录 message_id。
3. 插入成功，执行业务。
4. 插入失败，说明消费过，直接 ack。
5. 提交事务。
```

注意：

```text
消费记录和业务更新最好在同一个本地事务里提交。
```

## 六、幂等设计的层次

推荐组合：

```text
前端防重复点击
-> 网关限流
-> 接口 token 或 requestId
-> 业务状态机
-> 数据库唯一索引兜底
```

不要只依赖前端，前端防重只能优化体验，不能保证安全。

## 七、常见面试追问

### 1. 幂等是不是只要加分布式锁

不是。锁解决的是并发互斥，幂等解决的是重复请求结果一致。锁过期、服务重启、请求重试后仍可能重复，关键业务必须有业务唯一键或状态机兜底。

### 2. requestId 放 Redis 能保证绝对幂等吗

不能。Redis 可能过期、被淘汰、主从切换丢数据。强一致业务应配合数据库唯一索引。

### 3. 幂等返回什么

通常重复请求应该返回与第一次成功相同或兼容的结果，而不是直接报错。比如支付回调重复到达，应返回成功，避免第三方持续重试。

## 八、生产经验

- 业务唯一键必须由业务语义决定，不要随便用随机 UUID 替代。
- 幂等记录要有过期清理策略，但核心流水不应随意删除。
- 关键业务用数据库约束兜底。
- 状态机更新必须带上旧状态条件。
- 幂等和事务边界要一起设计，否则可能出现记录写了但业务没执行，或业务执行了但记录没写。
