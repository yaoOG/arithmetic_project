# MQ 消费堆积排查 SOP

## 一、先说结论

MQ 消费堆积的本质是：

```text
生产速度 > 消费速度
```

排查核心：

```text
看堆积量
-> 看生产速率
-> 看消费速率
-> 看消费者是否报错
-> 看单条消费耗时
-> 看下游依赖
-> 看分区/队列并行度
```

## 二、常见表现

- 消息 lag 持续增加。
- 消费延迟变大。
- 消费者 CPU 不一定高。
- 业务数据延迟同步。
- 重试队列增多。
- 死信消息增多。

## 三、常见原因

### 1. 消费者报错反复重试

表现：

- 同一消息重复消费。
- 重试次数增加。
- 消费位点推进慢。

处理：

- 查看异常日志。
- 判断是否脏数据。
- 修复代码后重新消费。
- 对无法处理的数据进入死信或人工处理。

### 2. 单条消费太慢

原因：

- 慢 SQL。
- 下游 RPC 慢。
- 调第三方接口。
- 大对象处理。
- 锁竞争。

处理：

- 优化下游。
- 批量处理。
- 异步化。
- 增加超时和熔断。

### 3. 消费并行度不足

Kafka 中并行度受 partition 数影响。  
RocketMQ 中并行度受 MessageQueue 数和消费者线程数影响。

如果队列数太少，单纯增加消费者实例可能无效。

### 4. 顺序消费限制

顺序消费为了保证顺序，会牺牲并发。

某个队列中的慢消息会阻塞后续消息。

处理：

- 优化单条处理耗时。
- 拆分顺序粒度。
- 避免全局顺序。

### 5. 下游限流或故障

消费者不是瓶颈，下游才是瓶颈。

比如：

- 数据库连接池满。
- Redis 慢。
- 第三方接口限流。
- ES 写入慢。

## 四、排查步骤

### 1. 看堆积趋势

判断是突发堆积还是持续堆积。

```text
突发：可能是流量峰值。
持续：消费能力长期不足或消费者异常。
```

### 2. 看消费者日志

重点：

- 是否异常。
- 是否大量重试。
- 是否有超时。
- 是否卡在同一条消息。

### 3. 看消费耗时

统计：

- 平均耗时。
- P95。
- P99。
- 最大耗时。

### 4. 看线程池

消费线程是否打满。  
业务处理线程池是否队列积压。

### 5. 看下游指标

查：

- 数据库慢 SQL。
- RPC RT。
- 连接池。
- Redis latency。
- ES bulk 写入。

## 五、应急处理

- 临时扩容消费者。
- 提高消费线程数。
- 暂停非核心逻辑。
- 跳过或隔离异常消息。
- 下游扩容。
- 临时关闭顺序要求不强的处理。
- 批量消费和批量写入。

注意：

```text
扩容消费者前先确认队列/分区数，否则可能无效。
```

## 六、长期优化

- 消费逻辑幂等。
- 消费耗时监控。
- 重试和死信机制。
- 慢消息隔离。
- 下游调用超时。
- 批处理。
- 队列数规划。
- 消费者水平扩展。
- 生产限流。

## 七、常见面试追问

### 1. 增加消费者实例一定能解决堆积吗

不一定。Kafka 消费并行度受 partition 数限制，RocketMQ 受 MessageQueue 数限制。如果队列数不足，增加实例也不能提升并行度。

### 2. 顺序消息堆积怎么办

先优化单条消息处理耗时，再考虑拆细顺序键，避免把所有消息放到同一个队列里。全局顺序代价很高，能不用就不用。

### 3. 消费失败一直重试怎么办

要区分临时异常和永久脏数据。临时异常可以重试，永久异常应进入死信或人工处理，不能阻塞整个队列。

## 八、面试表达模板

```text
MQ 堆积我会先看生产速率和消费速率，确认是流量突增还是消费异常。
然后看消费者日志、重试队列和单条消费耗时，判断是否报错重试或下游慢。
如果是并行度不足，会看 Kafka partition 或 RocketMQ MessageQueue 数量，单纯加机器不一定有效。
处理上先扩容和降级止血，再优化慢消费、幂等、重试、死信和队列规划。
```
