# 线程池打满排查 SOP

## 一、先说结论

线程池打满通常不是线程池本身的问题，而是任务处理速度低于任务进入速度。

排查核心：

```text
看活跃线程数
-> 看队列长度
-> 看拒绝次数
-> 看任务耗时
-> 看下游依赖
-> 看是否有锁等待或阻塞 IO
```

## 二、关键指标

必须监控：

- corePoolSize。
- maximumPoolSize。
- activeCount。
- poolSize。
- queueSize。
- queueRemainingCapacity。
- completedTaskCount。
- rejectedCount。
- taskCostTime。
- taskWaitTime。

其中最关键：

```text
活跃线程数、队列积压、拒绝次数、任务耗时。
```

## 三、常见表现

- 接口 RT 飙升。
- 请求超时。
- 线程池队列持续增长。
- 触发拒绝策略。
- CPU 不一定高。
- 下游连接池可能先满。

## 四、排查步骤

### 1. 确认哪个线程池打满

线程名必须有业务语义，例如：

```text
order-pay-executor-1
inventory-sync-executor-1
```

如果线程名都是 `pool-1-thread-1`，排查会非常痛苦。

### 2. 查看线程栈

```bash
jstack <pid> > jstack.log
```

看线程在做什么：

- 调数据库。
- 调 RPC。
- 等锁。
- 等 Redis。
- 写文件。
- 执行业务循环。

### 3. 看队列

队列持续增长说明：

```text
任务进入速度 > 任务处理速度
```

如果队列无界，可能最终 OOM。

### 4. 看下游连接池

线程池打满常常是下游慢导致：

- 数据库连接池满。
- Redis 慢。
- HTTP 连接池满。
- MQ broker 慢。
- 第三方接口慢。

### 5. 看锁竞争

jstack 中如果大量线程：

```text
BLOCKED
WAITING
parking
```

要继续查锁持有者。

## 五、常见根因

### 1. 下游慢调用

最常见。

例如数据库慢 SQL 导致每个任务从 50ms 变成 5s，线程很快被占满。

### 2. 队列过大

无界队列会让请求不断堆积，延迟越来越高，最终 OOM。

### 3. 拒绝策略不合理

默认 `AbortPolicy` 会直接抛异常。

常见策略：

- AbortPolicy。
- CallerRunsPolicy。
- DiscardPolicy。
- DiscardOldestPolicy。

生产中通常自定义拒绝策略：

- 记录日志。
- 打点告警。
- 快速失败。
- 返回降级结果。

### 4. 线程池共享

多个业务共用一个线程池，低优先级任务占满线程，核心任务被拖垮。

解决：

```text
按业务重要性做线程池隔离。
```

### 5. 任务内部阻塞

例如：

- Future.get 无超时。
- CountDownLatch 等待不释放。
- synchronized 锁竞争。
- IO 无超时。

## 六、处理策略

### 1. 先止血

- 限流。
- 降级。
- 暂停非核心任务。
- 扩容实例。
- 临时调大线程池和队列。

注意：

```text
临时调大线程池不是根治，可能把压力转移到数据库或下游。
```

### 2. 再定位

- 查线程栈。
- 查慢 SQL。
- 查 RPC RT。
- 查连接池。
- 查锁竞争。
- 查任务来源。

### 3. 最后修复

- 优化慢 SQL。
- 下游调用加超时。
- 加熔断降级。
- 线程池隔离。
- 队列有界。
- 任务拆批。
- 调整参数并压测。

## 七、线程池参数建议

不要只靠公式。

公式只能给初始值：

```text
线程数 = CPU 核数 * 目标 CPU 利用率 * (1 + 等待时间 / 计算时间)
```

最终必须通过压测和监控调整。

注意下游约束：

```text
如果数据库连接池只有 50，线程池开 300 可能只会制造更多等待。
```

## 八、生产经验

- 所有线程池必须命名。
- 队列必须有界。
- 拒绝策略必须可观测。
- 线程池必须暴露监控指标。
- 不同业务隔离线程池。
- 下游调用必须有超时。
- 动态线程池可以提高治理效率，但不能代替容量评估。

## 九、面试表达模板

```text
线程池打满时我会先看 activeCount、queueSize、rejectedCount 和任务耗时。
然后通过线程名定位具体线程池，用 jstack 看线程都阻塞在哪里。
如果大量线程卡在数据库或 RPC，根因通常是下游慢，不是简单把线程数调大。
处理上先限流降级止血，再优化下游、隔离线程池、设置有界队列和合理拒绝策略，并通过压测确定最终参数。
```
