# Spring 三级缓存解决循环依赖

## 一、先说结论

Spring 解决的是“单例 Bean 的属性注入循环依赖”，核心靠三级缓存提前暴露 Bean：

```text
一级缓存 singletonObjects：完整初始化好的单例 Bean
二级缓存 earlySingletonObjects：提前暴露的早期 Bean
三级缓存 singletonFactories：能生成早期 Bean 引用的 ObjectFactory
```

它能解决：

```java
class A {
    @Autowired B b;
}

class B {
    @Autowired A a;
}
```

它不能解决：

- 构造器循环依赖。
- prototype Bean 循环依赖。
- 一些异步初始化、手动 new、非 Spring 管理对象造成的循环依赖。

## 二、为什么会有循环依赖问题

假设 A 依赖 B，B 又依赖 A：

```text
创建 A
-> A 需要注入 B
-> 创建 B
-> B 需要注入 A
-> A 还没创建完
```

如果没有提前暴露机制，Spring 会陷入死循环或直接失败。

## 三、三级缓存分别存什么

### 1. 一级缓存：singletonObjects

存放完整可用的单例 Bean。

```text
bean 已完成实例化、属性注入、初始化、AOP 代理处理。
```

日常从容器获取单例 Bean，主要就是从一级缓存拿。

### 2. 二级缓存：earlySingletonObjects

存放提前暴露的早期 Bean 引用。

这个 Bean 还没有走完整生命周期，可能只是原始对象，也可能是提前生成的代理对象。

### 3. 三级缓存：singletonFactories

存放 `ObjectFactory<?>`，它不是 Bean 本身，而是一个“获取早期 Bean 引用的工厂”。

它的价值是：

```text
在真正需要提前引用时，再决定暴露原始对象还是代理对象。
```

这点对 AOP 很关键。

## 四、核心流程

以 A 和 B 属性注入循环依赖为例：

```text
1. 创建 A，先实例化 A，此时 A 还是半成品。
2. Spring 把 A 的 ObjectFactory 放入三级缓存。
3. 给 A 填充属性，发现需要 B。
4. 创建 B，先实例化 B。
5. 给 B 填充属性，发现需要 A。
6. Spring 发现 A 正在创建中，于是去缓存里找 A。
7. 一级缓存没有 A。
8. 二级缓存没有 A。
9. 三级缓存有 A 的 ObjectFactory。
10. 调用 ObjectFactory，得到 A 的早期引用。
11. 把 A 的早期引用放入二级缓存，并删除三级缓存。
12. B 注入 A，B 完成初始化，放入一级缓存。
13. A 继续注入 B，A 完成初始化，放入一级缓存。
```

关键点：

```text
Spring 不是等 A 完全创建完才给 B 注入 A，而是在 A 实例化后、初始化前提前暴露了一个引用。
```

## 五、为什么需要三级缓存，二级不够吗

如果没有 AOP，理论上二级缓存也能解决普通循环依赖，因为提前暴露原始对象即可。

但有 AOP 时，最终放进容器的可能是代理对象。如果 B 注入了 A 的原始对象，而容器最终暴露的是 A 的代理对象，就会出现同一个 Bean 在系统里有两个不同引用：

```text
B 里持有原始 A
容器里持有代理 A
```

这样事务、切面、异步等增强可能失效。

三级缓存的意义在于：

```text
延迟决定早期暴露对象的形态。
如果 Bean 需要 AOP，就通过 ObjectFactory 提前生成代理对象；
如果不需要 AOP，就暴露原始对象。
```

## 六、为什么构造器循环依赖无法解决

构造器注入要求对象创建时依赖必须已经存在：

```java
class A {
    A(B b) {}
}

class B {
    B(A a) {}
}
```

问题在于：

```text
A 连实例化都没完成，无法提前暴露引用。
```

三级缓存的前提是：

```text
对象已经实例化，只是属性还没填充、初始化还没完成。
```

构造器循环依赖卡在实例化之前，因此无解。

## 七、为什么 prototype 循环依赖无法解决

prototype Bean 每次获取都创建新对象，Spring 不维护完整单例缓存。

三级缓存是单例池机制的一部分，本质服务于 singleton。

如果 prototype A 依赖 prototype B，B 又依赖 A，Spring 没有一个全局稳定的“早期引用缓存”可用，所以无法解决。

## 八、Spring Boot 2.6 之后的注意点

Spring Boot 2.6 开始默认不鼓励循环依赖，配置项 `spring.main.allow-circular-references` 默认倾向于关闭循环依赖支持。

工程上不应该把三级缓存当成设计工具。出现循环依赖通常说明模块职责不清，优先考虑：

- 抽取第三个协调服务。
- 使用事件发布。
- 使用策略接口。
- 调整依赖方向。
- 通过领域服务拆开双向依赖。

## 九、常见面试追问

### 1. 三级缓存能解决所有循环依赖吗

不能。它主要解决 singleton Bean 的属性注入循环依赖。

### 2. 为什么不直接提前暴露代理对象

不是所有 Bean 都需要代理。三级缓存通过 `ObjectFactory` 延迟生成早期引用，可以让 AOP 有机会参与，同时避免过早创建代理。

### 3. 早期代理对象和最终代理对象如何保持一致

Spring 在提前暴露代理后，会把早期引用放入二级缓存，后续完成初始化时要保证最终暴露对象与早期引用一致，否则可能触发异常或导致代理不一致问题。

## 十、生产经验

- 构造器注入更推荐，因为它能尽早暴露循环依赖设计问题。
- 不要为了“解决循环依赖”改成字段注入，这通常是在掩盖设计问题。
- Service 互相调用时，优先检查是否职责边界混乱。
- 循环依赖叠加 AOP 时，最容易出现事务失效、代理不一致、启动失败等问题。
