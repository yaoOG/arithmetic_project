关于 Spring 中 Bean 的作用域（Scope），这是理解 Spring 容器对象生命周期管理和规避并发 BUG 的绝对基础。

在现代的 Spring 5.x 及以上版本中，官方总共支持 **6 种** Bean 的作用域。作为一个老兵，我通常会把它们划分为两大阵营向团队梳理：**“基础阵营”**和**“Web 阵营”**。

---

### 一、 基础阵营（任何 Spring 环境均适用）

这两种作用域是 Spring Core 包自带的，无论你是写 Web 应用还是写跑在后台的定时任务工程，它们都在默默工作：

**1. Singleton（单例 - 默认作用域）**

- **核心机制**：在整个 Spring IoC 容器中，该 Bean 只会存在**唯一的一个实例**。每次通过 `getBean()` 获取，或者通过 `@Autowired` 注入，拿到的全都是同一个内存地址的对象。
    
- **适用场景**：无状态（Stateless）的类，比如 `Dao` 层、`Service` 层、配置类、工具类。
    
- **老兵排雷**：因为是全局共享的，**绝对不能在 Singleton Bean 中定义有状态的共享成员变量**（比如用来保存用户请求参数的非线程安全集合），否则在多线程高并发下会引发极其严重的线程安全问题。
    

**2. Prototype（原型 / 多例）**

- **核心机制**：每次对该 Bean 发出请求（无论是手动 `getBean` 还是被注入到其他 Bean 中），Spring 容器都会在内存里 **`new` 一个全新的实例**返回。
    
- **适用场景**：有状态（Stateful）的类，也就是每个调用者都需要保存自己独立数据的对象。
    
- **老兵排雷**：正如咱们之前聊生命周期时探讨过的，Spring 对 Prototype Bean **“只管生，不管死”**。如果你的 Prototype Bean 内部持有了数据库连接或文件流，用完后不显式销毁，极易造成资源耗尽。
    

---

### 二、 Web 阵营（仅限 Web MVC / WebFlux 环境下生效）

当你的项目是一个 Web 工程（应用上下文是 `WebApplicationContext`）时，Spring 会自动点亮以下 4 种依赖于 HTTP 协议的作用域：

**3. Request（请求级别）**

- **核心机制**：每一次 HTTP 请求到来时，Spring 会为该请求创建一个全新的 Bean 实例。当这个 HTTP 请求响应结束，该 Bean 就会被立刻销毁。
    
- **适用场景**：存放单次请求链路中的极短生命周期的上下文信息（如当前请求的路由参数、解析 JWT Token 后的临时身份信息）。
    

**4. Session（会话级别）**

- **核心机制**：同一个 HTTP Session（用户级别的会话）共享一个 Bean 实例。不同的 Session 拥有不同的实例。当 Session 失效或过期时，Bean 被销毁。
    
- **适用场景**：存放用户维度的状态信息，比如传统的 Web 时代的“购物车信息”或“用户登录态”。
    
    - _(注：在现代前后端分离的分布式微服务架构中，由于服务端追求无状态化，Session 作用域已经极少使用，基本被 Redis + Token 方案取代)。_
        

**5. Application（应用级别）**

- **核心机制**：在一个 Web 应用的 `ServletContext` 生命周期内，共享一个 Bean 实例。
    
- **进阶拷问：它和 Singleton 有什么区别？**
    
    - Singleton 限制的是**每个 Spring IoC 容器**只有一个实例。
        
    - Application 限制的是**每个 ServletContext** 只有一个实例。如果同一个 Web Tomcat 容器里部署了多个 DispatcherServlet（各有自己的 IoC 上下文），它们可以共享同一个 Application 级别的 Bean，但在各自的容器里会有独立的 Singleton Bean。
        

**6. WebSocket（长连接级别）**

- **核心机制**：在 Spring 4.0 中引入。该 Bean 的生命周期与一个 WebSocket 的会话严格绑定。
    
- **适用场景**：实时双向通信场景中，维护单个客户端长连接的状态和身份数据。

### 💡 终极扩展：自定义作用域 (Custom Scope)

在高级架构设计中，如果这 6 种官方作用域都满足不了业务，Spring 留了后门：

比如你需要一个“线程级别”的作用域（同一个线程内无论怎么调都返回同一个 Bean，不同线程互不干扰），你可以实现 `org.springframework.beans.factory.config.Scope` 接口，然后通过 `CustomScopeConfigurer` 把它注册进 Spring 容器。其实 Spring 官方底层已经写好了一个叫 `SimpleThreadScope` 的类，只是默认没有激活而已。
