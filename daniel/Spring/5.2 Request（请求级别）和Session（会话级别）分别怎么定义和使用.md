在实际的 Spring Boot 开发中，要定义和使用 `Request` 和 `Session` 作用域的 Bean，其实代码非常简单，但**里面暗藏了一个极其凶险的“代理注入陷阱”**。

作为一个老兵，我为您分为“标准定义”、“实战使用”和“底层避坑”三个步骤来拆解：

### 一、 如何定义？（使用专属注解）

在早期的 XML 时代，我们要写 `<bean scope="request">`。但在现在的注解驱动开发时代，Spring 为我们提供了非常优雅的复合注解：**`@RequestScope`** 和 **`@SessionScope`**。

#### 1. 定义 Request 级别的 Bean

通常用来存储当前请求的链路追踪 ID（TraceId）、当前请求解析出的临时用户信息等。

Java

```
import org.springframework.stereotype.Component;
import org.springframework.web.context.annotation.RequestScope;
import java.util.UUID;

@Component
@RequestScope  // 标明这是一个请求级别的 Bean
public class RequestTraceContext {
    // 每次请求都会生成一个新的 ID
    private final String traceId = UUID.randomUUID().toString();

    public String getTraceId() {
        return traceId;
    }
}
```

#### 2. 定义 Session 级别的 Bean

通常用来存储传统 Web 登录用户的购物车、会话状态等（前提是你的应用启用了 HTTP Session）。

Java

```
import org.springframework.stereotype.Component;
import org.springframework.web.context.annotation.SessionScope;
import java.util.ArrayList;
import java.util.List;

@Component
@SessionScope  // 标明这是一个会话级别的 Bean
public class ShoppingCart {
    private List<String> items = new ArrayList<>();

    public void addItem(String item) {
        items.add(item);
    }
    public List<String> getItems() {
        return items;
    }
}
```

### 二、 如何使用？（直接注入）

使用起来极其自然，就像普通的单例 Bean 一样，直接在您的 Controller 或 Service 里使用 `@Autowired` 注入即可。

Java

```
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class ShopController {

    // 正常注入即可
    @Autowired
    private RequestTraceContext traceContext;

    @Autowired
    private ShoppingCart shoppingCart;

    @GetMapping("/buy")
    public String buyItem(String item) {
        // 使用 Session 级别的 Bean 记录商品
        shoppingCart.addItem(item);
        
        return "TraceID: " + traceContext.getTraceId() + 
               " | 购物车共有商品: " + shoppingCart.getItems().size();
    }
}
```

### 三、 ⚠️ 老兵避坑指南：必须理解的“代理魔法”

很多新人看到上面的代码，会产生一个巨大的疑问：

**“不对啊！`ShopController` 默认是 Singleton（单例）的，它在 Spring 刚启动时就被创建好、且只创建一次。如果它把 `RequestTraceContext` 注入进来了，那这个 Context 不就被写死在 Controller 里了吗？怎么可能做到每次请求都动态变化呢？”**

这就是面试中极易被问到的“长生命周期 Bean 注入短生命周期 Bean”的冲突问题。

Spring 的解法极其巧妙：**注入代理对象 (Proxy)！**

如果您点开 `@RequestScope` 的源码，会发现它长这样：

Java

```
@Scope(value = WebApplicationContext.SCOPE_REQUEST, proxyMode = ScopedProxyMode.TARGET_CLASS)
```

关键就在于这个 **`proxyMode = ScopedProxyMode.TARGET_CLASS`**。

1. 当 Spring 启动并实例化 `ShopController` 时，它并没有把真正的 `RequestTraceContext` 注入进去（因为这时候根本还没有 HTTP 请求）。
    
2. 它注入的是一个用 CGLIB 动态生成的 **代理对象 (Proxy)**。
    
3. 当真正的 HTTP 请求到来，你在 Controller 里调用 `traceContext.getTraceId()` 时，这个**代理对象**会去底层的 `RequestContextHolder`（基于 ThreadLocal 实现）中，找到绑定在当前线程上的、属于**本次真实 HTTP 请求**的那个 `RequestTraceContext` 实例，然后把方法调用转发过去。
