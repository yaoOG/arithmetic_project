
## 一、 JDK 动态代理 vs. CGLIB 详细对比

### 1. JDK 动态代理 (JDK Dynamic Proxy)

- **实现原理：** Java 官方提供的内置方式（位于 `java.lang.reflect` 包）。核心是利用**反射机制**创建一个实现了特定接口的代理类。
    
    - 开发者需实现 `InvocationHandler` 接口。
        
    - 所有方法调用都会转发到 `invoke()` 方法中，在此处编写增强逻辑。
        
- **前提条件：** **目标类必须实现至少一个接口**。代理类与目标类是“兄弟”关系（实现相同接口并继承 `Proxy` 类），无法代理普通类。
    

**代码示意：**


```Java
// Target object
UserService userService = new UserServiceImpl();

// Create Proxy
UserService proxyInstance = (UserService) Proxy.newProxyInstance(
    userService.getClass().getClassLoader(),
    userService.getClass().getInterfaces(), // 必须提供接口
    new MyInvocationHandler(userService)
);
```

### 2. CGLIB (Code Generation Library)

- **实现原理：** 第三方字节码生成库。通过**继承**的方式创建代理。在运行时动态生成目标类的**子类**，并重写父类中所有非 `final` 的方法。
- 核心是 `MethodInterceptor` 接口（类似于 JDK 的 `InvocationHandler`）。
- **前提条件：** **目标类不能是 `final` 的**，且目标方法不能是 `final` 或 `private` 的。

**代码示意：**

```Java
// Target object
OrderService orderService = new OrderService(); // 普通类

// Create Proxy
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(OrderService.class); // 设置父类
enhancer.setCallback(new MyMethodInterceptor(orderService));
OrderService proxyInstance = (OrderService) enhancer.create();
```

---

### 3. 异同点总结

|**特性**|**JDK 动态代理**|**CGLIB 代理**|
|---|---|---|
|**实现机制**|反射 + 接口实现|继承 + 字节码增强|
|**依赖性**|JDK 原生支持，无需额外库|需要引入 CGLIB 依赖（Spring 已内置）|
|**目标类要求**|必须实现接口|不能是 `final` 类|
|**代理对象类型**|接口的实现类|目标类的子类|
|**方法限制**|仅限接口定义的方法|不能重写 `final`/`private` 方法|

---

### 4. 性能总结

这是一个需要用“发展眼光”看待的话题：

- **早期观点（JDK 6/7 时代）：**
    
    - **创建性能：** CGLIB 通常优于 JDK，因为它生成的字节码更优化。
    - **调用性能：** JDK 略优于 CGLIB，因为 CGLIB 内部机制相对复杂。
    
- **现代观点（JDK 8 及以后）：**
    
    - **差距微乎其微：** 随着 JVM 和反射技术的不断迭代优化，在现代项目（JDK 8+）中，两者的性能表现基本持平。
    - **结论：** 不应将性能作为选择的主要依据，而应根据**功能场景**选择。

## 二、 Spring 的选择策略

### 1. 经典 Spring AOP / 老版本 Spring Boot

**默认策略：** 动态判断。

- 如果目标 Bean **实现了接口**，默认使用 **JDK 动态代理**。
    
- 如果目标 Bean **没有实现接口**，强制使用 **CGLIB**。
    
- **手动干预：** 可通过配置 `proxy-target-class="true"` 强制所有 Bean 统一使用 CGLIB。
    

### 2. 现代 Spring Boot (2.x 及以后)

**默认策略：** **统一使用 CGLIB**。

Spring Boot 2.x 开始，默认将 `spring.aop.proxy-target-class` 设置为 `true`。

**为什么要做这个改变？**

主要是为了避免 **类型转换异常（ClassCastException）**：

- **场景：** 假设 `UserServiceImpl` 实现了 `UserService` 接口。
- **JDK 代理的问题：** 代理对象是 `UserService` 的实现类，但它**不是** `UserServiceImpl` 的实例。如果你尝试用 `@Autowired` 注入具体的实现类（如下），会报错：

    ```Java
    @Autowired
    private UserServiceImpl impl; // 使用 JDK 代理时，这里会报错
    ```

- **CGLIB 的优势：** 代理对象是 `UserServiceImpl` 的子类，因此它既是接口类型，也是实现类类型，向上转型完全安全。


> **总结：** 通过默认使用 CGLIB，Spring Boot 统一了代理行为，使得 AOP 的表现更加稳定和可预测。若需切换回 JDK 代理，可手动设置 `spring.aop.proxy-target-class=false`。
