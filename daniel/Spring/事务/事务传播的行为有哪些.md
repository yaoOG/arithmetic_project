Spring 事务传播行为主要有 7 种，由 `Propagation` 枚举定义：

| 传播行为 | 含义 |
|---|---|
| `REQUIRED` | 默认值。当前有事务就加入，没有事务就新建 |
| `REQUIRES_NEW` | 无论当前有没有事务，都新建一个事务；如果已有事务，先挂起外层事务 |
| `SUPPORTS` | 当前有事务就加入，没有事务就以非事务方式执行 |
| `NOT_SUPPORTED` | 总是以非事务方式执行；如果当前有事务，先挂起 |
| `MANDATORY` | 必须在已有事务中执行；如果当前没有事务就抛异常 |
| `NEVER` | 必须以非事务方式执行；如果当前有事务就抛异常 |
| `NESTED` | 当前有事务就在嵌套事务中执行；没有事务则新建事务 |

常用的是这几个：

**`REQUIRED`**

```java
@Transactional(propagation = Propagation.REQUIRED)
```

最常见，也是默认值。

外层有事务：

```text
外层事务
  -> 内层方法加入外层事务
```

外层没有事务：

```text
创建一个新事务
```

所以外层和内层通常是同一个事务，一起提交，一起回滚。

**`REQUIRES_NEW`**

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```

每次都会开启一个新事务。

如果外层已经有事务：

```text
外层事务挂起
  -> 开启内层新事务
  -> 内层提交/回滚
外层事务恢复
```

内层事务和外层事务相互独立。

典型场景：记录日志、保存审计记录，希望即使主业务失败，日志也能单独提交。

**`NESTED`**

```java
@Transactional(propagation = Propagation.NESTED)
```

如果当前有事务，会在当前事务里创建一个保存点 `savepoint`。

```text
外层事务
  -> 创建 savepoint
  -> 内层执行失败，回滚到 savepoint
  -> 外层事务可以继续执行
```

它和 `REQUIRES_NEW` 的区别是：

| 对比 | `REQUIRES_NEW` | `NESTED` |
|---|---|---|
| 是否新建独立事务 | 是 | 不是 |
| 外层事务是否挂起 | 是 | 否 |
| 实现方式 | 新事务 | 保存点 savepoint |
| 内层提交是否立即生效 | 是 | 否，最终还要等外层提交 |
| 外层回滚是否影响内层 | 不影响已提交的内层事务 | 影响，外层回滚时内层也没了 |

举个简单例子：

```java
@Transactional
public void outer() {
    orderService.createOrder();

    try {
        stockService.deductStock();
    } catch (Exception e) {
        // 如果 deductStock 是 NESTED，
        // 可以只回滚 deductStock，outer 继续
    }

    payService.createPayRecord();
}
```

一句话记忆：

```text
REQUIRED：有就加入，没有就新建
REQUIRES_NEW：总是新建，外层挂起
SUPPORTS：有就加入，没有就算了
NOT_SUPPORTED：总是不用事务
MANDATORY：必须已有事务
NEVER：必须没有事务
NESTED：有事务就建保存点，局部回滚
```
