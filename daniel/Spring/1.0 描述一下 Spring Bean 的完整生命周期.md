### 阶段一：实例化 (Instantiation) - “Bean的诞生”

这是 Bean 生命周期的起点，Spring 容器根据 Bean 的定义信息创建出 Bean 的实例。

1. **加载Bean定义：** Spring 容器启动，通过 XML 配置文件、注解（@Component, @Service 等）或 Java Config（@Configuration, @Bean）加载 Bean 的定义信息（元数据）。

2. **实例化Bean：** 当容器需要一个 Bean 实例时（例如，启动时初始化单例 Bean，或首次请求原型 Bean），Spring 会通过反射机制调用其构造函数，创建一个空白的对象实例。此时，对象已经分配了内存空间，但其内部的属性还都是默认值（如 null）。


### 阶段二：属性填充 (Population) - “Bean的成长”

实例创建后，Spring 容器会为其注入所依赖的其他 Bean 或配置值。

1. **依赖注入 (DI)：** Spring 根据 Bean 定义中的依赖关系（如 @Autowired, @Resource, @Value 等注解），进行依赖注入。容器会从自身中查找所需的 Bean，并通过反射（字段注入、Setter注入等）将引用设置到当前 Bean 实例的属性上。

**至此，Bean 实例已经创建完成，并且其依赖的属性也已经被注入。但它还不能被外界使用，因为它尚未完成初始化。**

### 阶段三：初始化 (Initialization) - “Bean的成熟与对外服务”

这是生命周期中**最复杂、扩展点最多**的阶段。Spring 在此阶段提供了一系列回调，允许我们对 Bean 进行各种自定义处理，最终使其达到可用的状态。**这个阶段的调用顺序非常重要。**

1. **执行 Aware 接口相关方法：** 如果 Bean 实现了特定的 Aware 接口，Spring 会在此时调用相应的方法，将 Spring 容器自身的一些资源注入给 Bean。常见的有：
    
    BeanNameAware 的 setBeanName(): 注入 Bean 在容器中的ID（名字）。
    
    BeanFactoryAware 的 setBeanFactory(): 注入创建此 Bean 的 Bean 工厂。
    
    ApplicationContextAware 的 setApplicationContext(): 注入当前的应用上下文（ApplicationContext）。
    
2. **BeanPostProcessor 的前置处理：** 执行所有 BeanPostProcessor 实现类的 postProcessBeforeInitialization() 方法。这是一个非常重要的**容器级**扩展点，它会对容器中**所有**符合条件的 Bean 生效。我们可以在这里对 Bean 实例进行修改或包装。
    
3. **执行 Bean 自身的初始化方法：** Spring 会检查并执行 Bean 自身定义的初始化逻辑，调用顺序如下：
    
    **@PostConstruct 注解的方法：** 如果 Bean 的方法上标注了 @PostConstruct 注解，该方法会被调用。这是 JSR-250 规范定义的，**推荐使用**，因为它不与 Spring 框架耦合。
    
    **InitializingBean 接口的 afterPropertiesSet() 方法：** 如果 Bean 实现了 InitializingBean 接口，其 afterPropertiesSet() 方法会被调用。这是一种与 Spring 框架耦合较强的方式。
    
    **自定义的 init-method：** 如果在 Bean 定义中（XML 或 @Bean(initMethod=...)）指定了 init-method，该方法会被调用。
    
4. **BeanPostProcessor 的后置处理：** 执行所有 BeanPostProcessor 实现类的 postProcessAfterInitialization() 方法。这是另一个至关重要的扩展点。**Spring 的 AOP（面向切面编程）就是在这里实现的**。它会检查 Bean 是否需要被代理（例如，类上有 @Transactional 注解），如果需要，则会用一个代理对象替换原始的 Bean 实例返回给容器。我们最终从容器中获取的，可能就是这个代理对象。


**初始化阶段完成后，Bean 就进入了“可用状态”，被存放到单例池（Singleton Cache）中，可以被应用程序的其他部分注入和使用了。**

### 阶段四：销毁 (Destruction) - “Bean的终结”

当 Spring 容器关闭时（例如，调用 context.close() 方法），它会负责销毁所有单例 Bean。

1. **执行 Bean 自身的销毁方法：** 与初始化类似，Spring 会按以下顺序调用 Bean 的销毁逻辑：
    
    **@PreDestroy 注解的方法：** 如果 Bean 的方法上标注了 @PreDestroy 注解，该方法会被调用。同样，这是 JSR-250 规范，**推荐使用**。
    
    **DisposableBean 接口的 destroy() 方法：** 如果 Bean 实现了 DisposableBean 接口，其 destroy() 方法会被调用。
    
    **自定义的 destroy-method：** 如果在 Bean 定义中指定了 destroy-method，该方法会被调用。
    

**注意：** 销毁回调只对作用域为 singleton 的 Bean 有效。对于 prototype 作用域的 Bean，Spring 容器只负责创建和初始化，不会管理其销毁过程。
