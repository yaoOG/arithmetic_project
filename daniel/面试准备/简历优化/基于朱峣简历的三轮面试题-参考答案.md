# 基于朱峣简历的三轮面试题 - 参考答案

> 使用方式：  
> 这份答案不是让你逐字背，而是训练“先说结论 -> 解释机制 -> 结合项目 -> 补充风险和取舍”的回答方式。  
> 面试中优先结合你自己的真实数据补充，比如账单量、峰值 QPS、表数据量、任务耗时、故障案例等。

## 第一轮：基础核验 + 技术广度

### 1. 线程池核心参数有哪些？任务提交后线程池的执行流程是什么？

线程池核心参数包括 `corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`unit`、`workQueue`、`threadFactory`、`handler`。

执行流程是：

```text
1. 当前线程数小于 corePoolSize，直接创建核心线程执行。
2. 核心线程已满，任务进入 workQueue。
3. 队列满了，并且线程数小于 maximumPoolSize，创建非核心线程执行。
4. 线程数也达到 maximumPoolSize，触发拒绝策略。
```

生产中我更关注线程池的监控指标，比如活跃线程数、队列长度、拒绝次数、任务耗时。线程池参数不是只靠公式算出来，而是结合下游连接池、接口 RT、压测结果和线上监控持续调整。

### 2. 为什么不建议直接使用 Executors.newFixedThreadPool？

因为 `newFixedThreadPool` 默认使用无界的 `LinkedBlockingQueue`，如果任务生产速度长期大于消费速度，队列会持续堆积，最终可能导致 OOM。

生产中更推荐显式创建 `ThreadPoolExecutor`：

```text
有界队列 + 自定义线程名 + 可观测拒绝策略 + 监控指标
```

这样出现问题时能快速定位是哪个业务线程池打满。

### 3. 线程池队列满了以后会发生什么？生产中如何设计拒绝策略？

队列满后，如果线程数还没达到 `maximumPoolSize`，线程池会继续创建非核心线程；如果已经达到最大线程数，就触发拒绝策略。

JDK 内置拒绝策略有：

- `AbortPolicy`：抛异常。
- `CallerRunsPolicy`：调用方线程执行。
- `DiscardPolicy`：直接丢弃。
- `DiscardOldestPolicy`：丢弃队头任务。

生产中我通常会自定义拒绝策略，至少要做到：

```text
记录业务线程池名称、任务类型、队列长度、拒绝次数，打点告警，并根据业务选择快速失败、降级或调用方执行。
```

核心链路不能静默丢任务。

### 4. synchronized 和 ReentrantLock 有什么区别？

`synchronized` 是 JVM 层面的内置锁，语法简单，异常时自动释放锁；`ReentrantLock` 是 JUC 提供的显式锁，需要手动 `unlock`，通常写在 `finally` 中。

区别主要有：

- `ReentrantLock` 支持公平锁。
- 支持可中断获取锁：`lockInterruptibly()`。
- 支持超时获取锁：`tryLock(timeout)`。
- 支持多个条件队列：`Condition`。
- `synchronized` 使用更简单，日常优先使用。

如果只是普通互斥，我会优先用 `synchronized`；如果需要超时、可中断、公平锁、多条件队列，再考虑 `ReentrantLock`。

### 5. volatile 能保证什么？为什么不能保证 i++ 线程安全？

`volatile` 主要保证两点：

```text
1. 可见性：一个线程修改 volatile 变量，其他线程能及时看到。
2. 禁止指令重排：通过内存屏障约束读写顺序。
```

但它不保证复合操作的原子性。`i++` 实际包含读取、加一、写回三个步骤，多个线程并发执行时仍然可能丢失更新。

如果需要原子递增，应该使用 `AtomicInteger`、`LongAdder` 或加锁。

### 6. CAS 的 ABA 问题是什么？Java 中如何解决？

CAS 比较的是内存位置上的值是否等于预期值。ABA 问题是：一个值从 A 变成 B，又变回 A，CAS 会认为它没变过，但实际上中间发生过变化。

解决方式是引入版本号或时间戳：

- `AtomicStampedReference`
- `AtomicMarkableReference`

很多业务场景也会用数据库 `version` 字段做乐观锁，本质也是给状态变化加版本。

### 7. ThreadLocal 为什么可能导致内存泄漏？线程池场景下怎么用？

`ThreadLocalMap` 的 key 是弱引用，但 value 是强引用。如果 ThreadLocal 对象被回收，key 变成 null，但 value 仍然挂在线程上。在线程池里线程长期存活，如果不清理 value，就可能泄漏。

正确用法：

```java
try {
    threadLocal.set(value);
    // business
} finally {
    threadLocal.remove();
}
```

尤其是用户信息、traceId、租户信息这类上下文，在线程池场景必须 remove。

### 8. CompletableFuture 适合解决什么问题？使用时有哪些坑？

`CompletableFuture` 适合做异步编排和并行调用，比如一个接口需要同时查库存、价格、营销、用户权益，然后聚合结果。

常见坑：

- 默认使用 `ForkJoinPool.commonPool`，容易和其他任务互相影响。
- 异步任务异常如果不处理，容易丢失。
- `get()` 或 `join()` 没有超时会导致线程阻塞。
- 事务、ThreadLocal 上下文不会自动跨线程传播。

生产中我会使用自定义线程池、设置超时、处理异常，并避免在事务中随意开启异步任务。

### 9. HashMap 的 put 流程是什么？

JDK 8 中 HashMap 是数组 + 链表 + 红黑树。

`put` 流程：

```text
1. 计算 key 的 hash，并做扰动。
2. 根据 (n - 1) & hash 定位桶下标。
3. 桶为空，直接插入。
4. 桶不为空，先比较头节点 key。
5. key 相同则覆盖。
6. key 不同，如果是树节点，走红黑树插入。
7. 否则遍历链表，找相同 key 覆盖，找不到尾插。
8. 链表长度达到阈值且数组容量足够时树化。
9. size 超过 threshold 后扩容。
```

### 10. HashMap 为什么容量是 2 的幂？

因为 HashMap 使用：

```java
(n - 1) & hash
```

来计算数组下标。容量是 2 的幂时，`n - 1` 的低位全是 1，位运算能高效完成取模，同时扩容时节点要么留在原位置，要么移动到 `原位置 + oldCap`，迁移效率更高。

### 11. HashMap 链表什么时候转红黑树？

JDK 8 中链表长度达到 8，并且数组容量达到 64 时，链表会转红黑树。

如果容量还小于 64，优先扩容而不是树化，因为此时冲突大概率是容量小导致的，扩容比树化更划算。红黑树节点更重，短链表遍历成本反而更低。

### 12. ConcurrentHashMap 在 JDK 8 中如何保证线程安全？

JDK 8 的 `ConcurrentHashMap` 使用 `CAS + synchronized + volatile`。

核心点：

- 数组初始化使用 CAS。
- 插入空桶时使用 CAS。
- 桶不为空时，对桶头节点加 `synchronized`。
- table、Node 的 value、next 等字段配合 volatile 保证可见性。
- 扩容时多线程协同迁移，通过 ForwardingNode 标记迁移状态。

它锁的是桶级别，不是整张表，所以并发性能较好。

### 13. JVM 内存区域有哪些？哪些容易 OOM？

JVM 运行时数据区包括：

- 程序计数器。
- Java 虚拟机栈。
- 本地方法栈。
- Java 堆。
- 方法区 / 元空间。
- 直接内存不属于 JVM 运行时数据区，但也常见 OOM。

常见 OOM：

- `Java heap space`：堆对象太多或泄漏。
- `Metaspace`：类加载过多或类加载器泄漏。
- `Direct buffer memory`：堆外内存不足。
- `unable to create new native thread`：线程数过多。

### 14. 如何判断一个对象是否可以被回收？

JVM 主要使用可达性分析。从 GC Roots 出发，如果一个对象不可达，就可以被判定为可回收候选对象。

GC Roots 常见包括：

- 虚拟机栈中局部变量引用的对象。
- 方法区中类静态变量引用的对象。
- 常量引用的对象。
- JNI 引用。
- 被同步锁持有的对象。

最终是否回收还和引用类型、`finalize` 等机制有关，但现代实践里重点就是看 GC Roots 引用链。

### 15. 线上 Full GC 频繁怎么排查？

我会先看指标和日志：

```text
jstat 看 FGC 是否持续增长
GC 日志看触发原因、回收前后 Old 区变化、暂停时间
```

如果 Full GC 后 Old 区下降不明显，优先怀疑内存泄漏或长生命周期对象过多，导出 heap dump，用 MAT 看 Dominator Tree 和 GC Roots。

如果是大对象或晋升过快，则排查大查询、大 JSON、批量导出、队列积压、新生代过小等问题。

不会一上来只调大堆，因为可能只是把 Full GC 延后，并让 STW 更长。

### 16. Spring Bean 生命周期说一下。

核心流程：

```text
BeanDefinition 注册
-> 实例化
-> 属性填充
-> Aware 回调
-> BeanPostProcessor 前置处理
-> 初始化方法
-> BeanPostProcessor 后置处理
-> 单例池可用
-> 容器关闭时销毁
```

AOP 代理通常在 BeanPostProcessor 后置处理阶段生成。容器里最终暴露出来的 Bean 可能不是原始对象，而是代理对象。

### 17. BeanFactory 和 ApplicationContext 有什么区别？

`BeanFactory` 是基础 IoC 容器，提供 Bean 创建、获取和依赖注入能力。

`ApplicationContext` 是更完整的应用上下文，除了 BeanFactory 能力，还提供：

- 自动注册 BeanPostProcessor。
- 国际化。
- 事件发布。
- 资源加载。
- Environment 和 Profile。
- Web 应用上下文能力。

日常业务开发基本使用 ApplicationContext。

### 18. Spring 如何解决循环依赖？三级缓存分别存什么？

Spring 主要解决单例 Bean 的属性注入循环依赖。

三级缓存：

```text
一级缓存 singletonObjects：完整初始化好的单例 Bean。
二级缓存 earlySingletonObjects：提前暴露的早期 Bean。
三级缓存 singletonFactories：用于生成早期引用的 ObjectFactory。
```

流程是：A 实例化后先把 ObjectFactory 放进三级缓存，A 注入 B 时创建 B，B 需要 A，就通过三级缓存拿到 A 的早期引用，完成 B 初始化，再回到 A 完成初始化。

构造器循环依赖无法解决，因为对象实例都还没创建出来，无法提前暴露引用。

### 19. Spring AOP 的实现原理是什么？

Spring AOP 本质是代理模式：

```text
调用方 -> 代理对象 -> 拦截器链 -> 目标方法
```

有接口时通常用 JDK 动态代理，没有接口时使用 CGLIB。AOP 代理通常由 `AnnotationAwareAspectJAutoProxyCreator` 这类 BeanPostProcessor 在 Bean 初始化后创建。

事务、缓存、异步等能力都依赖这个代理机制。

### 20. @Transactional 失效场景有哪些？

常见失效场景：

- 同类内部方法调用，没有经过代理。
- 方法不是 public。
- 异常被 catch 吞掉。
- 抛出 checked exception，但没配置 `rollbackFor`。
- 对象不是 Spring 管理的 Bean。
- 手动 new 对象。
- 多线程中事务上下文不传播。
- final 方法无法被 CGLIB 正常增强。

本质上，很多失效都是因为没走 Spring 代理，或者异常规则不匹配。

### 21. REQUIRED、REQUIRES_NEW、NESTED 有什么区别？

`REQUIRED` 是默认传播行为，有事务就加入，没有就新建。

`REQUIRES_NEW` 总是开启新事务，如果外层有事务，会先挂起外层事务，内层事务独立提交或回滚。

`NESTED` 是嵌套事务，基于数据库保存点。内层失败可以回滚到保存点，但仍属于外层物理事务的一部分。

我会在审计日志、失败记录这类需要独立提交的场景考虑 `REQUIRES_NEW`。

### 22. MyBatis Mapper 接口为什么不需要实现类？

因为 MyBatis 会为 Mapper 接口创建 JDK 动态代理。

调用 Mapper 方法时，实际进入 `MapperProxy`，它根据接口全限定名和方法名找到对应的 `MappedStatement`，再通过 `SqlSession`、`Executor`、`StatementHandler` 执行 SQL，最后通过 `ResultSetHandler` 映射结果。

### 23. MyBatis 一级缓存和二级缓存有什么区别？

一级缓存是 SqlSession 级别，默认开启。同一个 SqlSession 内相同查询可以命中一级缓存，执行写操作、commit、rollback、clearCache 会清空。

二级缓存是 namespace 级别，需要显式开启，可以跨 SqlSession，但一致性风险更高。

在 Spring 项目中，如果没有事务，每次 Mapper 调用可能使用不同 SqlSession，一级缓存存在感不强；同一事务内更可能复用 SqlSession。

互联网业务中二级缓存通常慎用，更倾向 Redis 或本地缓存并明确失效策略。

### 24. #{} 和 ${} 有什么区别？

`#{}` 使用 PreparedStatement 占位符，能防 SQL 注入，也能做类型处理。

`${}` 是字符串直接拼接，有 SQL 注入风险。

能用 `#{}` 就不用 `${}`。只有动态表名、动态列名、order by 字段这类场景可能用 `${}`，但必须做白名单校验。

### 25. MySQL MVCC 是怎么实现的？

InnoDB MVCC 主要依赖：

- 隐藏字段：事务 ID、回滚指针。
- Undo Log：保存历史版本。
- Read View：定义当前事务可见性规则。

Read View 关键字段包括：

- 当前活跃事务 ID 集合。
- 最小活跃事务 ID。
- 下一个将要分配的事务 ID。
- 当前事务 ID。

普通快照读通过 Read View 判断版本是否可见，不可见就沿 Undo Log 版本链向前找。

### 26. MySQL RR 隔离级别如何解决幻读？

InnoDB 在 RR 下对不同读有不同处理。

快照读通过 MVCC 和 Read View，让同一个事务中多次读取看到一致快照。

当前读，比如 `select for update`、`update`、`delete`，通过 Next-Key Lock，也就是记录锁 + 间隙锁，阻止其他事务在范围内插入新记录。

所以 InnoDB 的 RR 在大多数场景下能避免幻读。

### 27. redo log、undo log、binlog 分别解决什么问题？

`redo log` 是 InnoDB 的重做日志，保证事务持久性，崩溃恢复时用。

`undo log` 保存旧版本，用于事务回滚和 MVCC。

`binlog` 是 MySQL Server 层的逻辑日志，用于主从复制、数据恢复、订阅消费。

两阶段提交用于保证 redo log 和 binlog 的一致性。

### 28. Explain 中 type、key、rows、Extra 怎么看？

`type` 看访问类型，从好到差大致是 `const、eq_ref、ref、range、index、ALL`，重点警惕 `ALL`。

`key` 看实际使用的索引。

`rows` 看预估扫描行数。

`Extra` 重点关注 `Using filesort`、`Using temporary`、`Using index`、`Using index condition`。

慢 SQL 优化不能只看有没有索引，还要看扫描行数、返回行数、排序、临时表和数据分布。

### 29. Redis 分布式锁如何正确实现？

加锁使用：

```text
SET lock_key unique_value NX EX seconds
```

解锁必须校验 value，并用 Lua 保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

原因是锁必须有过期时间，防止服务宕机锁永不释放；value 必须唯一，防止误删别人的锁。

### 30. 缓存穿透、击穿、雪崩分别是什么？

缓存穿透：查询不存在的数据，每次都打到数据库。解决：空值缓存、布隆过滤器、参数校验。

缓存击穿：热点 key 过期，大量请求同时回源数据库。解决：互斥重建、逻辑过期、热点 key 预热。

缓存雪崩：大量 key 同时过期或 Redis 故障。解决：TTL 加随机值、多级缓存、限流降级、Redis 高可用。

### 31. Kafka 为什么吞吐量高？

Kafka 高吞吐主要来自：

- 顺序追加写磁盘。
- 利用 PageCache。
- 零拷贝 sendfile。
- 批量发送。
- 压缩。
- 分区并行。
- 顺序读写减少随机 IO。

它本质是把磁盘和网络的效率用到极致。

### 32. MQ 重复消费如何保证幂等？

MQ 通常保证至少一次投递，所以消费者必须幂等。

常见方案：

```text
业务唯一键 + 消费记录表唯一索引
```

处理流程：

```text
1. 开启本地事务。
2. 插入 messageId 或业务唯一键到消费记录表。
3. 插入成功则执行业务。
4. 插入失败说明处理过，直接 ack。
5. 消费记录和业务更新在同一事务中提交。
```

也可以结合状态机、唯一索引、乐观锁等手段。

### 33. RocketMQ 事务消息的半消息和事务回查是什么？

RocketMQ 事务消息解决本地事务和消息发送的一致性。

流程：

```text
1. Producer 发送半消息，Broker 存储但 Consumer 不可见。
2. Producer 执行本地事务。
3. 本地事务成功，提交消息；失败，回滚消息。
4. 如果 Producer 在二次确认前宕机，Broker 会回查 Producer 本地事务状态。
```

注意它解决的是生产端本地事务和消息发送一致性，不保证消费者业务一定成功，消费者仍然要幂等、重试和补偿。

## 第二轮：项目深挖 + 实战能力

### 1. 商家分账系统的核心领域模型是什么？

我会把核心模型分为四层：

```text
分账方案：描述规则和适用范围。
分账账单：根据业务流水和规则计算出的应分账结果。
结算单：按商户、门店、周期、渠道等维度汇总后的待结算对象。
转账单：面向资金渠道的实际付款或划拨指令。
```

链路是：

```text
业务流水 -> 匹配分账方案 -> 规则计算 -> 分账账单 -> 汇总结算单 -> 转账单 -> 支付渠道 -> 对账
```

这样设计可以把规则配置、账单计算、结算汇总和资金执行拆开，便于扩展新业务，也便于资金安全校验。

### 2. 分账规则模型怎么抽象？

我会把规则拆成：

```text
适用条件 + 计算因子 + 计算方式 + 生效周期 + 结算维度
```

例如阶梯提点规则：

```text
条件：品牌、门店、品类、时间周期
因子：营业收入
方式：阶梯比例
结果：总部应收管理费
```

固定比例、按数量提点、品项分账，本质都是不同计算方式。系统应避免为每个业务写一套硬编码，而是抽象为可配置规则和统一计算上下文。

### 3. 规则引擎是硬编码、配置化、表达式还是 DSL？为什么？

这类资金规则我倾向于“配置化规则模型 + 受控表达式/枚举计算器”，而不是完全自由 DSL。

原因：

- 资金场景要求可解释、可审计、可回放。
- 完全自由表达式灵活但风险高。
- 枚举计算器稳定，但扩展性差。
- 配置化模型能兼顾业务扩展和风险控制。

核心是把规则限制在受控范围内，并为每次计算保留规则版本、输入因子、输出结果和计算明细。

### 4. 历史烟囱式模型的问题是什么？

烟囱式模型的问题通常是：

- 每个业务场景一套表、一套代码、一套任务。
- 相同能力重复建设。
- 新增分账场景需要复制修改大量逻辑。
- 规则、账单、结算、转账耦合。
- 资金链路可追溯性差。
- 测试和回归成本高。

重构目标是抽象共性链路，沉淀统一模型，让新业务只扩展规则和配置，而不是复制一套系统。

### 5. 重构后如何支持新业务快速接入？

重构后我会让新业务接入变成：

```text
定义业务流水 -> 配置分账方案 -> 选择计算规则 -> 生成账单 -> 复用结算和对账链路
```

技术上通过统一的计算上下文、规则执行器、账单模型、结算模型和资金安全校验来复用能力。

这样新业务更多是在配置和少量扩展点上接入，而不是改动主流程。

### 6. 分账账单生成如何保证幂等？

核心是定义业务唯一键。

例如：

```text
业务流水 ID + 分账方案版本 + 分账对象 + 结算周期 + 账单类型
```

幂等手段：

- 数据库唯一索引兜底。
- 状态机控制账单流转。
- 重复计算时对比历史结果。
- 任务重试时保证同一输入生成同一账单。
- 计算规则版本固定，避免重算时规则漂移。

资金系统里幂等不能只依赖 Redis，最终要有数据库唯一约束和状态机兜底。

### 7. 结算单生成如何避免重复结算？

结算单幂等键一般包括：

```text
商户/门店 + 结算周期 + 结算类型 + 渠道 + 币种/账户 + 规则版本
```

生成时：

- 使用唯一索引防重复。
- 只允许从待结算账单汇总。
- 账单被结算后状态更新为已结算或绑定结算单 ID。
- 汇总和状态更新放在同一事务中。
- 重试时先查是否已存在结算单。

这样即使任务重复执行，也不会重复生成结算单。

### 8. 第三方支付超时但实际成功，系统如何修复状态？

这类问题不能简单按失败处理。

处理方式：

```text
1. 转账请求必须有唯一业务单号。
2. 调用超时后状态置为处理中或未知，不直接置失败。
3. 通过主动查询渠道结果、渠道回调、离线对账确认最终状态。
4. 如果渠道成功，本地推进为成功。
5. 如果渠道明确失败，再走重试或重新发起。
```

资金链路里“未知状态”很重要，不能因为接口超时就重复打款。

### 9. 金额计算如何处理精度问题？

金额计算必须使用 `BigDecimal`，不能用 float/double。

关键点：

- 使用字符串或整数分创建 BigDecimal，避免 double 精度问题。
- 明确 scale 和 rounding mode。
- 所有规则计算保留中间精度和最终落库精度。
- 分摊场景要处理尾差，比如最后一笔补差。
- 计算过程保留明细，方便审计和回放。

资金系统中最重要的是规则确定性和可追溯，不只是算出一个结果。

### 10. 一分钱误差怎么排查？

我会按链路排查：

```text
输入流水是否正确
分账方案和规则版本是否正确
计算因子是否正确
BigDecimal scale 和 rounding mode 是否一致
分摊尾差是否处理
账单汇总是否重复或漏算
结算单汇总是否一致
转账金额是否被渠道手续费或渠道规则影响
对账口径是否一致
```

资金问题要保留计算快照，包括输入、规则版本、计算明细、汇总过程和转账流水。

### 11. 资金安全模块实时链路和离线链路分别做什么？

实时链路偏过程拦截和快速发现：

- 金额非负校验。
- 单据状态流转校验。
- 幂等冲突校验。
- 账单和结算单金额一致性校验。
- 转账请求前置校验。
- 异常告警。

离线链路偏全量对账和兜底：

- 分账单和业务流水对账。
- 结算单和分账单对账。
- 转账单和支付渠道流水对账。
- 日终批量校验。
- 差异归因和补偿。

实时保障时效，离线保障完整性。

### 12. 对账差异后自动补偿和人工处理边界是什么？

能自动补偿的前提是：

- 差异类型明确。
- 修复动作确定。
- 不涉及重复打款或资金损失风险。
- 有幂等和审计。

例如状态同步延迟、回调丢失，可以自动查询渠道后推进状态。

需要人工处理的场景：

- 金额不一致。
- 重复结算。
- 渠道和本地口径冲突。
- 规则配置错误。
- 影响资金真实流向。

资金系统里自动补偿必须保守，宁可进入人工复核，也不能扩大损失。

### 13. 分账计算任务积压 10 倍如何定位？

我会先拆成四类：

```text
生产变快：上游业务量突增。
消费变慢：计算逻辑、数据库、下游依赖慢。
消费者异常：大量失败重试。
并行度不足：队列、分片、线程池不足。
```

排查指标：

- 待处理任务数和增长趋势。
- 单条计算耗时 P95/P99。
- 线程池 activeCount、queueSize、reject。
- 数据库慢 SQL 和连接池。
- MQ lag、重试队列。
- 规则复杂度是否变化。

止血上可以临时扩容消费者、限流上游、降级非核心校验、拆批处理，同时保证幂等，避免重复账单。

### 14. 核心资金链路如何做监控告警？

我会从业务、系统、资金三层监控。

业务指标：

- 待计算账单数。
- 结算单生成数。
- 转账成功率。
- 处理中单据数。
- 对账差异数。

系统指标：

- 接口 RT。
- 线程池。
- MQ lag。
- 数据库连接池。
- 慢 SQL。
- GC。

资金安全指标：

- 金额不平。
- 重复结算。
- 漏结算。
- 状态长时间卡住。
- 渠道回调异常。

告警要能定位到商户、周期、单据、金额和链路阶段。

### 15. 大表拆分为什么必须做？按什么字段拆？

这个要结合真实项目回答。通用回答是：

大表拆分通常是因为单表数据量、索引大小、查询扫描、写入压力、归档和 DDL 成本已经影响核心链路。

分片键选择原则：

- 高频查询条件。
- 数据分布均匀。
- 业务稳定。
- 能减少跨分片查询。

在分账系统里，可能的分片维度包括商户 ID、门店 ID、结算周期、账单 ID。最终要结合主要查询路径选择，比如商户维度查询多，就倾向商户 ID；按周期批处理多，也可能引入周期维度做冷热分离。

### 16. 历史数据如何迁移？双写如何保证一致性？

常见方案：

```text
1. 新老表结构准备。
2. 开启增量双写或记录变更日志。
3. 全量迁移历史数据。
4. 回放迁移期间增量数据。
5. 数据校验。
6. 灰度读切换。
7. 全量切流。
8. 保留回滚窗口。
```

双写一致性要注意：

- 双写失败要有重试和补偿。
- 写入顺序和幂等。
- 新老数据要有校验任务。
- 切读前必须做抽样和全量校验。

更稳的方式是通过 binlog 增量同步减少业务双写复杂度。

### 17. 分库分表后跨分片查询、分页、排序怎么处理？

优先避免跨分片查询，通过产品和接口设计让查询带分片键。

无法避免时：

- 建异构索引表。
- 使用 ES 或 OLAP 系统。
- 应用层归并多个分片结果。
- 限制深分页。
- 基于游标翻页。
- 报表类查询走离线数仓。

核心交易库不应该承担复杂跨分片报表查询。

### 18. 在线问诊核心链路是什么？

链路可以概括为：

```text
用户进入问诊
-> 导诊收集症状和基础信息
-> AI/规则分科
-> 创建问诊单
-> 医生调度匹配
-> 建立医患会话
-> 自动话术和消息通信
-> 医生诊疗/转诊/结束问诊
-> 状态归档和后续服务
```

关键系统能力是导诊、调度、会话、消息和医生状态管理。

### 19. 医生调度模块如何建模？

核心对象：

```text
问诊单：用户、症状、科室、业务类型、优先级。
医生：科室、处方权、在线状态、繁忙度、接诊能力。
供给池：不同医院、科室、业务类型下的医生集合。
调度规则：过滤条件 + 排序规则 + 分配策略。
```

流程：

```text
先过滤不可接诊医生，再按业务规则排序，最后分配或进入抢单池。
```

过滤条件包括科室、处方权、在线状态、业务资质；排序因素包括繁忙度、响应能力、优先级等。

### 20. 多个患者同时匹配同一个医生，如何避免并发分配冲突？

需要并发控制。

常见方式：

- 医生维度库存或接诊容量。
- Redis 原子扣减。
- 数据库乐观锁。
- 分布式锁。
- 状态机更新带条件。

例如：

```sql
update doctor_capacity
set used = used + 1
where doctor_id = ?
  and used < capacity;
```

影响行数为 1 才表示分配成功。失败则重新选择医生。

### 21. 医生抢单场景如何保证并发安全？

抢单本质是多个医生抢同一个问诊单。

处理方式：

```text
问诊单状态机 + 条件更新
```

示例：

```sql
update inquiry
set doctor_id = ?, status = 'ACCEPTED'
where inquiry_id = ?
  and status = 'WAIT_ACCEPT';
```

只有一个医生能更新成功。失败的医生提示已被抢。

同时要保证抢单动作幂等，避免医生端重复点击导致异常。

### 22. 会话状态机如何设计？

会话可以有：

```text
待建立 -> 进行中 -> 转诊中 -> 已转诊 -> 已结束 -> 已关闭
```

转诊时要处理：

- 旧会话结束或转出。
- 新会话建立。
- 消息归属关系。
- 医生工作台展示。
- 用户侧连续体验。

状态流转必须通过状态机控制，避免任意更新。

### 23. 医患消息如何保证不丢、不重、不乱序？

不丢：

- 消息先落库再投递。
- MQ 可靠投递。
- 发送失败重试。
- 客户端 ACK 或拉取补偿。

不重：

- messageId 唯一。
- 消费记录。
- 客户端去重。

顺序：

- 会话维度生成递增 sequence。
- 同一会话消息按 sequence 排序展示。
- MQ 发送时按会话 ID 做顺序 key。

如果完全强顺序成本高，至少要保证同一会话内展示顺序可恢复。

### 24. 离线 Push 如何做幂等？

离线 Push 要避免同一条消息重复推送。

幂等键可以是：

```text
messageId + doctorId + pushType
```

发送前写 push 记录，唯一索引防重。第三方推送超时时，不直接重复无限发送，而是查询状态或有限重试。

还要考虑医生重新上线后，Push 和站内消息状态的一致性。

### 25. 导诊从编码化到配置化，抽象了哪些配置？

可以抽象：

- 症状问题树。
- 科室映射规则。
- 用户回答选项。
- 问题跳转关系。
- 业务来源。
- 年龄、性别、药品、疾病条件。
- 推荐科室优先级。
- 兜底科室。

配置化后要有：

- 配置校验。
- 灰度发布。
- 版本管理。
- 回滚能力。
- 线上监控。
- 配置变更审计。

### 26. 配置化后如何防止错误配置导致线上事故？

我会做几层保护：

```text
1. 配置发布前静态校验：必填、环路、不可达节点、规则冲突。
2. 预览和测试环境验证。
3. 灰度发布，只影响小流量。
4. 配置版本化，可一键回滚。
5. 运行时兜底规则。
6. 配置命中率、异常率、转化率监控。
```

配置化不是把风险交给产品，而是把规则变更纳入工程治理。

### 27. AI 医患沟通小结如何设计链路？

链路：

```text
医患上下文
-> 数据清洗和脱敏
-> 上下文裁剪
-> Prompt 模板
-> 大模型调用
-> 结构化输出
-> 安全规则校验
-> 医生审核编辑
-> 入库和反馈采集
```

定位上不是让 AI 直接诊断，而是辅助医生提升效率。最终对用户展示的内容必须经过医生确认。

### 28. 如何控制大模型幻觉和医疗安全风险？

医疗场景必须保守。

措施：

- 明确模型只做辅助，不直接给诊断结论。
- Prompt 中约束不得编造上下文不存在的信息。
- 输出结构化，字段级校验。
- 高风险词和诊疗建议规则拦截。
- 医生审核后才能使用。
- 记录输入、输出、版本和医生编辑结果，便于追溯。

### 29. 如何评估 AI 效果？

从三类指标评估：

业务效率：

- 医生书写耗时下降。
- 小结生成耗时。
- 医生采纳率。

内容质量：

- 医生编辑率。
- 抽检准确率。
- 风险问题率。

系统稳定性和成本：

- 调用成功率。
- P95/P99 延迟。
- 降级比例。
- 单次调用成本。

### 30. 模型调用慢或不可用怎么办？

设计时不能让模型成为主链路单点。

措施：

- 设置严格超时。
- 降级到模板或不生成。
- 异步生成。
- 限流和熔断。
- 多模型路由。
- 失败重试但要有上限。
- 缓存相似结果或阶段性结果。

医生问诊主链路必须可用，AI 能力是增强项，不应阻塞核心流程。

## 第三轮：高级架构 + 稳定性 + 负责人视角

### 1. 如果重新设计分账系统，如何划分领域边界？

我会按领域职责划分：

```text
规则域：分账方案、规则版本、生效范围。
计算域：业务流水解析、规则执行、账单生成。
结算域：周期汇总、结算单生成、结算状态流转。
资金域：转账请求、渠道交互、支付状态。
安全域：实时校验、离线对账、差异处理。
查询域：面向商户和运营的报表查询。
```

核心原则是：规则、计算、结算、资金执行、对账各自独立，但通过单据 ID、状态机和审计日志串起来。

### 2. 分账系统哪些流程同步，哪些异步？

同步适合：

- 商户配置校验。
- 小规模试算。
- 关键状态查询。

异步适合：

- 大批量账单计算。
- 周期结算。
- 转账结果同步。
- 对账。
- 补偿任务。

资金链路中用户请求不应该长时间阻塞等待批量计算，适合通过任务化、消息化、状态机推进。

### 3. 如何平衡吞吐量、准确性和可追溯性？

资金系统里准确性和可追溯性优先级高于极致吞吐。

做法：

- 计算异步化提升吞吐。
- 单据状态机保证过程可控。
- 幂等和唯一索引防重复。
- 计算明细和规则版本留痕。
- 对账和补偿兜底。
- 高风险动作人工复核。

吞吐可以通过分片、并行、批处理提升，但不能牺牲资金准确性。

### 4. 如何设计通用清结算规则模型？

核心抽象：

```text
规则主体：谁向谁收。
适用范围：品牌、门店、品类、渠道、时间。
计算因子：营业额、订单数、优惠券数、固定金额。
计算方式：固定比例、阶梯比例、固定金额、按数量。
结算周期：日、周、月、自定义。
规则版本：保证历史可追溯。
```

每次计算要保存：

```text
输入因子 + 规则版本 + 计算过程 + 输出结果
```

否则后续对账和客诉无法解释。

### 5. 高风险重构如何保证不影响线上资金准确性？

我会采用：

```text
新老链路并行
-> 影子计算
-> 结果对比
-> 小流量灰度
-> 分批切流
-> 实时监控
-> 可回滚
```

对于资金系统，重构上线前要让新链路先跑一段时间影子账单，与旧链路结果做差异对比。只有差异收敛且解释清楚后，才逐步切流。

### 6. 线上 CPU 飙高怎么定位？

流程：

```text
top 找高 CPU 进程
top -Hp 找高 CPU 线程
线程 ID 转 16 进制
jstack 搜 nid
连续采样判断线程栈
```

如果是业务线程 RUNNABLE，可能是死循环、热点计算、序列化、正则等。  
如果是 GC 线程高，结合 jstat 和 GC 日志。  
如果大量 BLOCKED，看锁竞争。

### 7. Java 堆 OOM 怎么分析？

先确认 OOM 类型。如果是 `Java heap space`：

```text
1. 保留 heap dump。
2. 用 MAT 看 Leak Suspects 和 Dominator Tree。
3. 找 Retained Heap 最大对象。
4. 看 Path to GC Roots。
5. 判断是缓存、ThreadLocal、队列积压、大对象还是集合泄漏。
```

修复上不是只调大内存，而是优化对象生命周期、缓存上限、批量处理方式和监控告警。

### 8. unable to create new native thread 可能是什么原因？

这是创建本地线程失败，常见原因：

- 线程数过多。
- 线程池失控。
- 每个请求创建线程。
- 定时任务重复创建线程池。
- 操作系统线程限制。
- `-Xss` 过大导致可创建线程数减少。

排查：

```text
ps -eLf | grep pid | wc -l
jstack 查看线程来源
ulimit 查看系统限制
```

### 9. 线程池打满如何判断是线程数不够还是下游慢？

看四类指标：

```text
activeCount
queueSize
rejectedCount
taskCostTime
```

再看 jstack 里线程卡在哪里。

如果大量线程卡在数据库、RPC、Redis、HTTP 调用上，根因通常是下游慢，简单加线程只会把下游打得更严重。

正确处理是先限流降级止血，再优化下游、设置超时、熔断、线程池隔离和有界队列。

### 10. MQ 消费堆积如何止血？

先判断：

```text
生产变快、消费变慢、消费者报错、下游慢、并行度不足
```

止血手段：

- 临时扩容消费者。
- 增加消费线程数。
- 跳过或隔离脏数据。
- 下游扩容。
- 关闭非核心逻辑。
- 批量消费和批量写。

注意 Kafka 受 partition 数限制，RocketMQ 受 MessageQueue 数限制，单纯加机器不一定提升并行度。

### 11. 慢 SQL 排查步骤是什么？

流程：

```text
慢查询日志定位 SQL
-> Explain 看 type、key、rows、Extra
-> 看索引是否命中
-> 看扫描行数和返回行数
-> 看是否 Using filesort / temporary
-> 看锁等待和大事务
-> 优化索引或改写 SQL
-> 压测验证
```

如果执行计划看起来没问题但仍然慢，要排查锁等待、IO、网络、大事务和返回数据量。

### 12. Redis 热点 Key 如何治理？

方案：

- 本地缓存 + Redis。
- 热点 key 副本拆分。
- 逻辑过期异步刷新。
- 请求合并。
- 限流。
- 预热。

如果是热点读，可以用本地缓存降低 Redis 压力；如果是热点写，要考虑拆分 key 或改变业务模型。

### 13. 核心接口 RT 从 100ms 升到 3s 怎么排查？

我会按链路分层：

```text
入口流量是否突增
应用线程池是否打满
数据库慢 SQL 或连接池是否满
Redis 是否慢
RPC 下游是否慢
MQ 是否堆积
GC 是否异常
锁竞争是否严重
```

工具上结合 APM trace、日志 traceId、线程栈、数据库监控、Redis latency、GC 日志。

先定位耗时在哪一段，再做针对性处理。

### 14. 资金系统幂等键如何设计？

幂等键必须有业务语义。

不同链路不同幂等键：

- 账单生成：业务流水 ID + 规则版本 + 分账对象。
- 结算单：商户/门店 + 周期 + 渠道 + 结算类型。
- 转账请求：结算单 ID + 收款账户 + 渠道。
- 支付回调：渠道流水号或支付单号。

不能所有链路都用随机 UUID，否则无法识别业务重复。

### 15. 本地事务和 MQ 消息发送如何保证一致性？

两种常见方案：

本地消息表：

```text
业务数据和消息表在同一个本地事务中提交，后台任务可靠投递 MQ。
```

RocketMQ 事务消息：

```text
先发半消息，执行本地事务，再提交或回滚消息，Broker 通过事务回查兜底。
```

无论哪种，消费者都必须幂等，因为消息仍可能重复投递。

### 16. 对账系统如何设计？

对账要明确：

- 对账对象。
- 对账周期。
- 对账口径。
- 差异类型。
- 修复流程。

分账系统可以分层对账：

```text
业务流水 vs 分账账单
分账账单 vs 结算单
结算单 vs 转账单
转账单 vs 渠道流水
本地状态 vs 渠道状态
```

输出差异后要有差异归因、风险等级、自动补偿或人工处理流程。

### 17. 发现少结算一批商户，如何修复？

先止血：

```text
暂停相关结算任务或冻结问题批次，避免扩大影响。
```

再定位：

- 是否规则配置问题。
- 是否任务漏跑。
- 是否分片遗漏。
- 是否状态卡住。
- 是否数据迁移问题。

修复：

- 生成补结算批次。
- 保证补偿幂等。
- 通知相关方。
- 对全量商户做扫描校验。
- 补充监控防止复发。

资金问题要保留修复记录和审计。

### 18. 发现重复结算，如何止血和回滚？

立即：

- 暂停结算和转账任务。
- 冻结相关批次。
- 确认是否已经出款。

如果未出款，直接拦截重复转账单。  
如果已出款，需要走资金追回、冲正、抵扣或人工处理流程。

技术上排查幂等键、唯一索引、状态机、任务重试、渠道回调处理是否有漏洞。

### 19. 作为系统负责人，如何拆解复杂需求？

我的方式是：

```text
先明确业务目标和边界
-> 拆领域对象和状态流
-> 梳理上下游依赖
-> 识别核心风险
-> 设计技术方案
-> 拆分里程碑
-> 明确灰度、监控、回滚
```

复杂需求不能只拆接口和表，还要拆风险、数据一致性、上线策略和验收指标。

### 20. 技术方案评审重点关注哪些风险？

我重点看：

- 业务边界是否清楚。
- 数据模型是否可扩展。
- 幂等和事务是否完整。
- 异常和补偿是否闭环。
- 容量是否评估。
- 灰度和回滚是否可行。
- 监控告警是否覆盖。
- 是否影响历史数据。
- 是否有安全和合规风险。

资金类项目还要特别关注金额准确性、对账和审计。

### 21. 如何推动跨团队项目落地？

我会先建立共同目标和明确责任边界：

- 梳理上下游接口和数据契约。
- 明确排期和里程碑。
- 识别阻塞点。
- 定期同步风险。
- 关键方案评审留痕。
- 联调、压测、灰度计划提前确定。

跨团队项目最大风险不是代码，而是边界不清和节奏不一致。

### 22. 如何安排重构和业务迭代优先级？

我会看三个因素：

```text
业务风险、技术债影响、收益确定性
```

如果技术债已经影响资金准确性、系统稳定性或迭代效率，就不能无限后置。

方式上尽量避免“大爆炸重构”，而是结合业务迭代做渐进式重构：

```text
抽模型 -> 建新链路 -> 影子运行 -> 灰度切流 -> 下线旧逻辑
```

### 23. 线上故障后如何复盘？

复盘不是追责，而是建立系统改进。

我会从：

- 事故时间线。
- 影响范围。
- 直接原因。
- 根因。
- 发现机制。
- 应急处理。
- 恢复过程。
- 预防措施。
- 责任人和完成时间。

最后要落到监控、流程、代码、测试、预案的改进，而不是只写“以后注意”。

### 24. 资深 Java 开发和普通 Java 开发最大的区别是什么？

普通开发更关注功能实现，资深开发要对系统结果负责。

区别体现在：

- 能抽象业务模型。
- 能识别架构风险。
- 能设计稳定性和一致性方案。
- 能处理线上复杂问题。
- 能推动跨团队落地。
- 能在速度、成本、风险之间做取舍。

结合我的经历，比如分账系统不仅要实现规则计算，还要保证资金准确、可追溯、可对账、可补偿，这就是资深开发需要承担的系统责任。

## 最重要的 10 个项目回答模板

### 1. 分账系统一句话介绍

这是一个面向连锁品牌商户的清结算和资金分账系统，核心能力是根据商户配置的分账方案和业务流水，生成分账账单、结算单并完成转账，同时通过实时校验和离线对账保障资金准确性。

### 2. 分账系统技术难点

难点不在 CRUD，而在：

```text
规则复杂、资金准确性要求高、幂等链路长、历史模型重、账单数据量大、对账补偿必须闭环。
```

### 3. 大表拆分一句话

大表拆分不是简单换表，而是围绕分片键、历史迁移、增量同步、数据校验、灰度切流、回滚预案的一整套治理。

### 4. 资金准确性一句话

资金准确性靠的不是单点机制，而是：

```text
精确计算 + 幂等 + 状态机 + 唯一约束 + 实时校验 + 离线对账 + 补偿闭环。
```

### 5. 在线问诊一句话

在线问诊系统核心是把导诊、问诊创建、医生调度、会话管理、消息通信和医生工作状态串成稳定链路，在保证用户体验的同时提升医生供给利用率。

### 6. 医生调度一句话

医生调度本质是过滤 + 排序 + 并发分配：先过滤不符合科室、处方权、在线状态的医生，再按繁忙度和业务优先级排序，最后通过容量扣减或状态机保证并发安全。

### 7. 消息可靠性一句话

医患消息要做到先落库再投递，messageId 幂等，失败可重试，客户端可拉取补偿，同一会话通过 sequence 恢复展示顺序。

### 8. AI 项目一句话

我在 AI 项目里的价值不是简单接模型接口，而是把大模型能力安全、稳定、可评估、可降级地嵌入真实业务链路。

### 9. 线上排障一句话

线上排障要先定位层次：应用线程、GC、数据库、缓存、MQ、下游依赖，再用指标、日志、trace、线程栈和 dump 交叉验证。

### 10. 技术负责人一句话

技术负责人要把需求拆成模型、链路、风险、计划和结果，不仅保证功能上线，还要保证系统长期可维护、可扩展、可观测、可回滚。
