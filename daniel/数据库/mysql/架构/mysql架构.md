### 架构图中的 3 个隐藏“技术风骨”：

1. **2PC（两阶段提交）的物理跨度：**
   请注意图中的 **Binlog** 和 **Redo Log (ib_logfile)**。Binlog 独立悬挂在 Server 层右侧，而 Redo Log 深埋在 InnoDB 磁盘区。当执行器下达提交指令时，数据流是先冲入引擎层的 Redo（标记为 Prepare），再冲入 Server 层的 Binlog，最后再回流把 Redo 改为 Commit。这种跨图层的握手，保障了主从和崩溃恢复的强一致性。
2. **为什么不直接写 `.ibd` 数据文件？**
   从图中可以看到，Buffer Pool 中的数据页被修改后，直接去往磁盘 `.ibd` 的链路是一条“异步刷脏”的虚线（Master Thread 异步慢慢干）。在核心交易链路中，只要右侧的 Redo Log 顺序追加成功了，MySQL 就敢给客户端返回成功。内存的读写隔离屏蔽了磁盘随机 I/O 的噩梦。
3. **MVCC 的前世因缘：**
   内存 Buffer Pool 里的数据页在变脏时，它吐出的旧版本数据会被丢进 **Undo Log Buffer**，并最终持久化到磁盘的 Undo 共享表空间。这也就是为什么在并发极高、且有长事务未提交时，Undo 磁盘空间会暴涨的架构原因。
