
在实际开发中，`EXPLAIN` 通常会输出 12 个字段。为了方便记忆和实战排查，我习惯把它们分为**四大阵营**：执行逻辑阵营、性能定性阵营、索引详情阵营、以及预估代价阵营。

---

### 一、 执行逻辑阵营（看懂 SQL 的拆解步骤）

这部分主要告诉你，MySQL 内部是怎么把你写的一坨复杂 SQL 拆解成一步步去执行的。

- **`id`**
    
    - **含义**：`SELECT` 查询的序列号，代表整个查询中各个子查询或操作的执行顺序。
        
    - **实战规则**：
        
        1. `id` 相同：执行顺序由上至下。
            
        2. `id` 不同：如果是子查询，`id` 的序号会递增，**`id` 值越大的优先级越高，越先被执行**。
            
        3. `id` 为 `NULL`：通常出现在 `UNION` 结果的合并阶段，最后执行。
            
- **`select_type`**
    
    - **含义**：查询的类型，主要用于区分普通查询、联合查询、子查询等复杂结构。
        
    - **常见类型**：
        
        - `SIMPLE`：极其简单的 SELECT，不包含子查询或 UNION。
            
        - `PRIMARY`：如果包含复杂的子查询，最外层的那个 SELECT 会被标记为 PRIMARY。
            
        - `SUBQUERY`：在 SELECT 或 WHERE 列表中包含了子查询。
            
        - `DERIVED`：在 FROM 列表中包含的子查询（MySQL 会把它当成一张派生出来的临时表）。
            
        - `UNION`：UNION 中的第二个或后面的 SELECT 语句。
            
- **`table`**
    
    - **含义**：这一行数据正在访问哪张表。可以是真实的表名、表的别名，也可以是 `<derivedN>`（代表引用了 `id` 为 N 的派生表）。
        
- **`partitions`** (MySQL 5.7 之后默认展示)
    
    - **含义**：如果表使用了分区，这里显示匹配的分区。通常为 `NULL`。

### 二、 性能定性阵营（调优的绝对核心）

这是我们看 `EXPLAIN` 时**第一眼必须盯住**的地方，决定了这条 SQL 是“丝滑”还是“拉胯”。

- **`type` (访问类型)**
    
    - **含义**：MySQL 决定如何查找表中的行。这是判断查询性能最直观的指标。
        
    - **性能从优到劣依次为**（面试必考）：
        
        - **`system`**：表中只有一行数据（系统表特例），基本遇不到。
            
        - **`const`**：通过主键或唯一索引进行等值匹配，最多查到 1 条数据。快如闪电。
	        ![[Pasted image 20260401154116.png]]
            
        - **`eq_ref`**：常见于多表 JOIN。对于前表的每一行，当前表**最多只有一条**记录匹配（通常是 JOIN 条件为主键或唯一索引）。
            ![[Pasted image 20260401154152.png]]
        - **`ref`**：使用了非唯一索引进行等值匹配。可能会查出多条行，但依然很快。
            ![[Pasted image 20260401154212.png]]
        - **`range`**：利用索引进行范围查找（如 `BETWEEN`、`>`、`<`、`IN`）。及格线。
            ![[Pasted image 20260401154231.png]]
        - **`index`**：全索引扫描（Full Index Scan）。遍历整棵索引树，通常是因为要查的字段都在索引里（覆盖索引），不用回表，所以比 ALL 快一点，但也挺慢的。
            
        - **`ALL`**：全表扫描（Full Table Scan）。灾难级别，直接遍历磁盘上的数据页，必须优化！
            
- **`Extra` (额外信息)**
    
    - **含义**：包含不适合在其他列中显示，但对执行计划非常重要的额外细节。
        
    - **常见关键类型**：
        
        - **`Using index` (极好)**：使用了**覆盖索引**！你要查的所有列都在当前这棵二级索引树上，不需要回表查聚簇索引，性能极佳。
            
        - **`Using index condition` (不错)**：使用了**索引下推 (ICP)** 技术。MySQL 把部分过滤动作下推到了存储引擎层完成，减少了回表次数。
            
        - **`Using where` (中性)**：在存储引擎把行返回给 Server 层后，Server 层又用 WHERE 条件进行了过滤。
            
        - **`Using temporary` (极差)**：MySQL 需要创建一张内部临时表来处理查询。常见于复杂的 `GROUP BY` 或 `ORDER BY`。极其消耗资源！
            
        - **`Using filesort` (极差)**：无法利用索引完成排序，只能把数据捞到内存（甚至磁盘）里用排序算法重排。必须通过调整联合索引来消除！

### 三、 索引详情阵营（看它用对了没）

- **`possible_keys`**
    
    - **含义**：MySQL 优化器在解析 SQL 时，认为**可能**会用到的索引列表。如果为空，说明没有相关索引。
        
- **`key`**
    
    - **含义**：MySQL **实际决定**使用的索引。
        
    - **老兵点评**：如果 `possible_keys` 有值，但 `key` 是 `NULL`，说明 MySQL 经过成本计算后，觉得走全表扫描比走索引还快（通常因为表太小，或者你要查的数据占了全表的绝大比例，回表成本太高）。
        
- **`key_len`**
    
    - **含义**：实际使用的索引字节数长度。
        
    - **老兵点评**：这个字段在判断**联合索引**到底生效了几个字段时极其有用！算一算字节长度，就能精准知道 MySQL 到底用到了联合索引的哪几列。
        
- **`ref`**
    
    - **含义**：显示哪个列或常量（const）被用来与 `key` 中指定的索引进行比对。如果是等值查询，这里通常是 `const`；如果是 JOIN，这里会显示前表的关联列。

### 四、 预估代价阵营（看数据的规模）

- **`rows`**
    
    - **含义**：MySQL 根据统计信息，**预估**必须要扫描的行数。
        
    - **老兵点评**：这不是最终结果集的行数，而是找到这些结果需要翻多少条记录！这个值越小，说明索引的区分度越高，查询越精准。
        
- **`filtered`**
    
    - **含义**：通过过滤条件之后，满足条件的记录数量占预估扫描行数（`rows`）的**百分比**。
        
    - **老兵点评**：最大值是 100（代表全都没被过滤掉）。在多表 JOIN 时，`rows * (filtered / 100)` 就是前表传递给后表的实际行数，这个值直接决定了 JOIN 的成本。

以上就是 `EXPLAIN` 的全景图。刚才在“索引详情阵营”中，我特别提到了可以通过 **`key_len`（索引长度）** 来精准反推联合索引的命中情况。
