其实三表关联（甚至更多表的关联），在 MySQL 底层的执行逻辑和两表关联并没有本质的区别。它本质上就是**两两关联的接力赛**。

MySQL 并没有什么神奇的“三表同时合并”算法。它的处理逻辑是：**先找出两张表进行 JOIN，生成一个“虚拟的中间结果集”，然后拿这个中间结果集作为新的驱动表，再去和第三张表进行 JOIN。**

理解了这个俄罗斯套娃一样的逻辑，索引的创建思路就非常清晰了。我们用一个相对宽泛的常见模型来拆解：

### 1. 场景设定

假设我们有三张表：

- `users`（用户表）
    
- `orders`（订单表）
    
- `order_items`（订单商品明细表）
    

SQL 是这样的：查出“张三”买过的所有商品明细。

SQL

```
SELECT u.name, o.order_no, oi.product_name
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
WHERE u.name = '张三';
```

### 2. 核心执行流与索引布置

在这个查询中，优化器通常会这样规划“接力”路线：`users` $\rightarrow$ `orders` $\rightarrow$ `order_items`。

- **第一棒：驱动表 (`users`)**
    
    - **动作**：MySQL 最先访问这张表，试图尽快过滤数据，把基数降下来。
        
    - **建索引策略**：在 `WHERE` 过滤条件上建索引。
        
    - **操作**：`CREATE INDEX idx_name ON users(name);`
        
- **第二棒：第一张被驱动表 (`orders`)**
    
    - **动作**：MySQL 拿着从 `users` 表里找到的张三的 `id`，去 `orders` 表里找对应的订单。
        
    - **建索引策略**：在关联字段上建索引。
        
    - **操作**：`CREATE INDEX idx_user_id ON orders(user_id);`
        
- **第三棒：第二张被驱动表 (`order_items`)**
    
    - **动作**：MySQL 拿着前两步合并出的结果（包含刚才查出的订单 `id`），去 `order_items` 表里找对应的商品明细。
        
    - **建索引策略**：同样，在关联字段上建索引。
        
    - **操作**：`CREATE INDEX idx_order_id ON order_items(order_id);`
        

### 3. 底层逻辑：嵌套循环的放大效应

为什么要严格按照这个套路建索引？因为 MySQL 处理 JOIN 默认走的是**嵌套循环（Nested-Loop Join）**。

你可以把它抽象成写代码时的三层 `for` 循环：

Java

```
for (User u : users) { // 第1层：走 idx_name 快速找到张三
    for (Order o : orders) { // 第2层：走 idx_user_id 快速找到该用户的订单
        for (OrderItem oi : order_items) { // 第3层：走 idx_order_id 快速找订单明细
            // 组装行数据返回
        }
    }
}
```

如果在第二层或第三层的关联字段上没有建索引，内层的循环就会变成全表扫描，整体扫描次数会呈指数级爆炸，这就是多表关联极易引发数据库 CPU 飙升的根本原因。

### 总结

不论是三表还是五表，打法思路都是一致的：

1. 梳理出优化器大致的 **JOIN 顺序**（谁是起点，谁是链路的下一站）。
    
2. 为**起点表**的 `WHERE` 条件建索引。
    
3. 为后面**所有被驱动表**的 `ON` 关联字段建索引。
    

按照这种宽泛而统一的逻辑去排查和建索引，就能最大限度地支撑效能，优化整个数据检索的保障流程。
