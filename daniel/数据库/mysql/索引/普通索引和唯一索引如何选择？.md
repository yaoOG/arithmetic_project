
在建表时，到底是用普通索引还是唯一索引，很多新手会觉得：“只要业务上这个字段不会重复，就建唯一索引，因为查询快。”

但作为一个在生产环境中排查过数据库性能瓶颈的老兵，我的观点是：**查询性能的差异微乎其微，真正的天壤之别在于“更新（写）性能”。**

这背后涉及到了 InnoDB 极为核心的一个内存组件——**Change Buffer（写缓冲）**。我们把这个问题拆解到最底层的内存和磁盘 I/O 层面来看：

### 一、 查询性能对比：几乎没有差别

假设我们执行 `SELECT id FROM users WHERE id_card = '123456';`

- **对于普通索引**：查找到第一条符合条件（123456）的记录后，还需要继续往下查找下一条记录，直到遇到第一个不等于 123456 的值为止。
    
- **对于唯一索引**：因为具有唯一性，查找到第一个匹配的记录后，就会立刻停止检索。
    

**老兵剖析**：这看起来普通索引多做了一次查找，是不是慢很多？**其实不是。**

因为 InnoDB 的数据是按“页（Page，默认 16KB）”为单位读写的。当找到第一条记录时，它所在的整个数据页大概率已经被加载到内存里了。普通索引多做的那一步“查找下一条”，仅仅是在内存里移动一下指针进行一次 CPU 计算而已，耗时对于数据库层面来说完全可以忽略不计。

### 二、 更新性能对比：天壤之别（核心分水岭）

当执行 `UPDATE` 或 `INSERT` 时，两者的性能差距就被无限放大了。

- **普通索引的“开挂”逻辑（极快）**：
    
    当你要更新一条记录时，如果这条记录所在的数据页**不在内存中**，InnoDB 不会立刻去磁盘里把页读出来（磁盘随机 I/O 太慢了）。它会把这次更新操作直接记录在内存的 **Change Buffer** 中，然后立刻给客户端返回执行成功！
    
    等到下一次有其他查询操作把这个数据页读进内存时，或者后台线程定期触发时，系统才会把 Change Buffer 中的操作“合并（Merge）”到真实的数据页上。这就把昂贵的磁盘随机写，优化成了极其高效的内存操作。
    
- **唯一索引的“无奈”逻辑（极慢）**：
    
    唯一索引**绝对不能**使用 Change Buffer！
    
    因为当你插入或更新一条数据时，数据库必须判断这个新值有没有违反唯一性约束。怎么判断？**必须把数据从磁盘读到内存里，实打实地校验一遍！** 这一步操作强制触发了一次磁盘的随机 I/O。如果高并发写入的都是唯一索引，磁盘瞬间就会成为巨大的瓶颈。
    

### 三、 老兵实战选型指南（到底怎么选？）

既然普通索引的写入性能远高于唯一索引，那我们是不是无脑用普通索引就行了？绝对不是。在架构设计时，我们遵循以下三个铁律：

#### 1. 业务正确性永远高于底层性能（首选唯一索引）

如果业务要求这个字段绝对不能重复（比如：用户的手机号、身份证号、订单号），**必须毫不犹豫地建唯一索引**。

很多开发同学过于自信，觉得在 Java 代码里加了校验就能防重。但在线上高并发场景下，如果没有唯一的分布式锁，代码防重一定会失效（脏读问题），最终产生脏数据。修复脏数据的代价，远远高于牺牲一点点数据库的写入性能。

#### 2. 如果业务不要求唯一，坚决用普通索引

比如用户的“姓名”、“年龄”、“所属部门ID”，或者一些操作日志表中的“操作人ID”。这些字段天然允许重复，并且有的表（如日志表）写入并发量极大，使用普通索引配合 Change Buffer，能让数据库的吞吐量提升一个数量级。

#### 3. 极端高并发下的“折中方案”

如果你的业务既要求唯一，同时写入并发量又到了极其变态的地步（比如秒杀系统的流水表），导致唯一索引带来的磁盘 I/O 数据库根本扛不住怎么办？

这时候大厂通常的做法是：**数据库降级为普通索引**（或者干脆去掉所有的非主键唯一约束以换取极限性能），把防重的工作向上层转移，依赖 **Redis 分布式锁** 或者是业务架构前置的 **唯一发号器（如雪花算法生成唯一请求 ID 并缓存）** 来保证。

**总结**：

一句话概括就是——**如果业务需要保证唯一，请把防重的底线交给唯一索引；如果业务只为了加速查询且没有防重需求，请放心地把性能交给普通索引和它的 Change Buffer。**
