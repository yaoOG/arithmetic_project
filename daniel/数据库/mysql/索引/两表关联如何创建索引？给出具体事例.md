
在处理两表关联（JOIN）时，索引的创建直接决定了查询是“毫秒级返回”还是“拖垮整个数据库”。

要给两表关联创建完美的索引，核心思想只有八个字：**“小表驱动大表，索引建在后表”**。在 MySQL 的嵌套循环连接（Nested-Loop Join）算法中，索引的布局需要遵循以下三个黄金准则：

---

### 一、 核心准则

1. **驱动表（左表）**：主要在 `WHERE` 过滤条件列上建索引，用于快速定位结果集。
    
2. **被驱动表（右表）**：必须在 **关联字段（JOIN ON 的列）** 上建索引。这是性能提升最明显的地方。
    
3. **覆盖索引**：如果可能，尽量将 `SELECT` 的字段也包含进索引中，避免回表。
    

---

### 二、 具体案例：用户与订单查询

假设我们有一个电商场景，需要查询“张三”这个用户的所有订单信息。

#### 1. 表结构准备

SQL

```
-- 用户表 (小表)
CREATE TABLE `users` (
  `id` INT PRIMARY KEY AUTO_INCREMENT,
  `name` VARCHAR(50),
  `city` VARCHAR(50)
) ENGINE=InnoDB;

-- 订单表 (大表)
CREATE TABLE `orders` (
  `id` INT PRIMARY KEY AUTO_INCREMENT,
  `user_id` INT,
  `order_no` VARCHAR(50),
  `amount` DECIMAL(10,2),
  `status` TINYINT
) ENGINE=InnoDB;
```

#### 2. 目标查询 SQL

SQL

```
SELECT u.name, o.order_no, o.amount 
FROM users u 
INNER JOIN orders o ON u.id = o.user_id 
WHERE u.name = '张三';
```

---

### 三、 索引创建步骤（实战演示）

在这个查询中，MySQL 优化器通常会选择 `users` 作为驱动表（因为有 `name` 过滤条件，结果集小），`orders` 作为被驱动表。

#### 第一步：给驱动表的过滤条件建索引

为了快速找到“张三”，我们需要在 `users` 表的 `name` 字段建索引。

SQL

```
CREATE INDEX idx_name ON users(name);
```

#### 第二步：给被驱动表的关联字段建索引（最关键！）

MySQL 拿到“张三”的 `id` 后，要去 `orders` 表里找对应的订单。如果没有索引，`orders` 表会对每一行进行扫描。

SQL

```
CREATE INDEX idx_user_id ON orders(user_id);
```

#### 第三步：进阶优化——创建联合索引（实现覆盖索引）

如果我们希望性能达到极致，可以针对 `orders` 表创建一个联合索引，把关联字段和查询字段都包进去：

SQL

```
-- 这样连回表拿 order_no 和 amount 的动作都省了
CREATE INDEX idx_user_id_order_info ON orders(user_id, order_no, amount);
```

---

### 四、 如何验证索引是否生效？

执行 `EXPLAIN` 查看执行计划：

- **第一行（驱动表 users）**：`type` 应该是 `ref`，`key` 应该是 `idx_name`。这代表我们精准定位到了张三。
    
- **第二行（被驱动表 orders）**：`type` 应该也是 `ref`（如果主键关联则是 `eq_ref`），`key` 应该是 `idx_user_id`。
    
- **关键指标**：看第二行的 `rows`。如果建立了索引，这个值应该是个很小的常数；如果没建索引，这个值会是 `orders` 表的总行数，意味着发生了极其恐怖的磁盘扫描。
    

---

### 五、 老兵排坑指南

1. **关联字段类型必须一致**：如果 `u.id` 是 `INT`，而 `o.user_id` 是 `VARCHAR`，MySQL 会因为隐式类型转换导致 **索引失效**，引发全表扫描。
    
2. **字符集必须一致**：如果一个表是 `utf8mb4`，另一个是 `utf8`，关联时同样会索引失效。
    
3. **消除 Using join buffer (Block Nested Loop)**：如果你在 `Extra` 列看到了 `Using join buffer`，说明被驱动表的关联字段没有命中索引。这在高并发下是绝对不允许的。
    

**总结**：两表关联时，请务必保证 **被驱动表的关联列** 有索引。如果查询中有针对驱动表的过滤条件，该条件也应该有索引。
