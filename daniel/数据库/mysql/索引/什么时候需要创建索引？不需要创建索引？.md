
在数据库架构设计中，有一句非常经典的行话：“**索引就像是强心剂，用对了起死回生，用滥了七窍流血。**”

很多初级开发喜欢“遇事不决建索引”，只要是个查询条件就往上加，最后导致线上数据库写性能崩盘。其实，建与不建的核心衡量标准只有一个：**查询带来的收益，是否能覆盖底层 B+ 树维护带来的空间和时间成本？**

我通常会把这些场景划分为明确的“红黑榜”，在团队内部的代码审查（Code Review）中严格执行：

### 一、 必须要建索引的场景（红榜：好钢用在刀刃上）

在以下几种高频、刚需的场景下，不建索引系统绝对扛不住：

1. **高频的查询条件（WHERE 核心列）**：
    
    这是最基本的常识。对于业务中频繁用于过滤数据的列（如 `user_id`、`order_no`、`create_time`），必须建立索引。
    
2. **多表关联的字段（JOIN 的 ON 条件）**：
    
    在执行多表 JOIN 时，MySQL 默认会采用 Nested-Loop Join（嵌套循环连接）算法。如果被驱动表（后表）的关联字段没有索引，那每一次前表的循环，都会导致后表进行一次全表扫描。**这种笛卡尔积式的灾难，极其容易把数据库 CPU 直接打满。**
    
3. **需要排序和分组的字段（ORDER BY / GROUP BY）**：
    
    我们在前面聊 `EXPLAIN` 时提过，如果不走索引排序，MySQL 会在内存或磁盘中开辟空间进行 `filesort`（文件排序）或创建临时表。因为 **B+ 树本身就是天然有序的**，如果把 `ORDER BY` 的列建上索引，引擎层拿到的数据就已经是排好序的，直接消除了昂贵的排序开销。
    
4. **为了实现“覆盖索引”的组合列**：
    
    如果有一个极高频的接口，比如只查用户的姓名和手机号：`SELECT name, phone FROM users WHERE name = '张三'`。我们强烈建议把这俩字段建成联合索引 `(name, phone)`，直接干掉回表操作，把 I/O 降到最低。
    

---

### 二、 坚决不要建索引的场景（黑榜：费力不讨好的无用功）

这是实战中最容易踩坑的地方。给下面这几种字段建索引，不仅查得不快，反而会严重拖垮 `INSERT` 和 `UPDATE` 的性能：

1. **区分度极低（低基数）的字段**：
    
    - **典型代表**：`gender`（性别，只有男女）、`status`（状态，比如只有 0/1/2）、`is_delete`（是否删除）。
        
    - **老兵剖析**：如果一个表有 100 万数据，性别为男的占 50 万。如果你给 `gender` 建索引，优化器在计算成本时会发现：通过索引查出 50 万个主键，然后再去聚簇索引**回表 50 万次**，这特么比我直接从头到尾扫一遍全表还要慢得多！因此，**MySQL 优化器会直接无视这个索引，强行走全表扫描**。你建的这棵 B+ 树除了浪费空间、拖慢写入之外，毫无用处。
        
2. **表的数据量极其微小**：
    
    如果一张表（如字典表、系统配置表）永远只有几百条数据，全部数据加起来还没占满一两个 16KB 的内存页。这时候直接全表扫描也就是一次内存 I/O 的事。建索引完全是脱裤子放屁。
    
3. **极其频繁更新的字段**：
    
    - **典型代表**：用户余额、文章阅读量、心跳时间戳。
        
    - **老兵剖析**：每一次字段的更新，InnoDB 都必须去修改对应的二级索引 B+ 树，甚至会引发大量的“页分裂”和合并。如果这个字段更新频率远大于查询频率，维护索引的代价就会反噬系统性能。
        
4. **超长文本字段（VARCHAR 太长或 TEXT）**：
    
    如果给一篇文章的 `content` 字段或者超长的 URL 建普通索引，不仅会导致 B+ 树的层级变得极高，而且每个内存页能存的节点极少，查询效率极低。
    
    - **破局之法**：如果非要查，请使用**前缀索引**（只给前 20 个字符建索引），或者直接上 Elasticsearch 搜索引擎。
        
5. **从不在 WHERE / GROUP BY / ORDER BY 中出现的字段**：
    
    索引不是用来“展示”的，而是用来“定位”的。只出现在 `SELECT` 后面用于展示，但从不参与条件过滤的字段，绝不能单独建索引。
    

### 老兵总结（索引架构箴言）

在真正的生产环境中，关于索引我们有三条“铁律”：

1. **能用联合索引，就坚决不建单列索引。**（尽量让一个联合索引覆盖多个业务场景）。
    
2. **单表索引数量尽量控制在 5 个以内。**（索引太多，DBA 看了想打人，写性能会直线下降）。
    
3. **定期清理“僵尸索引”。**（有些索引建了但业务早就不跑了，或者从来没被命中过，必须大胆删掉，给数据库减负）。
