
### 一、B树和B+树的区别
![[Pasted image 20260407135306.png]]
#### 1.数据结构
 
 B树的所有节点既存放键(key) 也存放数据(data)
 B+树只有叶子节点存放 key 和 data，其他内节点只存放 key。

B 树的叶子节点都是独立的。
B+树的叶子节点有一条引用链指向与它相邻的叶子节点。

#### 2.检索过程

B树的检索的过程相当于对范围内的每个节点的关键字做二分查找，可能还没有到达叶子节点，检索就结束了。

B+树的检索效率就很稳定了，任何查找都是从根节点到叶子节点的过程，叶子节点的顺序检索很明显。

#### 3.使用B树的优势

#####  B树（B-Tree）

B树的结构特点是：**所有的节点（包括非叶子节点和叶子节点）都同时存储 Key（索引键）和 Data（具体数据或行指针）。**

- **优势：最佳情况极快（查到就停）**
    
    - 由于非叶子节点也存数据，如果我们要查的数据刚好就在根节点或者第一层节点上，引擎在这个节点匹配到 Key 之后就可以直接返回数据，不需要一路查到底。最好的情况下，查询时间复杂度可以达到 $O(1)$。
        
- **劣势：致命的“高瘦”身材与范围查询噩梦**
    
    - **树太高，导致磁盘 I/O 多**：操作系统的磁盘是以“页”（InnoDB 默认 16KB）为单位预读的。因为 B树的非叶子节点里塞满了庞大的 Data，导致一个 16KB 的数据页能存放的 Key 的数量变得极少。节点存的 Key 少了，为了存下海量数据，树就只能往高了长。树越深，需要发生的磁盘 I/O 次数就越多，性能极速衰减。
        
    - **范围查询极其拉垮**：在业务中，`WHERE age > 20` 这种范围查询极其常见。在 B树中，数据散落在各个层级的树枝上，要做范围查询，只能使用极其繁琐的中序遍历，跨越不同的树枝去硬盘上反复横跳寻找数据，效率极其低下。
        

---

##### B+树（B+Tree）

B+树是 B树的变种，它的结构发生了两个极具破坏性的改变：**第一，非叶子节点绝对不存 Data，只存 Key 作为路由索引；第二，所有的叶子节点用双向链表连在了一起。**

- **优势 1：极致的“矮胖”身材，把磁盘 I/O 降到最低（核心杀手锏）**
    
    - 因为非叶子节点只存极其轻量的 Key 和指针，剥离了沉重的 Data 数据，一个 16KB 的内存页里现在能塞下成千上万个 Key。这就导致 B+树变得极其“扁平”。
        
    - **实战数据**：在 InnoDB 中，一棵高度为 3 的 B+树，通常就能支撑两千万级别的数据量！这意味着，你在两千万条数据里找一条记录，最多只需要 3 次磁盘 I/O，这在性能上是碾压级别的。
        
- **优势 2：天然的“范围查询”神器**
    
    - 由于所有真实数据全部集中在最底层的叶子节点上，并且叶子节点之间通过双向链表首尾相连。当你执行 `age > 20` 时，MySQL 只需要顺着树找到第一个 `20` 的叶子节点，然后顺着链表指针一路往后横向遍历就可以了，完全不需要再回到树枝上瞎折腾。连续的顺序 I/O 速度极快。
        
- **优势 3：查询性能绝对稳定**
    
    - 不管你查什么数据，由于真实数据全在最底层的叶子节点上，所以每一条 SQL 走的路径长度都是完全一样的（从根节点到底部）。性能表现非常可控，不会出现“查这条极快，查那条极慢”的怪现象。
        
- **劣势（相对而言）**
    
    - 如果数据刚好在树的最上面，它也不能直接返回，必须像过安检一样一层层走到最底层的叶子节点才能拿到真实数据。但在海量数据的宏观视角下，这个小小的牺牲换来的是整体极高的吞吐量，完全物超所值。
