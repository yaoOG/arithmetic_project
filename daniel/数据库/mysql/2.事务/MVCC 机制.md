**MVCC（多版本并发控制，Multi-Version Concurrency Control）**。

在传统的数据库理论中，读写操作是天生互斥的（加读写锁）。但互联网业务通常是“读多写少”，如果每次读写都加锁，数据库根本扛不住高并发。

MVCC 的诞生，就是为了实现一个终极目标：**让“读写不冲突”。读操作不上锁，写操作不阻塞读。**（注意：这里的“读”特指普通的 `SELECT`，也就是**快照读**）。

作为底层原理，我会把 MVCC 拆解为**“三大基石”**和**“一套算法”**来向您汇报：

---

### 一、 MVCC 的三大基石

InnoDB 实现 MVCC，完全依赖于底层数据结构的三把斧：**隐藏字段、Undo Log 版本链、Read View。**

#### 1. 隐藏字段（打上事务标签）

在 InnoDB 的每一行数据背后，其实都隐藏着几个不对外公布的字段。最核心的是这两个：

- **`DB_TRX_ID`（事务 ID）**：记录最后一次修改（或插入）这行数据的事务 ID。事务 ID 是严格递增的。
    
- **`DB_ROLL_PTR`（回滚指针）**：指向这行数据上一个版本的 Undo Log 地址。
    

#### 2. Undo Log 版本链（历史档案馆）

当一个事务去更新一行数据时，InnoDB 并不会直接抹掉旧数据。

它会把旧数据拷贝一份，塞进 **Undo Log（回滚日志）** 里。然后，用这行数据隐藏的 `DB_ROLL_PTR` 指针，指向 Undo Log 里的旧数据。

如果多个事务先后修改了同一行数据，这些历史记录就会通过指针串联起来，形成一条**单链表**，这就是著名的**“版本链”**。链表头是最新的数据，链表尾是最老的数据。

#### 3. Read View（读视图 —— 抓拍快照）

这是 MVCC 最精妙的设计。当你的事务去执行 `SELECT` 时，InnoDB 会在那一瞬间给你拍一张“系统级快照”，这就是 Read View。

这张快照里不存具体数据，而是存了当时系统里**活跃的事务状态**。它包含四个核心属性：

- **`creator_trx_id`**：当前正在执行 `SELECT` 的你自己的事务 ID。
    
- **`m_ids`**：生成快照那一刻，系统里所有**还未提交的活跃事务 ID 列表**。
    
- **`min_trx_id`**：`m_ids` 里最小的事务 ID。
    
- **`max_trx_id`**：生成快照那一刻，系统将要分配给下一个事务的 ID（也就是当时系统最大事务 ID + 1）。
    

---

### 二、 核心可见性算法（到底该读哪个版本？）

有了版本链和 Read View，当你的事务去 `SELECT` 一行数据时，InnoDB 就会顺着版本链从头（最新版本）开始往下比对。

比对规则（可见性算法）极其严密，完全依靠事务 ID 的大小关系来判定这行数据对你**“是否可见”**：

拿版本链中某个版本的 `DB_TRX_ID`（记为 `trx_id`）来套公式：

1. **如果 `trx_id == creator_trx_id`**：
    
    - **逻辑**：这行数据就是我自己这个事务刚改的。
        
    - **结果**：**绝对可见**。
        
2. **如果 `trx_id < min_trx_id`**：
    
    - **逻辑**：这个版本的修改者，比我快照里记录的所有活跃事务都要早，说明生成快照时它早就**已经提交**了。
        
    - **结果**：**绝对可见**。
        
3. **如果 `trx_id >= max_trx_id`**：
    
    - **逻辑**：这个版本是在我拍完快照之后，才开启的新事务去修改的。这就是妥妥的“未来数据”。
        
    - **结果**：**绝对不可见**。顺着链表继续往下找更老的版本。
        
4. **如果 `min_trx_id <= trx_id < max_trx_id`**（落在活跃区间内）：
    
    - 这时候就要去查一下 `m_ids`（活跃事务列表）了。
        
    - **情况 A：`trx_id` 在 `m_ids` 里**。说明拍快照时，这个修改数据的事务**还没提交**（是个脏数据）。$\rightarrow$ **不可见**，往下找。
        
    - **情况 B：`trx_id` 不在 `m_ids` 里**。说明拍快照时，虽然它在这个区间，但它已经**提前提交**了。$\rightarrow$ **可见**。
        

---

### 三、 终极奥义：RC 和 RR 隔离级别的本质区别

前面我们聊过，RC（读已提交）防不住不可重复读，而 RR（可重复读）防住了。它们的底层代码其实几乎一模一样，全套用了上面这套 MVCC 算法。

**它们唯一的区别，仅仅在于“拍快照（生成 Read View）的时机不同”！**

- **在 RC（读已提交）级别下：每次 SELECT 都会重新拍快照。**
    
    - 事务 A 第一次查，拍快照，看到数据是 100。
        
    - 事务 B 把数据改成 200 并**提交**。
        
    - 事务 A 第二次查，**重新拍快照**。此时事务 B 已经不在活跃列表 `m_ids` 里了，所以 200 对事务 A 变得可见了。这就发生了**不可重复读**。
        
- **在 RR（可重复读）级别下：一个事务只在第一次 SELECT 时拍快照，后面一直复用！**
    
    - 事务 A 第一次查，拍了一张快照（里面记录了事务 B 是活跃的）。
        
    - 事务 B 把数据改成 200 并**提交**。
        
    - 事务 A 第二次查，**不拍新快照，沿用旧的**。在旧快照里，事务 B 依然被认定为“未提交的活跃事务”。根据可见性算法，事务 B 改的 200 依然被判定为不可见。事务 A 只能顺着版本链找到 100。**完美实现了可重复读！**
        

### 老兵总结

MVCC 本质上就是一种**“拿空间换时间”**的艺术。InnoDB 用大量的 Undo Log 存储历史版本，再辅以精妙的 Read View 快照算法，让事务在读取时能够自动避开那些未提交的脏数据和未来的新数据。它把并发系统中最难的读写冲突，化解为了纯粹的内存比对。
