## 一、四大特性

### 1. 原子性 (Atomicity)

- **业务概念**：事务是一个不可分割的最小工作单元。事务内的数据库操作要么全部成功并提交（Commit），要么全部失败并回滚（Rollback），绝对不存在“只执行了一部分”的中间状态。
    
- **底层实现（Undo Log）**：InnoDB 保证原子性的核心武器是 **Undo Log（回滚日志）**。当事务对数据进行修改时，InnoDB 会同步生成与之相反的逻辑日志。如果事务执行失败或主动调用了 rollback，底层的回滚机制就会读取 Undo Log，执行反向操作（例如，你做了一次 Insert，它就执行一次 Delete），从而把数据完美恢复到事务开始前的状态。
    

### 2. 持久性 (Durability)

- **业务概念**：一旦事务提交成功，它对数据库的修改就是永久性的。哪怕下一秒数据库所在服务器直接拔电源宕机，重启后这部分数据也绝对不能丢失。
    
- **底层实现（Redo Log）**：这是通过 **Redo Log（重做日志）** 实现的。InnoDB 采用了典型的 **WAL（Write-Ahead Logging，日志先行）** 机制。当事务提交时，MySQL 不会立刻去把修改后的数据页刷入磁盘（随机 I/O 太慢了），而是先将修改记录以顺序 I/O 的形式追加到 Redo Log 中。只要 Redo Log 成功落盘，事务就算提交成功。如果随后发生宕机，MySQL 重启时会自动读取 Redo Log 恢复那些还没来得及刷入磁盘的脏数据。
    

### 3. 隔离性 (Isolation)

- **业务概念**：在极高并发的环境下，多个事务同时操作同一份数据时，相互之间不能产生干扰。
    
- **底层实现（锁 + MVCC）**：隔离性是我们在处理并发 Bug 时最常打交道的地方。InnoDB 实现隔离性主要依靠两大机制：
    
    1. **锁机制**：包括行锁（Record Lock）、间隙锁（Gap Lock）和临键锁（Next-Key Lock），用于控制写操作的并发冲突。
        
    2. **MVCC（多版本并发控制）**：通过隐藏的事务 ID 列和 Undo Log 组成版本链，让不同事务看到不同版本的数据，从而实现了“读写不冲突”，极大提升了并发性能。
        
- MySQL 定义了四种隔离级别（读未提交、读已提交、可重复读、串行化）。InnoDB 的默认隔离级别是 **可重复读（Repeatable Read）**。
    

### 4. 一致性 (Consistency)

- **业务概念**：事务执行前后，数据库的完整性约束没有被破坏，所有的状态依然满足业务逻辑上的合法性。最经典的例子就是转账：A 扣了 100 元，B 必须增加 100 元，两个人账户的总额必须保持不变。
    
- **底层理解**：在 ACID 这四个特性中，其实 **一致性（C）才是我们业务追求的终极目标**。而原子性（A）、隔离性（I）和持久性（D）都是数据库层面为了**实现这一目标而提供的底层手段**。也就是说，只要数据库底层的机制保障了 A、I、D，再加上我们开发人员写的业务代码没有逻辑漏洞，最终就能确保数据的一致性。

## 二、事务隔离级别

### 1. 读未提交 (Read Uncommitted - RU)

- **防御力**：几乎为零（**啥也防不住**）。
    
- **底层逻辑**：事务之间毫无隔离可言。一个事务可以直接看到另一个事务还没 `Commit` 的写操作。
    
- **产生问题**：会导致最致命的**脏读**、不可重复读、幻读。
    
- **实战建议**：**生产环境绝对禁用**。如果连脏读都能忍受，那完全没必要用事务，直接不用数据库就行了。
    

### 2. 读已提交 (Read Committed - RC)

- **防御力**：初级防御（**解决了脏读**）。
    
- **底层逻辑**：只能读到别人已经提交的数据。在 InnoDB 底层，每次执行 `SELECT` 都会生成一个**全新的 Read View（快照）**。
    
- **产生问题**：解决了脏读，但防不住**不可重复读**和幻读（同一个事务里，两次读的数据可能不一样）。
    
- **实战建议**：这是 **Oracle、SQL Server 的默认隔离级别**。在绝大多数互联网业务（如电商交易、社交 feed 流）中，RC 级别其实是最被推崇的。因为它在保证不读脏数据的前提下，释放了最大的并发性能。阿里等大厂的规范中，经常强制要求把 MySQL 的隔离级别从 RR 降级为 RC。
    

### 3. 可重复读 (Repeatable Read - RR)

- **防御力**：中高级防御（**解决了不可重复读，并在 InnoDB 中大体解决了幻读**）。
    
- **底层逻辑**：确保同一个事务里，多次读取同一行数据的结果绝对一致。在 InnoDB 底层，事务只在**第一次**执行 `SELECT` 时生成 Read View，之后一直**复用这个快照**。
    
- **产生问题**：SQL 标准中 RR 是防不住**幻读**的。但我们刚才深度剖析过，MySQL InnoDB 引擎通过临键锁（Next-Key Lock）把幻读问题一并按死了。
    
- **实战建议**：这是 **MySQL InnoDB 的默认隔离级别**。适用于对数据一致性要求极高的场景（比如纯金融账本核对），但在高并发下，由于间隙锁（Gap Lock）的频繁加持，非常容易引发死锁（Deadlock），对开发者的锁把控能力要求很高。
    

### 4. 串行化 (Serializable)

- **防御力**：终极绝对防御（**屏蔽一切并发问题**）。
    
- **底层逻辑**：最高级别的隔离。InnoDB 会隐式地将所有普通的 `SELECT` 语句全部转化为 `SELECT ... LOCK IN SHARE MODE`（加共享读锁）。这就意味着，读写彻底互斥，所有访问同一张表的并发事务全部要排队串行执行。
    
- **产生问题**：没有并发问题，因为**它把“并发”本身给干掉了**。性能极其低下。
    
- **实战建议**：**基本不用**。除非是极其极端的冷门单机业务，且完全不在乎性能，才会开启。
    

---

### 总结提炼

在架构选型时，隔离级别的选择就是一场关于**“锁的代价”**的博弈：

- 想要并发极高：选 **RC（读已提交）**，牺牲了可重复读，但锁粒度极小。
    
- 想要数据绝对一致：用 MySQL 默认的 **RR（可重复读）**，但要做好应对间隙锁带来死锁风险的准备。

## 三、并发带来的问题

### 1. 脏读 (Dirty Read) —— “读到了未提交的假数据”

- **业务场景**：假设财务系统正在给员工发工资，事务 B 将员工 A 的工资从 10000 修改成了 15000，但**还未提交 (Commit)**。此时事务 A 来查询员工 A 的工资，看到了 15000。紧接着，事务 B 因为系统异常发生了**回滚 (Rollback)**，工资实际还是 10000。但事务 A 已经拿着 15000 这个“假数据”去执行报税等后续逻辑了。
    
- **核心定义**：一个事务读取了另一个事务尚未提交的数据。
    
- **危害程度**：极其严重。脏读意味着业务逻辑建立在根本不存在的数据之上。

#### 如何解决脏读 (Dirty Read)？

- **适用级别**：读已提交 (RC) 及以上。
    
- **核心机制**：**Undo Log（回滚日志）+ 版本链隐藏字段**。
    
- **底层逻辑**：
    
    假设事务 B 正在修改某行数据，但**还未提交**。此时，InnoDB 并没有覆盖掉原来的数据，而是将修改前的旧数据放到了 Undo Log 中，并通过数据行隐藏的 `DB_ROLL_PTR`（回滚指针）串联成一个版本链。
    
    当你的事务 A 来读取这行数据时，InnoDB 的 MVCC 规则会进行严格的可见性判断：**它发现最新版本的数据所属的事务 B 还没有提交，就会直接拒绝读取这个最新物理数据**。
    
    接着，它会顺着回滚指针去 Undo Log 里找，直到找到一个**已经被提交的、对事务 A 可见的历史版本**。这样一来，事务 B 那些写了一半的“脏数据”，对事务 A 来说就完全隐身了。

### 2. 不可重复读 (Non-Repeatable Read) —— “同一行记录，两次读出来不一样”

- **业务场景**：事务 A 开启后，第一次查询订单号为 001 的状态是“待支付”。此时，并发的事务 B 将该订单状态**修改 (Update)** 为“已支付”并成功提交。当事务 A 在同一个事务内第二次查询订单 001 的状态时，发现变成了“已支付”。事务 A 懵了：我同一个事务里明明查了两次，怎么结果还变了？
    
- **核心定义**：在一个事务的多次读取中，同一条记录的数据内容发生了改变。
    
- **触发操作**：主要针对其他并发事务的 **`UPDATE`** 和 **`DELETE`** 操作。

#### 如何解决不可重复读 (Non-Repeatable Read)？

- **适用级别**：必须在 InnoDB 的默认级别 **可重复读 (RR)** 下。
    
- **核心机制**：**Read View（读视图）的复用机制**。
    
- **底层逻辑（区分 RC 和 RR 的分水岭）**：
    
    不可重复读的本质是：你查了一次数据；别人把这行数据改了并且**提交了**；你再查一次，发现数据变了。这在 RC 级别下是允许的，但在 RR 级别下被 InnoDB 完美解决。
    
    解法的精髓就在于 **Read View 的生成时机**：
    
    - **在 RC 级别下**：事务中**每一次**执行 `SELECT` 语句，InnoDB 都会生成一个**全新**的 Read View（相当于拍了一张当前系统状态的最新快照）。所以别人刚提交的新数据，你第二次读的时候立刻就能看到。
        
    - **在 RR 级别下（InnoDB 的杀手锏）**：InnoDB 只会在你事务中**第一次执行快照读（普通 `SELECT`）时，生成一个唯一的 Read View**。并且在整个事务结束之前，无论你执行多少次查询，**永远复用这同一个 Read View**。

### 3. 幻读 (Phantom Read) —— “同一个查询条件，两次查出来的条数变了”

- **业务场景**：事务 A 执行查询 `SELECT * FROM users WHERE age > 20`，查出来 5 条记录。就在这时，并发的事务 B 往表里**新增 (Insert)** 了一条 `age = 25` 的新用户记录并提交。当事务 A 再次执行完全相同的查询时，发现查出来了 6 条记录，就像产生了幻觉一样。
    
- **核心定义**：在一个事务的多次范围查询中，由于其他事务的插入操作，导致前后查询到的结果集（记录数量）发生了改变。
    
- **触发操作**：主要针对其他并发事务的 **`INSERT`** 操作。
    

> **老兵提示：不可重复读与幻读的区别（面试高频陷阱）** 很多新人会把这两个概念搞混。简单来说：
> 
> - **不可重复读**的重点在于**数据被修改了**（比如钱变少了）。防范的办法是锁住满足条件的**行记录（Record Lock）**。
>     
> - **幻读**的重点在于**数据变多了**（比如记录条数变了）。仅仅锁住现有的行是不够的，还要锁住行与行之间的空隙，防止新数据插进来，这就需要用到 InnoDB 的**间隙锁（Gap Lock）**。



#### InnoDB 到底有没有解决“幻读”？

在标准的 SQL 规范中，RR（可重复读）级别是无法解决幻读的，必须升级到 Serializable（串行化）。但是 **MySQL 的 InnoDB 引擎在 RR 级别下，通过两种截然不同的机制，把幻读问题解决掉了 99%。**

InnoDB 将查询分为了两类，分别采用了不同的解法：

##### 1）. 针对“快照读”（Snapshot Read）：靠 MVCC 解决

- **场景**：就是我们平时写的普通 `SELECT * FROM table WHERE ...`。
    
- **解法**：InnoDB 会在事务开启后的第一次查询时，生成一个 **Read View（读视图）**。依靠 MVCC（多版本并发控制）机制，你在整个事务期间，读到的都是这个历史快照的数据。
    
- **结果**：不管别的事务怎么疯狂 `INSERT` 新数据并提交，你的快照里根本没有这些新版本的数据。**完全屏蔽了幻读。**
    

##### 2）. 针对“当前读”（Current Read）：靠 Next-Key Lock（临键锁）解决

- **场景**：当你执行 `SELECT ... FOR UPDATE`、`SELECT ... LOCK IN SHARE MODE`，或者 `UPDATE`、`DELETE` 时，你需要读取并操作数据库中**最新**的数据。
    
- **解法**：此时 MVCC 就不顶用了。InnoDB 会祭出它的杀手锏——**Next-Key Lock（行锁 + 间隙锁 Gap Lock）**。
    
- **结果**：假设你执行 `SELECT * FROM users WHERE age > 20 FOR UPDATE`。InnoDB 不仅会锁住 `age > 20` 的现有记录，还会**把这些记录之间的“空隙（Gap）”死死锁住**。其他事务想要 `INSERT` 一个 `age = 25` 的新数据，会被阻塞卡住。**从物理层面阻止了幻读的发生。**
    

##### 💡 老兵追问：那剩下的 1% 极端情况是什么？（破防面试题）

在 RR 级别下，幻读依然有可能发生，前提是你**交叉使用了快照读和当前读**。

- **复现步骤**：
    
    1. 事务 A 先执行普通的 `SELECT`（快照读），查出 5 条数据。
        
    2. 事务 B 执行 `INSERT` 插入了第 6 条数据，并提交。
        
    3. 事务 A 脑抽了，执行了一个 `UPDATE` 语句，恰好更新到了这第 6 条数据（当前读，强行获取了新数据的写锁，导致系统为这行数据生成了属于事务 A 的新版本）。
        
    4. 事务 A 再次执行普通的 `SELECT`，此时会发现查出了 6 条数据。**幻读实打实地发生了！**
        

上面的问题如何解决？
update已经更新了预期之外的数据
##### 1. 实战最常用：“化面为点”的精准打击（先查 ID，再按 ID 更新）

这是业务开发中最推荐、性价比最高的做法。完全不需要加间隙锁，就能完美避开幻读的坑。

- **执行步骤**：
    
    1. **快照读获取具体 ID**：`SELECT id FROM staff WHERE status = '达标';` （查出了 ID 为 1, 2, 3, 4, 5）。在这步，业务代码在内存里算好账（5000元预算）。
        
    2. **按 ID 精准更新**：`UPDATE staff SET bonus = 1000 WHERE id IN (1, 2, 3, 4, 5);`
        
- **为什么能破局？** 即使就在你查完 ID 的下一毫秒，事务 B 插入了第 6 个人（ID 为 6，`status = '达标'`）并提交。但你的 `UPDATE` 语句的条件被死死限制在了 `id IN (1,2,3,4,5)`。 **此时，InnoDB 只会对这 5 行具体的记录加 行锁（Record Lock），根本不关心其他的间隙，也绝对不会去碰那条新插入的第 6 条数据。** 幻读危机被软件层面的逻辑完美化解！
    

##### 2. 业务架构层：引入“批次号（Batch ID）”或“状态机”

如果你非要用范围条件（`WHERE status = '达标'`）去更新，又不想加 `FOR UPDATE`，那么可以通过修改业务模型来防范。

- **执行步骤（批次号方案）**：
    
    1. **先打标（当前读）**：生成一个全局唯一的批次号（如 `batch_no = 'B20260331'`）。 `UPDATE staff SET batch_no = 'B20260331' WHERE status = '达标' AND batch_no IS NULL;` （这一步会更新当时的 5 个人，如果有新来的第 6 个人，因为还没插入，自然打不上这个批次号）。
        
    2. **基于批次号操作**： `SELECT * FROM staff WHERE batch_no = 'B20260331';` （算账：5 个人） `UPDATE staff SET bonus = 1000 WHERE batch_no = 'B20260331';` （发钱：5 个人）
        
- **为什么能破局？** 你把一个开放的范围（`status='达标'`）变成了一个封闭的、你独占的集合（`batch_no='B20260331'`）。后续事务 B 插入的新数据，它的 `batch_no` 是空的，根本混不进你的执行逻辑里。
**总结**：只要你在同一个事务里，统一使用快照读，或者统一使用当前读，InnoDB 就能完美解决幻读。如果混着用，神仙也救不了。




### 4. 丢失更新 (Lost Update) —— “你写的被我覆盖了”

- **业务场景**：事务 A 和事务 B 同时读取到某商品的库存为 100。事务 A 卖出 1件，计算得到 99，准备更新；同时事务 B 也卖出 1件，计算得到 99，准备更新。最终两个事务都提交了，但库存只扣减了 1 件，发生了严重的超卖现象。
    
- **核心定义**：两个并发事务同时读取并修改同一份数据，后提交的事务覆盖了先提交的事务的修改。
    
- **解决方案**：数据库底层自身的写锁（排他锁）通常能解决第一类丢失更新（回滚覆盖），但在业务层面应对第二类丢失更新（提交覆盖）时，我们通常需要引入**乐观锁（版本号机制）**或在查询时手动加悲观锁（`SELECT ... FOR UPDATE`）。

 #### InnoDB 解决了“丢失更新”吗？

丢失更新分为两类，InnoDB 的态度截然不同：

##### 1. 第一类丢失更新（回滚覆盖）—— InnoDB 彻底解决了

- **场景**：事务 A 和 B 同时更新一行，B 先提交，A 后回滚，把 B 的更新也撤销了。
    
- **解法**：任何隔离级别（哪怕是最低的读未提交）下，只要涉及到写操作（`UPDATE/DELETE`），InnoDB 都会强制加上**排他锁（X锁）**。事务 A 在写的时候，事务 B 根本进不来，只能排队等。所以第一类丢失更新在 MySQL 里根本不会发生。
    

##### 2. 第二类丢失更新（提交覆盖）—— InnoDB 需要开发者自己解决

- **场景**：经典的“电商超卖”。商品库存剩 1 件。事务 A 和 B 同时 `SELECT` 查出库存是 1。然后在各自的代码里计算 `1 - 1 = 0`。接着事务 A 执行 `UPDATE` 设为 0，事务 B 也执行 `UPDATE` 设为 0。最终两个人都提示购买成功，但库存只扣了 1 个。
    
- **解法**：InnoDB 的 RR 级别**无法自动识别**这种业务逻辑上的覆盖！它只能保证两个 `UPDATE` 语句排队执行，但无法阻止它们写入相同的结果。
    

作为开发者，我们必须利用 InnoDB 提供的“武器”来手动解决第二类丢失更新，通常有三种工业级方案：

1. **利用原子更新（最推荐，简单粗暴）**
    
    不要把计算逻辑放在 Java 代码里，直接让数据库去算：
    
    `UPDATE goods SET stock = stock - 1 WHERE id = 1 AND stock > 0;`
    
2. **悲观锁（Pessimistic Locking）**
    
    依靠 InnoDB 的当前读机制，在查询一开始就把这行记录死死锁住：
    
    `SELECT stock FROM goods WHERE id = 1 FOR UPDATE;` （拿到锁之前，别的事务连查都查不了）。
    
3. **乐观锁（Optimistic Locking）**
    
    不加数据库排他锁，而是在表里加一个 `version` 字段。利用 MVCC 的思想，在更新时校验版本号：
    
    `UPDATE goods SET stock = 0, version = version + 1 WHERE id = 1 AND version = 老版本号;` 如果失败了就重试。
    

**InnoDB 解决脏读，靠的是“绝对不让你看还没提交的数据”；解决不可重复读，靠的是“让你在整个事务期间，每次查都只能看同一张旧照片”。** 这两者的底层基石，都是 MVCC 机制下用 Undo Log 串联起来的数据版本链，从而极其优雅地实现了**“读写不冲突”**，让高并发下的数据库读操作如丝般顺滑。

**InnoDB 的 RR 隔离级别是一个“实用主义者”。它通过 MVCC 和 Next-Key Lock 帮我们挡住了绝大多数底层的幻读暗箭；但对于业务属性极强的“丢失更新（并发写覆盖）”，它只是递给了我们悲观锁和乐观锁的武器，扣动扳机的动作，必须由我们开发者自己来完成。
