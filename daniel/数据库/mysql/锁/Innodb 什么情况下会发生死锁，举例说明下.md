
死锁（Deadlock）的本质是：**两个或多个事务，各自持有对方需要的锁，并且互相等待对方释放，形成了一个闭环。**

InnoDB 检测到死锁后，会立马回滚其中一个事务（通常是改动数据较少的那个），让另一个继续执行。

以下是 **3 种最经典** 的死锁场景，按从易到难排序：

  

#### 场景一：AB-BA 顺序死锁（最常见，逻辑错误）

这是最简单的死锁，通常是因为**代码逻辑中，操作数据的顺序不一致**导致的。

**场景**：表 user，主键 id。


| **时间点** | **事务** **A**                                                                        | **事务** **B**                                                                        | **状态**  |
| ------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ------- |
| T1      | UPDATE user SET age=10 WHERE id=1;<br><br>  <br><br>**持有：****id=1** **的行锁** **(X)** |                                                                                     | A 拿到了 1 |
| T2      |                                                                                     | UPDATE user SET age=20 WHERE id=2;<br><br>  <br><br>**持有：****id=2** **的行锁** **(X)** | B 拿到了 2 |
| T3      | UPDATE user SET age=10 WHERE id=2;<br><br>  <br><br>**等待** **B** **释放** **id=2...** |                                                                                     | A 等 B   |
| T4      |                                                                                     | UPDATE user SET age=20 WHERE id=1;<br><br>  <br><br>**等待** **A** **释放** **id=1...** | B 等 A   |

结果：死锁。

解决：确保所有业务逻辑中，更新多行数据的顺序是一致的（比如永远先 update id 小的，再 update id 大的）。

  

#### 场景二：Gap Lock 互相踩踏（最隐蔽，高并发必现）

这就涉及到了我们刚才聊的 **间隙锁** 和 **插入意向锁**。这是高级开发必须掌握的案例。

场景：表 t，id 主键，c 是非唯一索引。现有数据 c=10 和 c=20。

间隙：(10, 20)。

|         |                                                                                         |                                                                                         |                                                                                |
| ------- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **时间点** | **事务** **A**                                                                            | **事务** **B**                                                                            | **状态**                                                                         |
| T1      | UPDATE t SET id=100 WHERE c=15;<br><br>  <br><br>**持有：****(10, 20)** **的** **Gap Lock** |                                                                                         | Gap Lock 是为了防幻读。<br><br>  <br><br>注意：**Gap Lock** **之间是兼容的！**                  |
| T2      |                                                                                         | UPDATE t SET id=200 WHERE c=16;<br><br>  <br><br>**持有：****(10, 20)** **的** **Gap Lock** | B 也拿到了同一个间隙的 Gap Lock。<br><br>  <br><br>**埋下隐患。**                              |
| T3      | INSERT INTO t VALUES(..., 15);<br><br>  <br><br>**申请：****(10, 20)** **的插入意向锁**          |                                                                                         | A 想插数据。<br><br>  <br><br>但必须等 B 的 Gap Lock 释放。<br><br>  <br><br>**A** **阻塞。**  |
| T4      |                                                                                         | INSERT INTO t VALUES(..., 16);<br><br>  <br><br>**申请：****(10, 20)** **的插入意向锁**          | B 也想插数据。<br><br>  <br><br>但必须等 A 的 Gap Lock 释放。<br><br>  <br><br>**B** **阻塞。** |

**结果**：**死锁**。

- A 等 B 释放 Gap Lock。
- B 等 A 释放 Gap Lock。
- 形成闭环。

**解决**：尽量避免在有高并发插入的业务中使用范围更新，或者降低隔离级别到 RC。

  

**场景三：S 锁升级死锁（坑爹的“加锁读”）**

这通常发生在两个事务都想“先读后改”，并且为了安全都加了 **S 锁**。

**场景**：id 是主键。

|   |   |   |   |
|---|---|---|---|
|**时间点**|**事务** **A**|**事务** **B**|**状态**|
|T1|SELECT * FROM t WHERE id=1 LOCK IN SHARE MODE;<br><br>  <br><br>**持有：****id=1** **的** **S** **锁**||A 读到了，加了 S 锁。|
|T2||SELECT * FROM t WHERE id=1 LOCK IN SHARE MODE;<br><br>  <br><br>**持有：****id=1** **的** **S** **锁**|S 锁和 S 锁兼容。<br><br>  <br><br>B 也拿到了 S 锁。|
|T3|UPDATE t SET ... WHERE id=1;<br><br>  <br><br>**申请：****X** **锁** **(****升级****)**||A 想把 S 升级为 X。<br><br>  <br><br>但必须等 B 释放 S 锁。<br><br>  <br><br>**A** **阻塞。**|
|T4||UPDATE t SET ... WHERE id=1;<br><br>  <br><br>**申请：****X** **锁** **(****升级****)**|B 也想升级为 X。<br><br>  <br><br>但必须等 A 释放 S 锁。<br><br>  <br><br>**B** **阻塞。**|

**结果**：**死锁**。

- A 拿着 S，等 B 释放 S 好让自己升级。
- B 拿着 S，等 A 释放 S 好让自己升级。
- **僵局。**

**解决**：既然知道后面要改，一开始就别用 SHARE MODE，直接用 FOR UPDATE（直接申请 X 锁），这样 T2 时刻 B 就会被阻塞，从而避免死锁。

  

**总结**

面试中，如果你能把 **场景二（Gap Lock 互相踩踏）** 讲清楚，说明你对锁的兼容性矩阵（Gap Lock 兼容，但排斥插入意向锁）理解得非常透彻。

场景二中如果事务A的更新语句改成：UPDATE t SET id=100 WHERE c=20;会发生死锁吗？事务T持有的是Gap Lock还是Next-key Lock

这是一个非常精准的延伸问题！你已经开始关注锁的细节差异了。

直接给结论：

1. **事务 A 持有的锁类型**：是 **Next-Key Lock**（临键锁）。
2. **会不会发生死锁**：**依然会**（只要事务 B 的行为逻辑不变）。

下面我为你详细拆解其中的原理。

  

**1. 为什么是 Next-Key Lock？**

在之前的场景中，UPDATE ... WHERE c=15，因为 c=15 **不存在**，所以 InnoDB 退化为 **Gap Lock**（只锁空隙，不锁记录）。

但在你的新问题中，UPDATE ... WHERE c=20，数据 c=20 是 **真实存在** 的。

根据 InnoDB 在 **RR 隔离级别 + 非唯一索引** 下的加锁规则：

- 它会锁住当前记录：**Record Lock** (c=20)。
- 它会锁住前面的间隙：**Gap Lock** (10, 20)。
- 合起来就是：**Next-Key Lock**，范围是 **(10, 20]**（左开右闭）。
- _(_注：为了防止在 _20_ 后面插入新的 _20_，它通常还会锁住下一个间隙 _(20, +∞)_ 的 _Gap Lock_，但我们主要关注前面的 _(10, 20))_。

  

**2. 为什么依然会发生死锁？**

虽然 A 拿的是 Next-Key Lock，B 拿的是 Gap Lock，但死锁的核心成因——**“Gap Lock 之间的兼容性”**——依然存在。

我们来看具体的执行时序（假设 B 依然是操作 c=16）：

|   |   |   |   |
|---|---|---|---|
|**时间点**|**事务** **A (****操作** **c=20)**|**事务** **B (****操作** **c=16)**|**状态解析**|
|**T1**|UPDATE t SET ... WHERE c=20;<br><br>  <br><br>**持有：****Next-Key Lock (10, 20****]**||A 锁住了 c=20 这一行，同时也圈占了 (10, 20) 这个地盘。|
|**T2**||UPDATE t SET ... WHERE c=16;<br><br>  <br><br>**申请：****Gap Lock (10, 20)**|**关键点：**<br><br>  <br><br>B 想要锁 (10, 20) 的间隙。<br><br>  <br><br>虽然 A 已经锁了这个间隙（包含在 Next-Key 里），但 InnoDB 规定：**Gap Lock** **之间是兼容的！**<br><br>  <br><br>所以，**B** **成功获得** **Gap Lock**。|
|**T3**|INSERT ... VALUES(..., 15);<br><br>  <br><br>**申请：插入意向锁**||A 想在自己锁住的范围内插入。<br><br>  <br><br>但是，它发现 **B** **也持有这个范围的** **Gap Lock**。<br><br>  <br><br>Gap Lock 挡住插入意向锁。<br><br>  <br><br>**A** **阻塞** **(Wait)**。|
|**T4**||INSERT ... VALUES(..., 16);<br><br>  <br><br>**申请：插入意向锁**|B 也想插入。<br><br>  <br><br>但是，它发现 **A** **也持有这个范围的** **Gap Lock** (Next-Key 的一部分)。<br><br>  <br><br>**B** **阻塞** **(Wait)**。|

**结果**：

- A 等 B 释放 Gap Lock。
- B 等 A 释放 Next-Key Lock 中的 Gap 部分。
- **死锁 (Deadlock)**。

  

**总结**

1. **锁的升级**：从 c=15 变成 c=20，锁从纯粹的 **Gap Lock** 升级为了 **Next-Key Lock** (10, 20]。
2. **死锁依旧**：因为 Next-Key Lock 包含了 Gap Lock 的特性。只要两个事务都成功获取了**同一段间隙的 Gap 属性**（无论是纯 Gap 还是 Next-Key），然后再尝试往这个间隙里 **INSERT** 数据，就一定会因为“互相阻挡”而触发死锁。
