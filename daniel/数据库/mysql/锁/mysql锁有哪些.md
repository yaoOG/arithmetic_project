MySQL 的锁机制非常庞大，这也是高并发架构排查死锁和性能瓶颈的核心所在。

为了让团队的新人或者在架构评审时能有一个清晰的全局视野，我通常会从**“粒度”、“属性”、“InnoDB 核心算法”以及“业务思想”**这四个维度，对 MySQL 的锁进行降维拆解：

### 1. 按“锁的粒度”划分（控制范围）

这是最直观的分类，决定了锁住数据的范围大小和并发度的绝对上限。

- **全局锁 (Global Lock)**：锁住整个 MySQL 实例，让整个库处于只读状态。典型命令是 `FLUSH TABLES WITH READ LOCK` (FTWRL)。在实战中，它通常只在**全库逻辑备份（mysqldump）**且存储引擎不支持 MVCC 时才会使用，业务极重，平时绝对禁用。
    
- **表级锁 (Table Lock)**：锁住整张表。开销小，加锁快，但并发度极低。包含以下几种子类：
    
    - **普通表锁**：通过 `LOCK TABLES t READ/WRITE` 手动加锁（极少用）。
        
    - **元数据锁 (MDL, Metadata Lock)**：不需要显式调用，只要你在执行增删改查（DML），系统就会自动加 MDL 读锁；当你在修改表结构（DDL，比如加减字段）时，会自动加 MDL 写锁。**老兵排坑**：线上执行 `ALTER TABLE` 导致整个库卡死，99% 都是因为 MDL 写锁被长事务阻塞了。
        
    - **意向锁 (IS/IX)**：InnoDB 自动加的表级锁。它的唯一目的就是为了“提升加表锁的效率”，相当于在表门口挂个牌子：“里面已经有行锁了，别来加表锁了”，省去了遍历全表查行锁的开销。
        
- **行级锁 (Row Lock)**：InnoDB 引擎的杀手锏。只锁住具体涉及的数据行。开销大，加锁慢，甚至会发生死锁，但并发度极高。**重点：InnoDB 的行锁，本质上是锁在“索引”上的，如果你的 SQL 没有命中索引，行锁会立刻退化为表锁！**
    

### 2. 按“操作属性”划分（读写互斥原则）

这是从锁的兼容性角度来看的，规定了谁能和谁共存。

- **共享锁 (Shared Lock / S 锁 / 读锁)**：允许多个事务同时读取同一行数据，大家和平共处。但如果有别的事务想去修改这行数据（获取写锁），就会被阻塞。可以通过 `SELECT ... LOCK IN SHARE MODE` 手动施加。
    
- **排他锁 (Exclusive Lock / X 锁 / 写锁)**：具有强烈的排他性。一旦某个事务拿到了某行的排他锁，其他事务既不能修改这行数据，也不能对这行数据加共享锁。`UPDATE`、`DELETE`、`INSERT` 默认都会加排他锁，查询时可以通过 `SELECT ... FOR UPDATE` 手动施加。
    

### 3. InnoDB 行锁的三大算法（防幻读的核心）

这是面试和实战中最核心、也最容易出错的地方。InnoDB 是如何用锁来解决幻读的？全靠下面这三个算法的组合：

- **记录锁 (Record Lock)**：最基础的行锁。精准锁住一条具体的索引记录。比如 `WHERE id = 1`（`id` 是主键），就会在这条聚簇索引上加记录锁。
    
- **间隙锁 (Gap Lock)**：锁住两条记录之间的“空隙”，但不包括记录本身。它的唯一目的就是**防止其他事务在这个空隙里插入新数据**。这是 RR（可重复读）隔离级别下防范幻读的物理基石。
    
- **临键锁 (Next-Key Lock)**：它是 **记录锁 + 间隙锁** 的完美结合。不仅锁住你查询的记录本身，还要锁住记录前面的间隙。**InnoDB 的行锁默认采用的就是 Next-Key Lock 算法。** 只有当查询条件是唯一索引且精准等值匹配时，它才会降级优化为纯粹的“记录锁”。
    

### 4. 按“业务思想”划分（架构选型）

这是我们平时写代码时经常挂在嘴边的锁方案，它不是数据库底层的具体机制，而是应用层的设计模式。

- **悲观锁 (Pessimistic Lock)**：抱着“总有刁民想害朕”的心态，认为每次操作都会发生冲突。所以在代码一开始，就利用数据库底层的排他锁（`FOR UPDATE`）把数据死死锁住。
    
- **乐观锁 (Optimistic Lock)**：认为系统很安全，几乎不会冲突。不在数据库层加物理锁，而是在表里加一个 `version` 字段。更新数据时利用 CAS 机制判断 `version` 有没有被别人改过。如果没改过就更新，改过了就重试或报错。

老兵总结：在日常的高并发微服务开发中，我们 90% 的精力和故障都在和 InnoDB 的**行锁（尤其是 Gap Lock 导致的互相等待）**打交道。
