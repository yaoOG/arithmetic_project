提到 Redis 的数据结构，很多初学者只能背出“String、Hash、List、Set、ZSet”这五种基础类型。但在高级开发和架构师的眼里，我们看到的是这五大外层 API 之下，Redis 极其精妙的**底层内存排布和结构设计**。

Redis 之所以单线程也能快得飞起，除了纯内存操作之外，最核心的原因就是它**为不同的数据类型、不同大小的数据量，量身定制了最高效的底层数据结构**。

我为您将最核心的 5 种基础结构，结合其实战场景和底层原理进行深度拆解：

### 一、 核心五大数据结构（外层 API 与 底层实现）

#### 1. String（字符串）

- **定位**：最基础的 K-V 存储。不仅可以存字符串，还能存数字（整数/浮点数）甚至二进制流（图片）。
    
- **实战场景**：常规缓存（JSON字符串）、**分布式锁（SETNX）**、计数器（文章阅读量、全局自增 ID - `INCR` 指令）。
    
- **底层黑科技（SDS）**：Redis 并没有用 C 语言原生的字符串，而是自己写了一个 **SDS（简单动态字符串）**。它预先记录了字符串的长度，获取长度的时间复杂度是 O(1)，而且杜绝了 C 语言字符串的缓冲区溢出问题，还能自动扩容。
    

#### 2. Hash（哈希）

- **定位**：类似于 Java 中的 `HashMap<String, Object>`。一个 Key 下面可以挂很多个 Field-Value 对。
    
- **实战场景**：极其适合存储**对象**（如用户画像信息：姓名、年龄、积分）。如果把用户对象序列化成一个 JSON String 存进去，修改单个字段会很麻烦；用 Hash 就可以精准做到 `HINCRBY user:1001 score 10`（只给该用户加 10 积分）。
    
- **底层黑科技**：
    
    - 数据量小的时候，使用紧凑的压缩列表（ziplist/listpack），极度节省内存。
        
    - 数据量大了，自动升级为真正的哈希表（hashtable），保证 O(1) 的查询效率。
        

#### 3. List（列表）

- **定位**：保持插入顺序的字符串列表。可以从两端（头/尾）进行压入和弹出。
    
- **实战场景**：用户的**时间线信息流**（Timeline，比如朋友圈最新动态）、轻量级的**异步消息队列**（利用 `LPUSH` 和 `BRPOP` 实现阻塞队列）。
    
- **底层黑科技（QuickList）**：早期是双向链表或压缩列表。后来 Redis 融合了两者，创造了 `quicklist`（快速列表），也就是将多个压缩列表用双向指针串起来，既避免了纯双向链表的内存指针开销，又保证了插入/删除的高效。


#### 4. Set（集合）

- **定位**：无序且**绝对唯一**的字符串集合。
    
- **实战场景**：**抽奖小程序**（利用 `SPOP` 随机抽出一个不重复的幸运儿）、**社交关系**（共同好友计算：利用 `SINTER` 求两个集合的交集，这个操作在关系型数据库里极其难写且慢）。
    
- **底层黑科技**：如果存的全是整数，底层就是一个极其紧凑的整数集合（intset）；如果存了字符串，就退化为哈希表（hashtable），只不过所有的 value 都是 null。


#### 5. ZSet（Sorted Set / 有序集合）—— Redis 的王冠

- **定位**：在 Set 的基础上，给每个元素强行绑定了一个浮点数属性（Score），并根据这个 Score 自动进行排序。
    
- **实战场景**：**各类排行榜**（微博热搜排行榜、游戏积分榜）、**延时队列**（把时间戳作为 Score 存入，起个定时任务按 Score 去查）。
    
- **底层黑科技（跳表 SkipList）**：这是面试必考点！为了在保证元素有序的同时，还能实现超高的插入和查找效率，Redis 底层采用了**跳表**和**字典**的结合。跳表通过多层级的索引指针，实现了媲美平衡二叉树的 O(logN) 查找效率，但代码实现远比红黑树简单，而且对范围查找（比如“查第 10 名到第 20 名”）极其友好。
    

---

### 二、 高级数据结构（老兵加分项）

除了这五种，实战中我们还会大量使用另外三种特殊结构：

- **Bitmap（位图）**：极致省内存，用于实现**千万级用户的每日签到**或者用户状态标记（只有 0 和 1）。
    
- **HyperLogLog**：以极小的内存开销（12KB），估算亿级海量数据的基数，用于统计**网站每日的真实 UV（独立访客）**，虽然有 0.81% 的误差，但在宏观统计下完全可接受。
    
- **GEO**：专门用于处理地理位置坐标，轻松搞定**“查找附近 3 公里内的外卖骑手”**。
