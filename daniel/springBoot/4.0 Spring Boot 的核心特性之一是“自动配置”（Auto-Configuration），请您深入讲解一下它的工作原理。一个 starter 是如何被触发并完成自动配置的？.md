如果说 Spring 框架的基石是 IoC 和 AOP，那么 Spring Boot 的绝对核心就是“约定优于配置（Convention over Configuration）”**，而支撑这个理念落地的就是**自动装配（Auto-Configuration）机制。

很多初学者容易产生一个误区，认为“Starter（启动器）里面写了装配代码”。作为一个老兵，我首先要纠正这个认知：**Starter 本质上只是一个空壳（一个只有 `pom.xml` 的 Maven 模块），它唯一的作用就是“打包引入相关依赖”。真正的装配逻辑，全部写在 `spring-boot-autoconfigure` 这个核心包里！**

那么，当我们引入一个 `spring-boot-starter-xxx` 时，底层的自动装配是如何被“触发”并完美拼装的呢？我们可以把这个过程拆解为“三级火箭”模型：

---

### 第一级火箭：点火开关 —— `@SpringBootApplication`

一切的源头都在我们写在 `main` 方法上的那个 `@SpringBootApplication` 注解里。这是一个复合注解，点进去看，最核心的其实是这三个：

1. `@SpringBootConfiguration`：标明这是一个配置类。
    
2. `@ComponentScan`：用来扫描我们自己写的业务代码（Controller、Service 等）。
    
3. **`@EnableAutoConfiguration`：这就是自动装配的绝对总开关！**
    

继续追溯 `@EnableAutoConfiguration`，里面包含了一个极其关键的导入语句：

`@Import(AutoConfigurationImportSelector.class)`

这个 `AutoConfigurationImportSelector`（导入选择器）就是整个自动装配的“总导演”，它负责去寻找所有的配置类。

### 第二级火箭：寻址雷达 —— SPI 机制与全量加载

“总导演”被激活后，他面临的问题是：世界上有无数个第三方组件，我该去哪里找它们的装配说明书？

Spring Boot 采用了类似 Java SPI（Service Provider Interface）的机制来解决这个问题：

- **在 Spring Boot 2.7 以前**：它会扫描所有 jar 包下 `META-INF/spring.factories` 文件。
    
- **在 Spring Boot 2.7 及 3.0 以后**：为了提升启动性能和隔离性，改为扫描 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件。
    

不管文件后缀怎么变，本质是一样的：在这个文件里，**罗列了上百个 Spring 官方写好的 `XxxAutoConfiguration` 类的全限定名**（比如 `RedisAutoConfiguration`、`DataSourceAutoConfiguration` 等）。

此时，总导演把这上百个配置类的名字全部读到了内存里。

### 第三级火箭：精准筛查 —— `@Conditional` 条件注解（真正的魔法）

这是最精彩、最能体现框架设计美学的一步。

内存里现在虽然有上百个配置类，但 Spring Boot 绝对不会傻到把它们全实例化，否则启动要花半个小时，内存直接炸了。

框架如何判断当前应该装配哪个？秘密就在于配置类上面贴满的 **`@Conditional` 条件注解系列**。

我们以 `RedisAutoConfiguration` 为例，看看它头顶上的注解：

```Java
@AutoConfiguration
@ConditionalOnClass(RedisOperations.class) // 核心触发点！
@EnableConfigurationProperties(RedisProperties.class)
public class RedisAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean(name = "redisTemplate") // 兜底让权策略！
    public RedisTemplate<Object, Object> redisTemplate(RedisConnectionFactory redisConnectionFactory) {
        // ... 创建并返回 RedisTemplate
    }
}
```

**触发流转大揭秘：**

1. 当你没有在 `pom.xml` 里引入 `redis-starter` 时，Classpath 下是没有 `RedisOperations.class` 这个类的。因此 `@ConditionalOnClass` 判断为 **false**，整个 `RedisAutoConfiguration` 直接被忽略，相当于没加载。
    
2. **有一天，你在 `pom.xml` 引入了 `redis-starter`**。Starter 会把 Jedis/Lettuce 等 jar 包拉进项目。
    
3. Spring Boot 启动再次扫描时，发现 Classpath 下有了 `RedisOperations.class`！`@ConditionalOnClass` 瞬间判断为 **true**。**这正是 Starter 触发自动装配的物理本质：引依赖 -> 类存在 -> 触发装配！**
    
4. 接着，Spring 看到里面有一个 `@Bean` 叫 `redisTemplate`。但上面加了 `@ConditionalOnMissingBean`。
    
5. Spring 会在现有的容器里找：“程序员有没有自己定义 `RedisTemplate`？”如果没定义，那就由框架自己 new 一个放进去；如果你自己写了一个贴了 `@Bean` 的 `RedisTemplate`，框架就会**谦卑地退让**，不再创建。这就完美实现了“约定优于配置”。
