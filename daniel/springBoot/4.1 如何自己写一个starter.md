作为一个老兵，我带过很多新人写过公司内部的 Starter（比如分布式日志追踪 Starter、内部 OSS 上传 Starter）。为了让您彻底看清它的物理样貌和触发机制，我们现在就一起“手撸”一个最经典的业务组件：**阿里老兵短信发送组件（`alien-sms-spring-boot-starter`）**。

按照官方最佳实践，一个完整的 Starter 应该分为两个模块：

1. `alien-sms-spring-boot-starter`（空壳，只负责引入依赖）
    
2. `alien-sms-spring-boot-autoconfigure`（真正的自动装配逻辑）
    

为了演示直观，我们把它们合二为一。下面是纯正的实战代码和触发原理解析：

---

### 第一步：引入“核动力”依赖 (`pom.xml`)

开发 Starter，不需要引入一整个 Spring Boot Web，只需要引入**自动装配的 API** 即可：



``` XML
<dependencies>
    <!-- 引入自动装配核心依赖 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-autoconfigure</artifactId>
    </dependency>
    <!-- 引入配置提示处理器（可选，但老兵强烈建议：能让业务方在 yml 里敲代码时有提示） -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-configuration-processor</artifactId>
        <optional>true</optional>
    </dependency>
</dependencies>
```

### 第二步：定义配置类（赋予业务方控制权）

框架必须提供让业务方在 `application.yml` 中自定义参数的入口。

```Java
package com.alien.sms.config;

import org.springframework.boot.context.properties.ConfigurationProperties;

// 绑定 yml 中的 alien.sms 前缀
@ConfigurationProperties(prefix = "alien.sms")
public class SmsProperties {
    /**
     * 是否开启短信功能，默认开启
     */
    private boolean enable = true;
    private String appKey;
    private String appSecret;
    private String signName;

    // ... 省略 getter/setter
}
```

### 第三步：定义核心服务（我们真正想提供给别人用的东西）

这是组件的核心功能类。注意：**这里千万不要加 `@Service` 或 `@Component` 注解！** 因为它是由自动装配类来决定是否注入的，不能写死。

```Java
package com.alien.sms.service;

import com.alien.sms.config.SmsProperties;

public class SmsService {
    private SmsProperties properties;

    public SmsService(SmsProperties properties) {
        this.properties = properties;
    }

    public void send(String phone, String msg) {
        System.out.println("【Alien SMS】使用 AppKey: " + properties.getAppKey());
        System.out.println("【Alien SMS】向 " + phone + " 发送签名 ["+ properties.getSignName() +"] 的短信: " + msg);
    }
}
```

### 第四步：编写灵魂类 —— 自动装配类 (AutoConfiguration)

这就是咱们上一节提到的“第三级火箭（筛选器）”，里面贴满了条件注解。



```Java
package com.alien.sms.autoconfigure;

import com.alien.sms.config.SmsProperties;
import com.alien.sms.service.SmsService;
import org.springframework.boot.autoconfigure.AutoConfiguration;
import org.springframework.boot.autoconfigure.condition.ConditionalOnClass;
import org.springframework.boot.autoconfigure.condition.ConditionalOnMissingBean;
import org.springframework.boot.autoconfigure.condition.ConditionalOnProperty;
import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Bean;

// 1. 声明这是一个自动配置类 (Spring Boot 2.7+ 推荐，替代 @Configuration)
@AutoConfiguration
// 2. 让 SmsProperties 这个配置类生效并注入到 Spring 容器
@EnableConfigurationProperties(SmsProperties.class)
// 3. 💥 触发点 A：只有当 Classpath 下存在 SmsService 这个类时，本配置类才生效！
@ConditionalOnClass(SmsService.class)
// 4. 💥 触发点 B：提供一个软开关。如果 yml 里配置了 alien.sms.enable=false，则不装配。默认是 true。
@ConditionalOnProperty(prefix = "alien.sms", name = "enable", havingValue = "true", matchIfMissing = true)
public class SmsAutoConfiguration {

    @Bean
    // 5. 💥 兜底与退让策略：如果业务方自己 @Bean new 了一个 SmsService，我就退让不生成了。
    @ConditionalOnMissingBean(SmsService.class)
    public SmsService smsService(SmsProperties properties) {
        // 创建并返回框架默认的 SmsService
        return new SmsService(properties);
    }
}
```

### 第五步：注册到 SPI 文件（极其关键的一步）

Spring Boot 启动时并不知道你的 `SmsAutoConfiguration` 藏在哪个包里。我们需要主动上报。

在项目的 `src/main/resources/` 目录下，依次创建文件夹：`META-INF/spring/`。

在里面创建一个文件，名字极其长且固定：**`org.springframework.boot.autoconfigure.AutoConfiguration.imports`**

_(注：Spring Boot 2.7 以前叫 `spring.factories`，新版本改为了这个文件)_

文件内容只有一行，就是你刚才写的自动配置类的全限定名：

```Plaintext
com.alien.sms.autoconfigure.SmsAutoConfiguration
```

至此，`mvn clean install`，我们的 Starter 就开发完成了！

### 🚀 业务方使用与触发全流程推演

现在，假设公司里的小王正在开发一个订单系统，他需要发短信。我们来看看这个 Starter 是如何被“触发”并完成装配的：

**1. 引入依赖（触发动作开始）**

小王在订单系统的 `pom.xml` 里写下：

```XML
<dependency>
    <groupId>com.alien</groupId>
    <artifactId>alien-sms-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

**💥 底层反应**：因为引入了依赖，`SmsService.class` 和 `SmsAutoConfiguration.class` 被下载到了小王的本地仓库，并加入到了订单系统的 Classpath（类路径）中。

**2. 启动扫描（雷达发现目标）**

小王点击 `OrderApplication.main()` 启动项目。

`@SpringBootApplication` 底层的 `AutoConfigurationImportSelector` 开始干活。它扫描了所有 jar 包下的 `META-INF/spring/org...imports` 文件。

**💥 底层反应**：扫描到了咱们刚刚写的 `com.alien.sms.autoconfigure.SmsAutoConfiguration`，把它加载进内存准备评估。

**3. 条件注解评估（筛选放行）**

Spring 开始评估 `SmsAutoConfiguration` 头顶上的注解：

- 看 `@ConditionalOnClass(SmsService.class)`：因为小王引了依赖，类路径下有这个类，**判断为 True，放行！**
    
- 看 `@ConditionalOnProperty(prefix = "alien.sms", name = "enable", matchIfMissing = true)`：小王还没写配置文件，触发 `matchIfMissing=true`，**判断为 True，放行！**
    

**4. 执行装配**

Spring 进入类内部，看到 `@Bean` 的 `smsService()` 方法。

- 看 `@ConditionalOnMissingBean(SmsService.class)`：小王没有自己写这个 Bean，**判断为 True，放行！**
    
- Spring 实例化 `SmsProperties`，再把 properties 传给 `SmsService` 的构造函数，最终把装配好的 `SmsService` 扔进了 IoC 容器（单例池）。
    

**5. 完美使用**

小王只需在自己的 Controller 里直接注入，啥配置都不用写，直接就能跑：

```Java
@RestController
public class OrderController {

    @Autowired
    private SmsService smsService; // 完美注入！直接可用！

    @PostMapping("/order")
    public String makeOrder() {
        smsService.send("13888888888", "您的订单已生成");
        return "OK";
    }
}
```

如果小王想改短信签名，他只需要在自己的 `application.yml` 里加上：

```YAML
alien:
  sms:
    app-key: "AK123456"
    sign-name: "阿里云"
```

**老兵总结**：

看明白了吗？这就是自动装配的魔法！**“Starter”本身并不执行代码，它只是提供了一份配置蓝图。真正扣动扳机的，是使用方在 `pom.xml` 里引入依赖的那一瞬间。** 依赖带来了 Class，Class 满足了 `@ConditionalOnClass`，从而像多米诺骨牌一样，一路触发了整个装配流程。
