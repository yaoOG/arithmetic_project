`BeanFactory` 和 `ApplicationContext` 都是 Spring IoC 容器，但定位不同：

| 对比点 | BeanFactory | ApplicationContext |
|---|---|---|
| 定位 | 最基础的 IoC 容器 | 更完整的应用级容器 |
| Bean 创建时机 | 默认懒加载，调用 `getBean()` 时才创建 | 默认启动时创建单例 Bean |
| 功能范围 | 只提供基本的 Bean 管理和依赖注入 | 在 BeanFactory 基础上扩展了大量企业级功能 |
| 国际化 | 不直接支持 | 支持 `MessageSource` |
| 事件机制 | 不直接支持 | 支持事件发布与监听 |
| AOP / 后处理器 | 需要手动注册部分组件 | 自动识别并注册 `BeanPostProcessor`、`BeanFactoryPostProcessor` 等 |
| 资源加载 | 能力较基础 | 支持统一资源访问，如 classpath、文件、URL |
| Web 集成 | 基础能力 | 更适合 Web、企业应用场景 |

简单说：

`BeanFactory` 是 Spring IoC 的底层容器，功能精简，主要负责 Bean 的创建、装配和管理。

`ApplicationContext` 是 `BeanFactory` 的子接口，除了 IoC 能力外，还提供国际化、事件发布、资源加载、自动注册后处理器等高级功能。

实际开发中几乎都使用 `ApplicationContext`，例如：

```java
ApplicationContext context =
    new ClassPathXmlApplicationContext("applicationContext.xml");

UserService userService = context.getBean(UserService.class);
```

而 `BeanFactory` 更偏底层，适合资源受限或框架内部场景。
