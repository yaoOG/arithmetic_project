我看待这两个关键字组合，通常会从**内存模型**和**生命周期**的角度来区分。

用最精炼的一句话来概括核心区别：**`final` 是“实例级常量”，而 `static final` 是“类级（全局）常量”。**

我们可以从以下四个核心维度来深入对比：

### 1. 归属层级与唯一性

- **`final`（实例常量）**：属于**对象（实例）**。每个对象在被创建时，都会拥有自己的一份 `final` 变量。也就是说，一千个对象有一千份 `final` 变量，**它们的值可以各不相同**，只是各自一旦赋值后就不能再变了。
    
- **`static final`（类常量）**：属于**类**。被所有的对象共享。不管你 `new` 了多少个对象，内存中永远只有一份。它是真正的“全局常量”。
    

### 2. 初始化时机的限制

这是在日常编码中最容易踩坑的地方。

- **`final`**：具有一定的灵活性。它可以在声明时赋值、在非静态代码块中赋值，**最重要的是，可以在构造方法中赋值**。这意味着我们可以根据外部传入的参数，在对象创建时动态决定这个“常量”的值。
    
- **`static final`**：极其严格。它必须在类加载的阶段就确定下来。因此，它**只能在声明时赋值，或者在静态代码块中赋值**，绝对不能在构造方法里赋值（因为构造方法是实例化时调用的，那时类早就加载完了）。
    

### 3. 代码直观对比

看下面这段代码，差异就非常明显了：

Java

```
public class Config {
    
    // 【static final】：类级全局常量，必须在类加载时确定，大家共用这一个配置
    public static final String GLOBAL_VERSION = "v1.0.0";
    
    // 【final】：实例级常量，可以在 new 的时候通过构造器动态赋值
    public final String instanceId; 

    public Config(String id) {
        // 每次 new Config 的时候，可以传入不同的 id
        // 但一旦赋值给 instanceId，这个对象的 instanceId 就不准再改了
        this.instanceId = id; 
    }
}
```

### 4. 内存分布与编译器优化（进阶）

- **内存位置**：
    
    - `final` 变量作为对象的成员变量，随着对象一起分配在**堆内存（Heap）**中。
        
    - `static final` 变量跟随着类的元数据，在 Java 8 及以后，存储在**元空间（Metaspace）**中。
        
- **编译期优化（常量折叠）**：
    
    - 如果 `static final` 修饰的是基本数据类型（如 `int`）或者 `String` 字面量，Java 编译器会把它当作**编译期常量（宏变量）**。
        
    - 在编译阶段，任何调用这个 `static final` 变量的代码，都会被**直接替换成真实的值**。比如代码里写了 `int a = Config.MAX_AGE;`，编译后的字节码里实际上直接就是 `int a = 100;`，运行时根本不需要去访问 `Config` 类，效率极高。
        

---

### 总结

- 当你需要一个**全系统共享、永远不变**的死值（比如错误码 `404`，圆周率 `3.14`），用 `static final`。
    
- 当你需要一个**每个对象独有、但在对象生命周期内不能被修改**的属性（比如用户的身份证号、不可变的配置项），用 `final`。
