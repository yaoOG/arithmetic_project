核心秘诀只有一句话：**继承 `java.lang.ClassLoader`，并且重写它的 `loadClass()` 方法。**

在标准的 `ClassLoader` 源码中，`loadClass()` 方法内部写死了“先找父类，父类不行再找自己”的逻辑。如果我们在自定义类加载器中只重写 `findClass()`，这叫**“遵循”**双亲委派；只有把 `loadClass()` 的逻辑给掀翻重写，这才是真正的**“打破”**。

为了让您一目了然，我手写一个极简版的“叛逆”类加载器 Demo：

### 一、 核心代码 Demo：打破委派机制的类加载器

在这个 Demo 中，我们的查找逻辑变更为：**“先看自己目录有没有，自己有就直接加载（不问父类），自己没有再扔给父类。”**



```Java
import java.io.*;

public class BreakDelegationClassLoader extends ClassLoader {

    // 自定义的类路径（比如放一些我们自己编译的特殊 .class 文件）
    private String classPath;

    public BreakDelegationClassLoader(String classPath) {
        this.classPath = classPath;
    }

    /**
     * 💥 核心叛逆区：重写 loadClass 方法，掀翻双亲委派法则
     */
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            // 1. 检查该类是否已经加载过（本地缓存）
            Class<?> c = findLoadedClass(name);
            
            if (c == null) {
                // ⚠️ 极其重要的安全防线：JDK 的核心类库必须交给父类加载！
                // 否则如果你自己写了个 java.lang.Object，JVM 底层会直接抛出 SecurityException 弄死你。
                if (name.startsWith("java.") || name.startsWith("javax.") || name.startsWith("sun.")) {
                    c = super.loadClass(name, false);
                } else {
                    // 2. 💥 打破双亲委派的地方：对于业务类，我们不去找父类，直接自己强行查找并加载！
                    try {
                        c = findClass(name);
                    } catch (ClassNotFoundException e) {
                        // 3. 如果我们自己指定的文件路径下实在找不到，再委派给父类（作为保底）
                        c = super.loadClass(name, false);
                    }
                }
            }
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }

    /**
     * 具体的查找与读取字节码文件的逻辑
     */
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            byte[] classData = getClassData(name);
            if (classData != null) {
                // 调用 JVM 底层的 defineClass，将 byte 数组转化为真正的 Class 对象
                return defineClass(name, classData, 0, classData.length);
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
        throw new ClassNotFoundException("找不到类: " + name);
    }

    // IO 操作：从磁盘读取 .class 文件到 byte 数组中
    private byte[] getClassData(String className) throws IOException {
        String path = classPath + File.separatorChar + className.replace('.', File.separatorChar) + ".class";
        File file = new File(path);
        if (!file.exists()) {
            return null;
        }
        
        try (InputStream is = new FileInputStream(file);
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            byte[] buffer = new byte[1024];
            int len;
            while ((len = is.read(buffer)) != -1) {
                baos.write(buffer, 0, len);
            }
            return baos.toByteArray();
        }
    }
}
```

### 二、 测试类验证

我们写一个测试程序来验证它。假设我们有一个 `com.example.TestObject` 类，并且我们把它编译后的 `TestObject.class` 文件放在了 `D:/my_classes/com/example/` 目录下。

```java
public class MainTest {
    public static void main(String[] args) throws Exception {
        // 创建我们的“叛逆”类加载器，指定它去 D:/my_classes 找代码
        BreakDelegationClassLoader loader = new BreakDelegationClassLoader("D:/my_classes");

        // 加载我们的测试类
        Class<?> clazz = loader.loadClass("com.example.TestObject");

        // 实例化对象
        Object obj = clazz.getDeclaredConstructor().newInstance();

        // 见证奇迹的时刻：打印加载这个类的 ClassLoader
        System.out.println("加载 TestObject 的类加载器是: " + obj.getClass().getClassLoader().getClass().getName());
        System.out.println("当前类加载器的父加载器是: " + obj.getClass().getClassLoader().getParent().getClass().getName());
    }
}
```

**运行结果预判：**

- 即使您的项目 `ClassPath` 下也存在一个 `com.example.TestObject`（会被默认的 `AppClassLoader` 加载），这个 Demo 也会**强行越过** `AppClassLoader`，优先从 `D:/my_classes` 去读取您放置的那个特殊版本的字节码。
    
- 打印出来的加载器名字必然是 `BreakDelegationClassLoader`。
    

### 三、 老兵实战避坑指南（安全沙箱红线）

在重写 `loadClass` 时，千万不要彻底“放飞自我”，有一条红线绝对不能踩：**不能打破 `java.*` 包的委派。**

这就是为什么我在 Demo 的代码里写了这句拦截逻辑：

`if (name.startsWith("java.") || name.startsWith("javax.")) { ... }`

- **底层机制**：JVM 的 `defineClass()` 方法内部写死了 C++ 级别的安全校验（SecurityManager）。如果你试图用自定义加载器去加载以 `java.` 开头的包（比如你伪造了一个 `java.lang.String`），一旦执行到 `defineClass`，JVM 就会抛出 `java.lang.SecurityException: Prohibited package name: java.lang`。
    
- **结论**：我们可以打破业务类的双亲委派，但**JDK 核心类的双亲委派是底层写死的强规则，绝对无法打破。** 这是 Java 保护核心沙箱安全的最后底线。
