
JVM 类加载机制的核心，也就是我们常说的**双亲委派模型（Parents Delegation Model）**。

在 JVM 中，并不是所有的类都由同一个加载器负责加载。Java 内部提供了一个层级分明的类加载器体系，它们各司其职，又通过一种类似于“踢皮球”的委派机制来协同工作。

我为您把这个机制的参与者和工作流程拆解一下：

### 1. 核心参与者：类加载器层级体系

在标准的 JVM 中，主要有三种系统提供的类加载器，再加上我们开发者自己编写的，通常构成了四层架构：

- **启动类加载器（Bootstrap ClassLoader）：** 处于金字塔的最顶端。它是由 C++ 实现的（在 HotSpot 中），负责加载 JVM 最核心的类库，比如 `$JAVA_HOME/jre/lib` 目录下的 `rt.jar`，或者被 `-Xbootclasspath` 参数指定的路径。像 `java.lang.String`、`java.lang.Object` 都是它加载的。
    
- **扩展类加载器（Extension ClassLoader）：** 负责加载 Java 的扩展库，一般是 `$JAVA_HOME/jre/lib/ext` 目录下的，或者由系统变量 `java.ext.dirs` 指定位置中的类库。
    
- **应用程序类加载器（Application ClassLoader / System ClassLoader）：** 这是我们打交道最多的加载器。它负责加载用户类路径（Classpath）上所有的类。如果没有自定义过类加载器，一般我们写的业务代码就是由它来加载的。
    
- **自定义类加载器（Custom ClassLoader）：** 也就是咱们前面手写过的，继承自 `java.lang.ClassLoader`，用于实现热部署、代码加密或应用隔离等特殊需求。
    

### 2. 运作机制：向上委派，向下尝试加载

当一个类加载器收到了加载某个类（比如 `com.example.UserService`）的请求时，它的运作逻辑非常严谨，通常分为两步：

- **第一步（向上委派）：** 它绝对不会自己立刻去加载这个类，而是**把这个请求委派给父类加载器去完成**。每一个层次的类加载器都是如此，因此所有的加载请求最终都会被传送到最顶层的启动类加载器（Bootstrap）中。
    
- **第二步（向下尝试）：** 只有当父类加载器反馈自己无法完成这个加载请求（在它的搜索范围中没有找到所需的类）时，子加载器才会尝试自己去加载。
    

打个通俗的比方：这就像是在公司里接到了一个疑难需求，底层程序员（AppClassLoader）先不干，直接把需求向上汇报给主管（ExtClassLoader），主管也搞不定，继续汇报给老板（Bootstrap）。如果老板能解决，就直接解决了；老板说这不归我管，再一层层打回来，最后底层程序员才只能自己动手去写代码。

### 3. 为什么必须采用这种机制？（核心价值）

JVM 煞费苦心地设计这套机制，核心目的只有两个：**安全沙箱**和**避免重复**。

- **保证核心类库的绝对安全（防止篡改）：** 假设黑客在应用代码里自己写了一个 `java.lang.String` 类，里面塞满了恶意代码。如果没有双亲委派，这个假的 `String` 类就会被加载。但是有了这套机制，加载请求会一路送到最顶层的 Bootstrap 处，Bootstrap 发现自己已经加载过官方的 `java.lang.String` 了，就会直接返回官方的那个类。黑客写的假类永远得不到加载的机会。
    
- **保证类的全局唯一性：** 我们前面讨论过，判定两个类是否“相等”，前提是它们由同一个类加载器加载。双亲委派保证了无论哪个加载器去加载基础类（如 `Object`），最终都会委派给处于模型最顶端的 Bootstrap 去加载。这就确保了 `Object` 类在各种类加载器环境中都是同一个类。
