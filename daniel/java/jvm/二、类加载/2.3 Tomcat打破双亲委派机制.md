
您敏锐地抓住了 Tomcat 架构中最精妙、也是最具“叛逆精神”的设计。

在标准的 Java 世界里，**双亲委派机制**是不可亵渎的铁律（永远让爸爸先加载）。但是，Tomcat 作为一款 Web 容器，它如果不打破这个铁律，根本无法满足企业级多应用部署的需求。

作为一个老兵，我给您详细拆解 Tomcat 是如何“造反”的，以及底层的 JVM 是如何配合演出的。

### 一、 为什么 Tomcat 必须“打破”双亲委派？

假设我们严格遵守双亲委派机制。Tomcat 启动时，它的 `AppClassLoader` 会去加载所有的依赖。

- **灾难场景**：App1 在 `WEB-INF/lib` 下放了 Spring 4.0 的 jar 包；App2 放了 Spring 5.0 的 jar 包。
    
- 如果按双亲委派，请求最终都交给了公共的父加载器。父加载器在解析 `org.springframework.context.ApplicationContext` 这个类时，只要加载了一次（比如碰巧先加载了 App1 的 Spring 4.0），内存里就只有这一份图纸了。
    
- 此时 App2 启动，调用了 Spring 5.0 才有的新方法，JVM 就会无情地抛出 `NoSuchMethodError`，App2 直接崩溃。
    

为了解决这个问题，Tomcat 引入了 **`WebAppClassLoader`**。

### 二、 Tomcat 的“叛逆”查找顺序

Tomcat 为部署在里面的**每一个 Web 应用（每一个 war 包）**，都在底层 `new` 了一个专属的 `WebAppClassLoader` 实例。

当 App1 内部的代码需要加载一个类时，它的专属 `WebAppClassLoader` 的查找顺序**故意颠倒了**：

1. **查本地缓存**：看看自己以前加载过这个类没有？有就返回。
    
2. **委托给 JVM 顶层（保底防线）**：交给 `BootstrapClassLoader`。这是为了绝对的安全，防止 Web 应用自己写一个 `java.lang.String` 替换掉系统的核心类。
    
3. **💥 核心叛逆点：优先自己加载**：它**不**去问父加载器，而是直接去当前 Web 应用的 `/WEB-INF/classes` 和 `/WEB-INF/lib` 目录下找！找到了就自己加载。
    
4. **最后才委派给父类**：如果自己目录下实在没有，再去求助父加载器（`CommonClassLoader` -> `AppClassLoader` 等）。
    

这种**“除核心类外，优先自己加载”**的策略，彻底打破了双亲委派“优先父类加载”的规则。

### 三、 JVM 的底层铁律：类命名空间隔离

您可能会问：就算 App1 和 App2 分别用自己的类加载器加载了 Spring，但在 JVM 的方法区（元空间）里，它们的类名都是 `org...ApplicationContext`，JVM 不会搞混吗？

这就是 JVM 最神奇的地方：**在 JVM 眼里，一个类的唯一身份标识，不仅仅是“类的全限定名”，而是【类的全限定名 + 加载它的 ClassLoader 实例】！**

这就是所谓的**“命名空间隔离”**。

- App1 里的 Spring：`[WebAppClassLoader 实例 A] + [org...ApplicationContext]`
    
- App2 里的 Spring：`[WebAppClassLoader 实例 B] + [org...ApplicationContext]`
    

在 JVM 看来，这是两个截然不同、毫无瓜葛的类！它们在内存中各自占据独立的区域，拥有独立的静态变量。

### 四、 实战示例推演

我们来看一个具体的物理目录结构和代码执行过程：

#### 1. Tomcat 目录部署结构

Plaintext

```
tomcat/
├── lib/  (Tomcat 自身的公共依赖，由 CommonClassLoader 加载)
│   └── servlet-api.jar 
│
├── webapps/
│   ├── App1/
│   │   └── WEB-INF/
│   │       ├── classes/com/example/MyService.class
│   │       └── lib/spring-core-4.0.jar  <-- Spring 4
│   │
│   └── App2/
│       └── WEB-INF/
│           ├── classes/com/example/MyService.class
│           └── lib/spring-core-5.0.jar  <-- Spring 5
```

#### 2. 代码执行推演

假设 `MyService` 类里有这样一段代码：

Java

```
public void doSomething() {
    // 触发类加载
    ApplicationContext context = new ClassPathXmlApplicationContext("...");
}
```

**当 App1 执行这段代码时：**

1. JVM 发现需要加载 `ApplicationContext` 类。
    
2. 获取当前类 `MyService` 的加载器：`WebAppClassLoader_App1`。
    
3. `WebAppClassLoader_App1` 开始它的查找逻辑：
    
    - 本地缓存？无。
        
    - 属于 `java.*` 核心类？不是。
        
    - **去自己的目录找？** 在 `App1/WEB-INF/lib/spring-core-4.0.jar` 里找到了！
        
    - **直接加载并返回 Class 对象。** 根本不理会父加载器。
        

**当 App2 执行同样的代码时：**

1. JVM 获取当前加载器：`WebAppClassLoader_App2`。
    
2. `WebAppClassLoader_App2` 按照同样的逻辑，在 `App2/WEB-INF/lib/spring-core-5.0.jar` 中找到了 Spring 5.0 的字节码。
    
3. **独立加载并返回一个全新的 Class 对象。**
    

**结果：**

- App1 愉快地使用着 Spring 4.0 的特性，App2 愉快地使用着 Spring 5.0 的特性。
    
- 如果它们都打印 `ApplicationContext.class.getClassLoader()`，你会看到它们输出的是不同的实例哈希码。
    
- 这就是 Tomcat 利用自定义类加载器为您搭建的完美“隔离病房”！
