整个类加载的过程，也就是一个 `.class` 字节码文件从磁盘被读取到内存，并最终形成可以被 JVM 直接使用的 `java.lang.Class` 对象的完整生命周期。

这个过程被严格划分为了三个大阶段，细分下来通常说是
- **五个核心步骤：加载、验证、准备、解析、初始化**。（其中验证、准备、解析这三步统称为**链接**阶段）。

我为您逐一详细拆解这个流水线：

### 1. 加载 (Loading)：把字节码搬进内存

这是类加载的第一个动作。在这个阶段，JVM 主要完成三件事：

- 通过类的全限定名（包名 + 类名）获取定义这个类的**二进制字节流**。这个流可以从本地 `.jar` 包读、从网络获取，甚至是通过动态代理技术在运行时直接生成。
    
- 将这个字节流代表的静态存储结构转化为**方法区【元空间】**的运行时数据结构。
    
- 在**Java 堆**中生成一个代表这个类的 `java.lang.Class` 对象，作为程序访问方法区中这些类型数据的外部接口。 _（注：我们之前聊到的双亲委派模型和打破双亲委派的自定义类加载器，主要就是在这个“加载”阶段发挥作用的。）_
    

### 2. 链接 (Linking)：承上启下的核心加工期

链接阶段负责把加载进来的字节码合并到 JVM 的运行状态中去，它包含了三个子步骤：

- **2.1 验证 (Verification)：安全第一**
    
    - 这是 JVM 的“安检”过程。确保读进来的二进制字节流符合《Java虚拟机规范》，不会做出危害虚拟机安全的行为。比如检查魔数（`0xCAFEBABE`）、版本号，校验字节码指令是否合法等。
        
- **2.2 准备 (Preparation)：分配内存与赋“零值”**
    
    - 这个阶段非常关键，也是线上容易踩坑的地方。JVM 会为类的**静态变量（static）**分配内存，并设置**默认的初始值**（比如 `int` 设为 `0`，引用类型设为 `null`）。
        
    - **避坑点：** 假设代码里写的是 `public static int value = 123;`，在准备阶段过后，`value` 的值其实是 `0`，而不是 `123`。真正的业务赋值动作要等到后续的初始化阶段。
        
    - **特例：** 如果是 `public static final int value = 123;`（加了 `final` 的常量），在准备阶段就会直接赋值为 `123`。
        
- **2.3 解析 (Resolution)：符号变指针**
    
    - JVM 会把常量池内的**符号引用**（也就是一组描述目标的字面量，比如你在代码里调用的某个类名或方法名的字符串）替换为**直接引用**（直接指向目标内存地址的真实物理指针、句柄或偏移量）。
        

### 3. 初始化 (Initialization)：执行类构造器

这是类加载过程的最后一步，也是真正开始执行类中定义的 Java 业务代码的时刻。

- **执行 `<clinit>()` 方法：** 编译器会自动收集类中的所有**静态变量的赋值动作**和**静态代码块（static block）**中的语句，合并产生一个 `<clinit>()` 方法。JVM 会在这个阶段安全地（保证多线程互斥同步）执行这个方法。
    
- 只有到了这个阶段，刚才那个 `public static int value = 123;` 里的 `value` 才会被真正赋值为 `123`。
    

### 总结

总而言之，JVM 的类加载就是一个**“加载字节码 -> 安全校验 -> 分配静态内存 -> 解析底层引用 -> 执行静态初始化代码”**的严谨过程。
