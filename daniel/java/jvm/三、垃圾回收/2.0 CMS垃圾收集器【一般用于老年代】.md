
关于 CMS（Concurrent Mark Sweep）垃圾回收器，虽然它在 JDK 9 被标记为废弃（Deprecated），并在 JDK 14 中被彻底移除，但在目前许多依然坚守 JDK 8 的生产系统（尤其是对响应延迟要求较高的 Web 服务）中，它依然是老年代的主力收集器。


CMS 的核心设计目标是**获取最短的垃圾收集停顿时间（Low Latency）**，它主要作用于老年代，并在底层采用了**标记-清除（Mark-Sweep）**算法。

它的整个收集过程主要可以分为四个核心阶段：
  

**1. 初始标记（Initial Mark）**
- **状态：** **STW (Stop-The-World)**
- **过程：** 这个阶段非常短暂。GC 线程会暂停所有应用线程，仅仅去标记那些被 **GC Roots 直接关联**到的对象，以及年轻代中存活对象所引用的老年代对象。
- **特点：** 因为只扫描直接关联的第一层，所以速度极快，STW 的停顿时间很短。

**2. 并发标记（Concurrent Mark）**
- **状态：** **并发执行**（与业务线程同时运行）
- **过程：** 从初始标记阶段找到的 GC Roots 开始，遍历整个老年代的对象图，找出所有存活的对象。
- **特点：** 这是 CMS 中耗时最长的一个阶段。但因为它和应用线程是并发执行的，所以不会阻塞业务。不过，由于业务线程同时在跑，这就导致对象的引用关系可能会在这个阶段发生变化（比如某个原本存活的对象变成了垃圾，或者新生了一个对象）。


**3. 重新标记（Remark）**
- **状态：** **STW**
- **过程：** 为了修正并发标记阶段因为业务线程继续运行而导致引用关系发生变动的那部分对象的标记记录。
- **核心机制：** 与 G1 使用的 SATB（起始快照）算法不同，CMS 在这里使用的是**增量更新（Incremental Update）**算法。简单来说，就是把并发标记期间新增的引用记录下来，在这个阶段以这些新增引用为根重新扫描一次（这相当于处理三色标记法中，黑色对象突然指向了白色对象的情况）。
- **特点：** 这个阶段的停顿时间通常会比初始标记稍长，但远比并发标记的时间短。


**4. 并发清除（Concurrent Sweep）**
- **状态：** **并发执行**
- **过程：** GC 线程和业务线程同时运行，清理掉标记阶段判断为死亡的垃圾对象，释放内存空间。
- **特点：** 由于不需要移动存活对象（使用的是标记-清除算法），所以这个阶段也可以与业务线程并发执行。但同样因为业务线程在运行，这个阶段产生的新的垃圾（即“浮动垃圾”）只能留到下一次 GC 时再去清理。

  

**生产环境中的痛点与隐患**

作为有经验的后端开发，我们在实际线上运维 CMS 时，往往最头疼的是它带来的三个经典缺陷：

1. **CPU 资源敏感：** 在并发标记和并发清除阶段，GC 线程会抢占 CPU 资源，导致应用吞吐量下降。
2. **空间碎片（Memory Fragmentation）：** 这是**标记-清除**算法的通病。CMS 不会像 G1 那样去移动对象（不整理内存），长期运行会导致老年代出现大量不连续的内存碎片。当有大对象需要分配，明明老年代总空间够，但就是找不到连续的内存时，就会被迫触发 Full GC。
3. **Concurrent Mode Failure (并发模式失败，CMF)：** 也是最致命的问题。如果在并发清除阶段，业务线程产生大对象的速度太快，导致老年代空间在垃圾回收完成前就被撑爆了，CMS 就会宣告失败。此时 JVM 会触发终极的兜底策略：**直接暂停所有应用线程（STW），切换为单线程的 Serial Old 收集器来进行老年代的清理和整理。** 这会导致一次极其漫长的停顿，对于线上服务是灾难性的。
