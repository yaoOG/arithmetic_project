

三色标记法（Tri-color Marking）主要是为了解决 **“在垃圾回收线程与用户线程并发执行（同时运行）的时候，如何保证标记结果的准确性”** 这个问题。

主要应用在：

**CMS 收集器：** 在其耗时最长的“并发标记”阶段，使用的就是三色标记法。

**G1 收集器：** 同样在“并发标记”阶段使用。

  

**1.三色标记法的核心逻辑**

它将对象在 GC 过程中的状态逻辑分为三种：

- ⚪ **白色 (White)：** 还没被 GC 访问过的对象。（如果 GC 结束了还是白色，就是垃圾，要被杀掉）。
- 🔘 **灰色 (Grey)：** 对象本身已经被 GC 访问过了，但它的**孩子（成员变量引用）还没扫完**。（代表正在处理中，是黑白之间的边界）。
- ⚫ **黑色 (Black)：** 对象本身和它的**所有孩子**都已经被 GC 访问完了。（认为是绝对安全的存活对象）。

  

**2.它重点要解决的那个“致命 Bug”：对象消失**

如果只用三色标记法，不做任何额外处理，在并发场景下会出现一个致命问题：**把活着的对象当成垃圾给回收了（误杀）**。

这被称为 **“对象消失”** 问题。

**场景还原（致命两步走）：**

假设 GC 正在扫描，现在的状态是：**A (灰) -> B (白)**。

1. **第一步（断开引用）：** 用户线程执行 A.B = null;。

- A 和 B 断开了。此时 B 变成了孤儿（暂时）。

3. **第二步（建立新引用）：** 用户线程执行 C.B = new_ref; （C 是一个**黑色**对象，已经扫完的）。

- 用户把 B 挂到了 C 的下面。

**结果：**

- GC 回到 A（灰色），发现 A 下面没有引用了，于是把 A 涂成 **黑色**。
- GC 看到 C（黑色），**因为它已经是黑色的，GC 默认黑色是处理完的，不会再回头去扫描 C 的孩子。**
- **结局：** 可怜的 B，虽然挂在 C 下面，但 GC 永远看不见它了。GC 结束后，B 依然是 **白色**，被当做垃圾回收掉。程序崩溃！

  

**3. 解决方案：三色标记 + 读写屏障**

三色标记法本身只是一个**模型**，为了解决上面那个“对象消失”的 Bug，JVM 必须配合 **屏障技术 (Barrier)** 来使用。

这引出了两个流派（分别破坏导致 Bug 的两个条件）：

1. **增量更新 (Incremental Update) —— CMS 采用**

- **逻辑：** 破坏“建立新引用”这个条件。
- **做法：** 当黑色对象 C 插入新的引用指向 B 时，就把 C **涂回灰色**。意思是：“我都扫完了，你又给我塞个新孩子，那我只好重新再排队扫一遍。”

2. **原始快照 (SATB) —— G1 采用**

- **逻辑：** 破坏“断开引用”这个条件。
- **做法：** 当 A 断开对 B 的引用时，把 B **记录下来**（强制标灰/保活）。意思是：“不管你现在的引用关系怎么变，我只认 GC 开始那一刻的样子。GC 开始时 B 是活的，那我这次就不杀它。”
