在 Java 体系中，能作为 GC Roots 的对象，本质上都是那些==**当前阶段绝对不能被回收的“活跃核心引用==”**。我们可以将它们划分为“线程运行上下文”、“全局范围”以及“JVM 内部与特定场景”三大类。

具体来说，主要包含以下几种：

### 1. 线程运行上下文中的引用（动态）

这部分引用跟随着线程的执行而不断变化，生命周期通常较短，但处于执行状态时绝对不可回收。

- **虚拟机栈（栈帧中的局部变量表）中引用的对象：** 这是最常见的一种。比如各个线程当前正在执行的方法里的局部变量、入参等。只要方法还没有出栈，这些变量引用的堆中对象就是存活的。
    
- **本地方法栈中 JNI（Native 方法）引用的对象：** 如果 Java 程序调用了 C/C++ 的底层库，Native 代码中持有的 Java 对象引用（如 Global Reference 或 Local Reference）也会作为 GC Roots，以防止这些对象在使用期间被 JVM 意外回收。
    

### 2. 全局范围的引用（静态）

这部分引用通常生命周期极长，也是**日常开发中引发内存泄漏的重灾区**。

- **方法区中类静态属性引用的对象：** 类的静态变量属于类级别，只要类没有被卸载，这些引用就会一直存在。如果我们在业务代码里用 `static HashMap` 做本地缓存且没有清理机制，它引用的所有对象都会因为被静态变量拽着而无法回收，最终导致 OOM。
    
- **方法区中常量引用的对象：** 例如字符串常量池（String Table）里的引用，或者 `final` 修饰的常量对象。
    

### 3. JVM 内部机制与锁持有的引用

- **被同步锁（`synchronized` 关键字）持有的对象：** 既然一个对象正在被当作锁使用，说明有线程正在进行并发控制，该对象显然不能在这个时候被回收。
    
- **JVM 内部的基本引用：** 这包括基本数据类型对应的 `Class` 对象，一些常驻的预分配异常对象（如 `NullPointerException`、`OutOfMemoryError`），以及系统类加载器（System ClassLoader）等。
    
- **反映 JVM 内部情况的引用：** 如 JMXBean、JVMTI 中注册的回调、本地代码缓存等。
    

### 4. 局部回收时的“临时” GC Roots（实战进阶点）

结合我们前面聊到的 G1 或传统的分代垃圾回收器，这里还有一个非常重要的动态 GC Roots 概念：

- 当 JVM 只进行部分区域的垃圾回收时（例如只做 Young GC），**其他非收集区域指向收集区域的引用，会被临时加入到 GC Roots 集合中**。
    
- 就像我们在 G1 中聊的 RSet，老年代对象指向新生代对象的跨区引用，在 Young GC 时，老年代所在的 Region 虽然不被回收，但那些跨区引用的指针会被当作临时的 GC Roots 来扫描，以保证新生代对象的存活判定不出错。
    

### 总结与实战意义

理解了这几类 GC Roots，我们在排障时才能有的放矢。当线上发生 OOM 报警，我们导出 Heap Dump 文件放进 MAT（Memory Analyzer Tool）里分析时，通常第一步就是找到占用内存最大的对象，然后执行 **"Path to GC Roots"（到 GC Roots 的最短路径）**。
