

### 一、特点

1.分区模型：将堆划分为多个大小相等的独立区域（Region），不再坚持物理分代，region的类型分为

![[Pasted image 20260421113417.png]]

Eden Regions、Survivor Regions、Old Regions、Humongous Regions

  

它将整个Java堆划分为多个大小相等（默认约2048个Region，每个Region大小为1MB~32MB）的独立内存区域（Region）。

  

2.可预测的停顿: G1跟踪每个Region的垃圾价值（回收所能释放的空间大小和所需时间），优先回收价值最大的区域（Garbage-First名称的由来）。用户可以设置期望的最大停顿时间（-XX:MaxGCPauseMillis）。

  

### 二、垃圾回收过程

G1 的运作不是线性的，而是一个状态流转的循环。主要包含三个阶段：

**Young GC (新生代回收) \ Concurrent Marking (并发标记周期) \ Mixed GC (混合回收)**

  

**1. Young GC (新生代回收)**

这是最频繁发生的阶段。

- **触发时机：** 当 Eden 区的 Region 被填满时。
- **行为 (STW)：**
- 这是一个 Stop-The-World 事件。
- G1 计算出 Eden 和 Survivor 区的 Region 集合（CSet）。
- 使用**复制算法**，将存活对象复制到空的 Survivor Region 或晋升到 Old Region。
- **特点：** 既然是复制算法，就天然**没有内存碎片**。

  

**2. 并发标记周期 (Concurrent Marking Cycle) 【五步】**

当堆内存占用达到一定阈值（默认 45%，由 -XX:InitiatingHeapOccupancyPercent 控制）时，G1 不会立刻回收老年代，而是启动并发标记，为后面的“混合回收”做准备。

这个过程与 CMS 有点像，但细节不同：

**① 初始标记 (Initial Mark) [STW]**
- 标记 GC Roots 直接引用的对象。
- 关键点： 这一步通常是**搭 Young GC 的便车**（Piggyback）同步完成的，利用 Young GC 的 STW 时间，所以没有额外的单独停顿。

**② 根区域扫描 (Root Region Scan)**
- 扫描 Survivor 区中引用到老年代的引用。必须在下一次 Young GC 发生前完成。

**③ 并发标记 (Concurrent Mark)**【**最耗时阶段**。】
  **并发执行**，没有停顿。GC 线程与应用线程一起运行，从 GC Roots 开始遍历整个对象图，找出所有的存活对象。G1 在这个阶段使用 **SATB (Snapshot-At-The-Beginning)** 算法来解决并发情况下的漏标问题。

 **④ 最终标记 (Remark) [STW]**
- 暂停用户线程，处理 SATB 缓冲区中的遗留记录（解决并发时引用变化的问题）。比 CMS 的 Remark 快。

**⑤ 清理 (Cleanup) [STW] + [Concurrent]**
- **STW 部分：** 同时，它会根据每个 Region 的回收价值（垃圾越多，价值越高）对 Region 进行排序，并根据用户设置的期望停顿时间制定回收计划，选出要回收的 Region 集合（即 **CSet**, Collection Set）。
- **并发部分：** **直接回收那些完全空闲（全垃圾）的 Region**。注意，如果一个 Region 全是垃圾，现在就直接清掉了，不需要等到 Mixed GC。

  

**3. Mixed GC (混合回收) —— G1 的精髓**

并发标记结束后，G1 就知道哪些 Old Region 的垃圾最多。接下来的几次 GC 通常就是 Mixed GC。它不仅回收所有的新生代 Region，还会**根据停顿预测模型，优先回收那些垃圾最多、收益最高的老年代 Region**（这就是 Garbage-First 名字的由来）。回收过程同样是采用复制算法，将存活对象拷贝到空闲的 Region 中，因此**不会产生内存碎片**

  

并发标记结束后，G1 知道了哪些老年代 Region 垃圾最多。此时进入 Mixed GC 阶段。

- **回收范围：** **所有的 Young Regions** + **一部分高价值的 Old Regions**。
- **触发时机：** 并发标记结束后。
- **行为 (STW)：**
- G1 不会一次性回收所有老年代（怕停顿时间太长）。
- 它会根据用户设置的**最大停顿时间 (-XX:MaxGCPauseMillis)**，计算出本次能回收多少个老年代 Region。
- 使用**复制算法**（Evacuation），将存活对象拷贝到新的 Region。
- **多次执行：** Mixed GC 通常不是一次做完，而是分多次（默认 8 次）进行，每次回收一部分老年代，直到回收足够多的内存或没有高回报的 Region 为止。之后又回到 Young GC 模式。

  

**三、 异常兜底：Full GC**

这是面试中必须提到的“反面教材”。G1 的设计目标是避免 Full GC，但在以下情况会触发：

1. **Allocation Failure (分配失败)：** Mixed GC 期间，内存回收速度赶不上分配速度，或者复制对象时没有足够的空闲 Region (To-Space Exhausted)。
2. **Metaspace 满了。**
3. **巨型对象 (Humongous Object) 分配失败**。

- **后果：** 触发单线程（JDK 10 之前）或多线程（JDK 10+）的 Full GC，对**整个堆**进行标记-整理，导致极长时间的 STW。

  

  

**四、优点**

1. **可预测的停顿**：能尽可能地在用户设定的停顿时间内完成垃圾收集。
2. **空间整合**：基于“标记-复制”算法，避免了CMS那样的内存碎片问题。
3. **高效的回收**：采用“Garbage-First”策略，每次回收收益最大的Region，性价比高。
4. **能管理超大堆**：分区模型使其能更好地扩展到大容量堆。

  

**五、缺点**

1. **内存占用更高**：由于每个Region都需要维护一个RSet，RSet本身会占用额外的内存空间。
2. **写操作开销**：为了维护RSet，每次对引用进行写操作（如 objA.field = objB）时，都会产生一个“写屏障”的开销。虽然经过优化，但这个开销依然存在。
3. **Full GC** **的尴尬**：虽然比CMS更不容易发生Full GC，但一旦发生，其性能同样很差。
