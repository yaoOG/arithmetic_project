在 JVM 的内存结构中，垃圾回收（GC）并不是对所有的内存区域一视同仁的。我们可以把 JVM 的内存分为**线程私有**和**线程共享**两大类。GC 的目标几乎完全集中在**线程共享**的区域。

我为您将 JVM 内存各个区域的回收情况拆解如下：

### 1. 绝对不需要 GC 介入的区域（线程私有）

这些区域的内存分配和回收具有强烈的确定性，它们随着线程的创建而生，随着线程的结束（或方法调用的结束）而自动释放，因此不需要垃圾收集器来管理：

- **程序计数器（Program Counter Register）：** 记录当前线程执行的字节码行号，占用内存极小。
    
- **虚拟机栈（Java Virtual Machine Stacks）：** 存放 Java 方法执行时的栈帧（局部变量表、操作数栈等）。方法进栈分配内存，出栈自动释放。
    
- **本地方法栈（Native Method Stacks）：** 为 JVM 调用 Native (C/C++) 方法服务，机制与虚拟机栈类似。
    

### 2. GC 的绝对主战场：Java 堆（Java Heap）

这是 JVM 中内存最大的一块，也是**所有垃圾收集器（CMS、G1、ZGC 等）99% 的工作重心所在**。

- **回收目标：** 主要是**对象实例**和**数组**。
    
- **回收特点：** 由于 Java 程序中对象的朝生夕灭特性非常明显，堆内的垃圾回收是最频繁的。无论是之前的分代模型（新生代的 Minor GC、老年代的 Major GC），还是 G1 的 Region 级别回收，本质上都是在堆内存中寻找并清理那些不再被任何存活对象引用的“死对象”。
    

### 3. GC 的“冷门”战场：方法区 / 元空间（Method Area / Metaspace）

很多人在初学时会误以为方法区不发生 GC，但其实是会的。尤其是在我们后端极其依赖 Spring、MyBatis 等框架（大量使用 CGLib 动态代理生成类）的场景下，这里的垃圾回收至关重要。

方法区主要回收两部分内容：

- **废弃的常量：** 例如常量池中某个字符串字面量（比如 `"meituan_saas_system"`），如果当前系统中没有任何一个 String 对象引用它，它就会被清理。
    
- **不再使用的类型（Class）：** 也就是类的卸载。这部分的回收条件极其苛刻（这也是为什么之前我们聊到自定义类加载器时，我特别强调处理不当会导致 Metaspace OOM 的原因）。要卸载一个类，必须**同时满足三个条件**：
    
    1. 该类所有的实例都已经被回收（Java 堆中不存在该类及其任何派生子类的实例）。
        
    2. 加载该类的**类加载器（ClassLoader）已经被回收**（这就是为什么热部署或者插件隔离必须废弃旧的 ClassLoader）。
        
    3. 该类对应的 `java.lang.Class` 对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法。
