在 JDK 8 中，方法区是逻辑概念，其一部分实现就是 **CodeCache（代码缓存）**，专门存放 JIT 即时编译器产生的本地机器码。下面用一个具体场景来说明。

---

### 例子：一个被频繁调用的方法
假设你有这样一个方法，在业务中被循环调用几十万次：
```java
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
}

// 使用
Calculator cal = new Calculator();
for (int i = 0; i < 100_000; i++) {
    cal.add(i, i + 1);
}
```

#### 1. 开始时：解释执行
- JVM 先通过解释器逐条字节码执行 `add` 方法。每次调用都要经历“字节码 → 机器码”的转换，虽然简单但速度一般。
- 同时，JVM 统计每个方法的调用次数和循环回边次数（这就是**热点探测**）。

#### 2. 达到阈值：触发 JIT 编译
- 当 `add` 方法被调用超过一定次数（默认 Client 模式 1500 次，Server 模式 10000 次，可通过 `-XX:CompileThreshold` 调整），它就被标记为**热点方法**。
- **C2 编译器**（Server Compiler）立即介入，将 `add` 方法的字节码**编译成平台相关的本地机器码**，比如在 x86 平台上生成高效的 `mov/add/ret` 指令序列。

#### 3. 存放本地机器码：CodeCache
- 生成的这段机器码不是放在堆里，也不是放在元空间（Metaspace，存类元数据），而是存放在 JVM 分配的独立本地内存区域——**CodeCache**。
- CodeCache 的大小由 `-XX:ReservedCodeCacheSize` 控制（默认值因平台和 JVM 模式而异，通常 240MB 左右）。
- JVM 会记录这个方法的入口点，后续对该方法的所有调用都会直接跳转到 CodeCache 中的机器码执行，完全绕过解释器。

#### 4. 执行加速
- 循环剩下的几万次调用，全部执行 CodeCache 里的本地代码，性能大幅提升，接近静态编译语言（如 C++）的速度。
- 如果此时通过 `-XX:+PrintCompilation` 参数观察，会看到类似这样的输出：
  ```
  123  1  %  Calculator::add @ 2 (24 bytes)   made not entrant
  124  2     Calculator::add (24 bytes)
  ```
  表示方法已被编译并缓存。

---

### CodeCache 与元空间、方法区的关系
- **逻辑上**：CodeCache 是方法区的一部分。JVM 规范规定方法区存储“已被虚拟机加载的类信息、常量、静态变量、即时编译器编译后的代码等数据”。
- **物理实现上**（JDK 8 的 HotSpot）：
    - 类元数据 → **元空间（Metaspace）**，在本地内存。
    - 即时编译代码 → **CodeCache**，也在本地内存，但独立于元空间，有单独的大小控制和垃圾回收机制。
    - 两部分合起来，构成了逻辑方法区。

---

### CodeCache 满的后果
由于 CodeCache 有大小限制，如果热点代码过多（如大量动态代理类、Spring AOP 生成的方法、Lambda 表达式等），可能填满 CodeCache，导致：
- JIT 编译被迫停止，后续所有方法都只能用解释执行，系统性能急剧下降。
- 抛出 `OutOfMemoryError: CodeCache is full`，甚至 JVM 可能选择关闭。
- 典型解决：增大 `-XX:ReservedCodeCacheSize=512m`，或通过 `-XX:+UseCodeCacheFlushing` 让 JVM 在满时自动清扫废弃的编译代码。

---

**总结一句话**：  
CodeCache 就是热点方法的“加速仓库”——JIT 把频繁调用的方法变成机器码存进去，以后直接执行本地代码，速度飙升；这块空间在逻辑上属于方法区，在物理上是 JDK 8 中一块独立可调的本地内存。
