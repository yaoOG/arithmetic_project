### 一、 线程私有区域（生命周期与线程相同，随线程而生，随线程而灭）

这部分内存区域不需要进行垃圾回收（GC），因为它们的内存分配和回收都具有确定性。

**1. 程序计数器（Program Counter Register）**

- **作用：** 它是一块较小的内存空间，可以看作是当前线程所执行的字节码的**行号指示器**。字节码解释器工作时，就是通过改变这个计数器的值来选取下一条需要执行的字节码指令（比如分支、循环、跳转、异常处理等）。
    
- **特点：** 这是整个 JVM 内存中**唯一一个没有规定任何 `OutOfMemoryError` 情况**的区域。如果线程正在执行的是一个 Java 方法，这个计数器记录的是正在执行的虚拟机字节码指令的地址；如果正在执行的是 Native 方法，这个计数器值则为空（Undefined）。
    

**2. Java 虚拟机栈（Java Virtual Machine Stacks）**

- **作用：** 它是 Java 方法执行的线程内存模型。每个方法被执行的时候，JVM 都会同步创建一个**栈帧（Stack Frame）**，用于存储**局部变量表、操作数栈、动态连接、方法出口**等信息。每一个方法被调用直至执行完毕的过程，就对应着一个栈帧在虚拟机栈中从入栈到出栈的过程。
    
- **异常：** 如果线程请求的栈深度大于虚拟机所允许的深度，将抛出 `StackOverflowError`（比如死递归）；如果栈扩展时无法申请到足够的内存，则会抛出 `OutOfMemoryError`。
    

**3. 本地方法栈（Native Method Stacks）**

- **作用：** 它与虚拟机栈发挥的作用非常相似，区别在于：虚拟机栈为 JVM 执行 Java 方法（也就是字节码）服务，而本地方法栈则是为 JVM 使用到的 **Native 方法（通常是 C/C++ 编写的底层方法）**服务。
    
- **异常：** 同样会抛出 `StackOverflowError` 和 `OutOfMemoryError`。在 HotSpot 虚拟机中，本地方法栈和虚拟机栈其实已经合二为一了。
    

### 二、 线程共享区域（生命周期与 JVM 相同，GC 的主战场）

这部分内存由所有线程共享，也是我们在做线上 JVM 调优时最关注的区域。

**4. Java 堆（Java Heap）**

- **作用：** 这是 JVM 所管理的内存中最大的一块。它的唯一目的就是**存放对象实例**（“几乎”所有的对象实例都在这里分配内存）和**数组**。
    
- **特点：** 它是垃圾收集器（GC）管理的绝对核心区域。为了更好地进行内存分配和回收，我们前面聊到的 G1 会把它划分为多个 Region，而传统的收集器会把它划分为新生代和老年代。
    
- **异常：** 如果在 Java 堆中没有内存完成实例分配，并且堆也无法再扩展时，Java 虚拟机将会抛出 `OutOfMemoryError`（这是线上最常见的 OOM 场景）。
    

**5. 方法区（Method Area）**

- **作用：** 用于存储已被 JVM 加载的**类信息（Class Metadata）、常量、静态变量、即时编译后的代码缓存等运行时数据。**
- **详细作用：**
```
  1.支撑类的动态加载：JVM 必须知道一个类有哪些方法、字段，才能进行查找、访问校验和字节码执行。方法区就是这种信息的“仓库”。
  2.实现反射与动态代理：Class.getMethod()、Field.get() 等反射操作，就是从方法区中读取类的元数据。没有方法区，反射就无从谈起。
  3.支撑动态链接：符号引用在首次调用时解析为直接引用，解析的信息就存在运行时常量池中。
  4.管理静态变量：静态变量被所有实例共享，需要随类加载而初始化，随类卸载而回收。
  5.方法调用与优化：解释器执行方法字节码时需要去方法区读取；JIT 编译后的机器码也要存放于此。
```
- **演进与实现：** 在 JDK 8 之前，HotSpot 虚拟机使用“永久代（PermGen）”来实现方法区，这部分内存受限于 JVM 的堆大小，很容易发生 OOM。**从 JDK 8 开始，永久代被彻底废弃，取而代之的是“元空间（Metaspace）”。** 元空间不再使用虚拟机的堆内存，而是直接使用**本地内存（Native Memory）**。
    
- **异常：** 如果方法区无法满足新的内存分配需求时，将抛出 `OutOfMemoryError`（比如系统利用 CGLib 动态生成了大量的代理类导致 Metaspace 撑爆）。
