
我们在上一个问题中知道了 STW 可能包含了漫长的“等线程”时间。那么在实战排查中，我们要如何把这几秒钟的黑盒像解剖一样，切开来看究竟是哪一步慢了呢？

作为一个排查过无数线上抖动问题的老兵，我通常会根据您使用的 **JDK 版本**，采用以下三种维度的观测手段：

### 一、 经典解法：JDK 8 及以前的 Safepoint 统计日志

在 JDK 8 时代，这是我们最常用的排查神技。我们需要在 JVM 启动参数中加上这两句核心指令：

Plaintext

```
-XX:+PrintSafepointStatistics 
-XX:PrintSafepointStatisticsCount=1
```

开启后，每当发生一次 STW，标准输出日志里就会打印出极其硬核的一行数据，格式大概是这样的：

Plaintext

```
vmop                    [threads: total initially_running wait_to_block] [time: spin block sync cleanup vmop] page_trap_count
RevokeBias              [      10                 1             1  ] [      0     2    2       0     5  ]               0
```

**老兵为您拆解这串“天书”中的 `[time]` 区域（单位全为毫秒）：**

1. **`spin`（自旋等待）**：JVM 刚发出安全点集结令，正在空转等待那些还在跑代码的线程到达安全点的时间。如果这个值很大，说明有**长循环或者大计算没有插入安全点**（就是咱们上一个问题聊的情况）。
    
2. **`block`（阻塞等待）**：等待部分线程挂起（Park）所花费的时间。如果这个值很大，通常和操作系统的**缺页中断（Page Fault）或者 CPU 调度抢占**有关。
    
3. **`sync`（同步总耗时）**：这是个汇总值，**`sync = spin + block`**。它代表了从发起命令到所有人都乖乖停下，总共花了多久。
    
4. **`cleanup`（清理动作）**：到达安全点后，真正干活前，JVM 会做一些内部清理（比如更新内联缓存、释放死掉的线程等）。通常是 0 或 1 毫秒，很高的话可能是内部 Bug。
    
5. **`vmop`（VM 操作真身）**：这才是**真正干正事的时间**！如果这次 STW 是因为垃圾回收，这就是 GC 耗时；如果是为了撤销偏向锁（像上面那行日志的 `RevokeBias`），这就是撤销锁的耗时。
    

**结论：这一次 STW 的总耗时 ≈ `sync` + `cleanup` + `vmop`。**

---

### 二、 现代解法：JDK 9+ 的统一日志框架 (Unified Logging)

JDK 9 彻底重构了 JVM 的日志系统，彻底废弃了上面那些杂乱的 `-XX` 参数。现在查看 STW 阶段变得极其优雅。

您只需要在启动参数加上：

Plaintext

```
-Xlog:safepoint=debug
```

日志会输出类似这种极其人类可读的信息：

Plaintext

```
[0.123s][debug][safepoint] Entering safepoint region: RevokeBias
[0.124s][debug][safepoint] Safepoint synchronization initiated. (10 threads)
[0.139s][info ][safepoint] Safepoint "RevokeBias", Time since last: 1.000 s, Reaching safepoint: 0.015 s, At safepoint: 0.005 s, Total: 0.020 s
```

**核心指标秒懂：**

- **`Reaching safepoint` (0.015 s)**：对应老版本的 `sync` 时间，也就是等大家集合花了多长时间。
    
- **`At safepoint` (0.005 s)**：对应老版本的 `vmop` + `cleanup`，也就是真正干活花了多长时间。
    
- **`Total` (0.020 s)**：这就是应用线程总共被 STW 冻结的时间。

### 三、 终极无感利器：JFR (Java Flight Recorder) + JMC

如果您在极其核心的生产环境，不敢乱加日志参数（怕日志狂打拖垮磁盘），那么 **JFR（Java 飞行记录器）** 就是最终极的武器。

它是集成在 JVM 内核深处的诊断工具，性能损耗通常不到 1%。

1. **开启 JFR 录制**：可以直接用 `jcmd` 命令动态开启一段 60 秒的录制。
    
    Bash
    
    ```
    jcmd <pid> JFR.start duration=60s filename=safepoint_record.jfr
    ```
    
2. **使用 JMC (JDK Mission Control) 分析**：将下载下来的 `.jfr` 文件导入 JMC。
    
3. **定位图表**：在 JMC 的界面中，找到 `Java Virtual Machine` -> `Safepoints` 选项卡。
    

您会看到一个极其直观的图形化界面，直接帮您把每一次 Safepoint 拆解成了三个阶段，并画出耗时柱状图：

- `Safepoint State Synchronization` (对应等待集合时间)
    
- `Safepoint Cleanup`
    
- `Safepoint End`
    
    甚至它还能直接列出是**哪一个具体的线程**拖慢了大家到达安全点的速度，直接抓出罪魁祸首！

### 💡 老兵实战避坑指南（绝对红线）

**千万、千万不要把 JDK 8 的 `-XX:+PrintSafepointStatistics` 长期开在生产环境中！**

这往往是很多新手容易犯的致命错误。因为这个参数一旦开启，JVM 每发生一次 Safepoint，就会强行去系统底层获取一次高精度的硬件时钟，并且要往标准输出（stdout）写数据。

如果您的系统因为高并发或者偏向锁冲突，每秒钟发生几百次 Safepoint，**打印统计日志本身的动作，就会引发极其严重的性能损耗甚至新的 STW 卡顿！**

排查完问题，或者定位到病因后，一定要立刻把这个参数摘掉。这也是为什么到了 JDK 9 以后，官方把它融入了异步的统一日志框架中的根本原因。
