### 1. 生命周期两端：起与灭

- **NEW（新建）**： 当我们 `new Thread()` 创建了一个线程对象，但还没有调用 `start()` 方法时，线程就处于这个状态。此时它只是堆内存中的一个普通 Java 对象，操作系统还没有真正为其分配底层线程资源。
    
- **TERMINATED（终止）**： 线程的 `run()` 方法执行结束，或者因为未捕获的异常导致强制退出时，就会进入这个状态。这就意味着线程的生命周期彻底结束了。
    

### 2. 核心执行状态

- **RUNNABLE（可运行）**： **这是面试中极其容易产生误解的一个状态。** 在操作系统的层面，线程通常被分为 Ready（就绪，等待 CPU 调度）和 Running（正在运行）。但是 **Java 并没有细分这两种状态**。在 Java 内存模型中，只要线程调用了 `start()`，无论它当前是在 CPU 上飞速执行，还是在就绪队列里排队等待操作系统分配时间片，它的 Java 状态统统都是 `RUNNABLE`。 _(实战经验：在查生产问题时，如果 `jstack` 看到大量线程处于 `RUNNABLE`，不代表系统运行良好，反而可能是死循环或者在进行密集的 CPU 计算导致 CPU 打满。)_

### 3. 三种“暂停”状态（核心考点）

在实际开发中，导致系统吞吐量下降的往往是这三种状态。特别是 `BLOCKED` 和 `WAITING` 的区别，是衡量候选人底层功底的试金石。

- **BLOCKED（阻塞）**： **这个状态是专门为 `synchronized` 关键字准备的。** 只有当一个线程试图获取一个内部的对象监视器锁（Monitor Lock），但该锁正被其他线程持有时，这个线程才会进入 `BLOCKED` 状态。
    
- **WAITING（无限期等待）**： 线程在这里会一直“睡死”过去，直到被其他线程显式地唤醒。触发 `WAITING` 状态的常见条件有：
    
    1. 没有设置 timeout 值的 `Object.wait()`（需要其他线程调用 `notify/notifyAll`）。
        
    2. 没有设置 timeout 值的 `Thread.join()`。
        
    3. `LockSupport.park()` 方法。 **进阶重点**：我们日常用的 JUC 包下的锁（比如 `ReentrantLock`），底层是基于 AQS 和 `LockSupport.park()` 实现的。所以，**获取 `ReentrantLock` 失败被挂起的线程，其状态是 `WAITING`（或 `TIMED_WAITING`），绝对不是 `BLOCKED`！**
        
- **TIMED_WAITING（限期等待）**： 这就是带有“超时机制”的等待。线程不会死等，时间一到系统会自动唤醒它。触发场景包括：
    
    1. `Thread.sleep(long millis)`（这是最常见的）。
        
    2. 设置了超时的 `Object.wait(timeout)` 或 `Thread.join(timeout)`。
        
    3. `LockSupport.parkNanos()` 或 `LockSupport.parkUntil()`。
