
###  一、线程池的七大核心参数

- **`corePoolSize`（核心线程数）**：线程池中的“常驻员工”。即使它们闲着，线程池也不会回收它们（除非设置了 `allowCoreThreadTimeOut`）。
    
- **`maximumPoolSize`（最大线程数）**：线程池允许的最大并发数，也就是“常驻员工” + “临时工”的总和。
    
- **`keepAliveTime`（空闲存活时间）**：“临时工”线程空闲多久后会被销毁。
    
- **`unit`（时间单位）**：`keepAliveTime` 的单位（如秒、毫秒）。
    
- **`workQueue`（阻塞队列）**：用于存放来不及处理的等待任务。**这是最容易引发 OOM 的地方**，必须使用有界队列（如 `ArrayBlockingQueue` 或指定大小的 `LinkedBlockingQueue`）。
    
- **`threadFactory`（线程工厂）**：用于创建线程。在生产中，我们通常用它来给线程起一个有业务意义的名字（如 `order-processor-thread-1`），方便线上 `jstack` 排查问题。
    
- **`handler`（拒绝策略）**：当队列塞满，且线程数达到最大限制时，对新任务的“兜底”处理方案。

#### 三、如何配置自定义线程池？

**核心参数的预估（CPU 密集型 vs IO 密集型）**：

- 如果是**计算密集型**（如加密、复杂算法），线程数通常设为 `CPU 核心数 + 1`，避免过多的线程上下文切换带来额外开销。
    
- 如果是**IO 密集型**（如频繁调用数据库、RPC 远程请求），大部分时间都在等待网络响应，线程数可以设大一点，通常经验值是 `CPU 核心数 * 2`，或者根据 `(线程等待时间 / 线程计算时间 + 1) * CPU 核心数` 的公式精确压测推算。
