从底层本质上来说，Java 中创建线程**有且仅有一种方式**：那就是通过 `new Thread()` 创建对象，并调用其底层的 native 方法 `start0()` 来向操作系统申请线程资源。

至于我们在代码里写的那些不同的实现方式，本质上都是对这个底层过程的**API 层面封装**。日常开发中，我们主要有以下几种主流方式：

---

### 1. 基础方式：继承 `Thread` 类

这是最直观的方式，重写 `run()` 方法，然后调用 `start()` 启动。

- **优点**：代码简单，直接使用 `this` 就能获取当前线程对象。
    
- **痛点**：Java 是单继承机制。如果你的类已经继承了其他业务父类，就无法再继承 `Thread` 了。在实际业务代码中，这种方式因为**侵入性太强**，我们几乎不使用。


```Java
public class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("执行任务的线程：" + Thread.currentThread().getName());
    }
}

// 调用方式
// new MyThread().start();
```


### 2. 接口解耦：实现 `Runnable` 接口

将“线程要执行的任务”和“线程对象本身”分离开来。我们创建一个实现了 `Runnable` 接口的任务对象，然后把它作为 target 丢给 `Thread` 去执行。

- **优点**：完美避开了单继承的限制，符合面向对象设计中“组合优于继承”的原则；并且多个线程可以共享同一个 `Runnable` 任务实例，适合处理并发共享资源的场景。
    
- **痛点**：`run()` 方法的签名是没有返回值的（`void`），并且不能抛出受检异常（Checked Exception）。如果我们需要知道任务执行的结果，或者捕获任务里的异常，它就无能为力了。


```Java
// 使用 Lambda 表达式直接定义 Runnable 的 run 逻辑
Runnable myTask = () -> {
    System.out.println("执行无返回值的任务：" + Thread.currentThread().getName());
};

// 调用方式（仍然是自己 new 线程，生产不推荐）
// new Thread(myTask, "Runnable-Thread-1").start();
```

### 3. 异步回调：实现 `Callable` 接口 + `FutureTask`

为了弥补 `Runnable` 的缺陷，JDK 1.5 引入了 `Callable` 接口。它的 `call()` 方法不仅有泛型返回值，还能抛出异常。

- **工作机制**：由于 `Thread` 类的构造器只认 `Runnable`，不认 `Callable`，所以我们需要一个桥梁——`FutureTask`。`FutureTask` 实现了 `Runnable` 和 `Future` 接口。我们把 `Callable` 包装进 `FutureTask`，再把 `FutureTask` 丢给 `Thread` 执行，最后通过 `FutureTask.get()` 阻塞获取返回值。
    
- **适用场景**：多线程并行计算，最后需要汇总结果的场景（比如大促页面同时并行拉取商品信息、优惠券信息和库存信息）。


```Java
import java.util.concurrent.Callable;
import java.util.concurrent.ExecutionException;
import java.util.concurrent.FutureTask;

public class CallableDemo {
    public static void main(String[] args) {
        // 1. 定义 Callable 任务，泛型指定返回 Integer
        Callable<Integer> calcTask = () -> {
            System.out.println("正在进行复杂计算...");
            Thread.sleep(1000); // 模拟耗时操作
            return 1024;
        };

        // 2. 将 Callable 包装进 FutureTask
        FutureTask<Integer> futureTask = new FutureTask<>(calcTask);

        // 3. 丢给 Thread 执行
        new Thread(futureTask, "Callable-Thread").start();

        try {
            // 4. 主线程阻塞等待获取结果 (get 方法会阻塞直到任务完成)
            Integer result = futureTask.get(); 
            System.out.println("主线程拿到结果：" + result);
        } catch (InterruptedException | ExecutionException e) {
            e.printStackTrace(); // 可以捕获异步任务里抛出的异常
        }
    }
}
```

### 4. 生产环境的绝对标准：线程池（`ExecutorService`）

作为一名老兵，我必须向您强调：**在实际的生产环境中，我们严禁通过手工 `new Thread()` 来创建线程。** 所有的异步任务，都必须提交给**线程池**来统一管理和调度。我们严禁手动 `new Thread()`，也严禁使用 `Executors.newFixedThreadPool()` 这种隐藏了阻塞队列大小的“毒药”方法（容易导致 OOM）。我们必须手动声明线程池的 7 大核心参数。

- **为什么这是生产强制规范？**
    
    1. **降低资源消耗**：手工无限制地创建和销毁线程，会造成巨大的系统开销甚至导致 OOM。线程池通过复用已存在的线程，极大降低了系统负担。
        
    2. **提高响应速度**：当任务到达时，不需要等待线程创建就能立即执行。
        
    3. **便于统一管理**：我们可以配置线程池的核心线程数、最大线程数、拒绝策略等，实现对系统并发度的精确控制。
        


```Java
import java.util.concurrent.*;

public class ThreadPoolDemo {
    
    // 声明一个全局可复用的线程池
    private static final ThreadPoolExecutor EXECUTOR = new ThreadPoolExecutor(
            2,                               // 1. 核心线程数 (常驻线程)
            5,                               // 2. 最大线程数 (峰值线程)
            60L,                             // 3. 空闲线程存活时间
            TimeUnit.SECONDS,                // 4. 时间单位
            new ArrayBlockingQueue<>(100),   // 5. 阻塞队列 (必须指定有界队列大小，防止OOM)
            Executors.defaultThreadFactory(),// 6. 线程工厂 (生产中通常用 Guava 的 ThreadFactoryBuilder 自定义线程名)
            new ThreadPoolExecutor.CallerRunsPolicy() // 7. 拒绝策略 (队列满了且线程满了，由提交任务的线程自己执行)
    );

    public static void executeBiz() {
        // 提交无返回值任务 (execute)
        EXECUTOR.execute(() -> {
            System.out.println("异步发送邮件：" + Thread.currentThread().getName());
        });

        // 提交有返回值任务 (submit)
        Future<String> future = EXECUTOR.submit(() -> {
            return "订单数据处理完毕";
        });
        
        // 正常系统关闭时，需要优雅停机
        // EXECUTOR.shutdown();
    }
}
```
