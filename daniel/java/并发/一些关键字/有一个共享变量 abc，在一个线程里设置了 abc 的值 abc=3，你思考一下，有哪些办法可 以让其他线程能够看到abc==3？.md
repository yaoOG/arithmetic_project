
面试官您好！这个问题非常切中并发编程的要害，它直指 Java 内存模型（JMM）中的核心痛点：**内存可见性（Visibility）**。

在多线程环境下，由于 CPU 高速缓存和 JMM 工作内存的存在，如果一个线程修改了共享变量 `abc = 3`，默认情况下这个修改可能只会停留在该线程的本地缓存中。要让其他线程立刻看到这个修改，本质上是需要在代码之间建立 JMM 规定的 **Happens-Before（先行发生）关系**。

作为有经验的开发者，如果遇到这种需求，我通常会根据具体的业务场景，从以下 5 个维度来给出解决方案：

### 1. 最直接、最轻量级的解法：`volatile` 关键字

这是专门为了解决可见性和有序性而生的关键字。

- **做法**：将变量声明为 `volatile int abc = 0;`。
    
- **底层逻辑**：当一个线程将 `abc` 赋值为 3 时，JVM 会插入写屏障，强制将新值立刻刷新回主内存；同时通过底层的缓存一致性协议（如 MESI），立即使其他所有线程工作内存中的 `abc` 缓存失效。当其他线程再去读 `abc` 时，必须重新去主内存拉取最新值。
    
- **适用场景**：仅仅作为状态标记（如 `boolean flag`），不涉及依赖当前值的复合操作。
    

### 2. 无锁的高并发利器：JUC 原子类（`AtomicInteger`）

如果我们不仅需要让别人看到 `abc = 3`，后续还可能涉及 `abc++` 这种累加操作，`volatile` 就不够用了。

- **做法**：使用 `AtomicInteger abc = new AtomicInteger(0);`，通过 `abc.set(3)` 来修改。
    
- **底层逻辑**：翻开 `AtomicInteger` 的源码，你会发现它内部包装的真正数据 `value`，本身就是用 `volatile` 修饰的！所以它天然完美具备可见性。同时，它利用底层 Unsafe 类的 CAS（Compare-And-Swap）机制，还保证了后续修改操作的原子性。
    

### 3. 重量级互斥同步：`synchronized` 关键字

老牌的同步机制，虽然重，但非常稳。

- **做法**：在修改和读取 `abc` 的方法或者代码块上，加上 `synchronized` 锁。
    
- **底层逻辑**：JMM 规范明确规定了关于锁的 Happens-Before 规则：**对一个锁的解锁，先行发生于随后对这个锁的加锁**。更通俗地说，线程在释放锁（`unlock`）之前，必须把变量同步回主内存；而其他线程在获得锁（`lock`）时，会清空工作内存，重新从主内存读取。
    

### 4. 显式锁与并发工具：`ReentrantLock` / `ReadWriteLock` 等

- **做法**：使用 JUC 包下的锁对象，`lock.lock()` 和 `lock.unlock()` 包裹住对 `abc` 的操作。
    
- **底层逻辑**：`ReentrantLock` 的底层是 AQS（AbstractQueuedSynchronizer）框架。AQS 的核心是一个名为 `state` 的控制同步状态的变量，而这个 `state` 正是 **`volatile` 修饰**的。在使用 JUC 锁时，获取和释放锁的过程必然会读写这个 `volatile state`，根据 JMM 的**传递性规则**，这就顺带保证了被锁保护的 `abc` 变量的可见性。
    

### 5. 利用线程生命周期的“天然”可见性

这属于在特定架构或流程设计时经常使用的“巧劲”，不需要额外加锁：

- **`Thread.start()` 规则**：如果线程 A 里执行了 `abc = 3`，紧接着执行了 `threadB.start()`。那么 JMM 保证线程 B 启动后，绝对能看到 `abc == 3`。
    
- **`Thread.join()` 规则**：如果线程 B 修改了 `abc = 3`，然后线程 A 执行了 `threadB.join()` 等待 B 结束。只要 `join()` 成功返回，线程 A 绝对能看到 B 所做的修改。
    
- **`Future.get()`**：在使用线程池提交 `Callable` 任务时，只要 `Future.get()` 拿到了结果，任务内部对 `abc` 的修改对主线程一定是可见的。
    

### 总结

在实际的代码评审（Code Review）中：

- 如果 `abc` 只是一个**单向传递的状态/开关**，我会毫不犹豫地使用 **`volatile`**。
    
- 如果 `abc` 是一个**业务指标或计数器**，我会使用 **`AtomicInteger`**。
    
- 如果修改 `abc` 只是**一段复杂复合业务逻辑（需要保证一致性）的其中一步**，那我才会动用 **`synchronized`** 或 **`ReentrantLock`**。
