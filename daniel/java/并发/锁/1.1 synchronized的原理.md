在 JDK 1.6 之前，`synchronized` 被大家嫌弃为“重量级锁”，性能低下，大家都喜欢去用 JUC 里的 `ReentrantLock`。但 JDK 1.6 的开发团队痛定思痛，对 `synchronized` 进行了极其疯狂的底层重构，引入了“锁升级”机制，让它的性能有了质的飞跃。

作为一个老兵，我通常会从“字节码层面”、“对象内存布局”**，以及最核心的**“锁升级过程”三个维度，为您深度剖析它的底层原理：

### 一、 字节码层面：锁是怎么加上的？

当我们在代码里写下 `synchronized` 时，编译器（`javac`）会把它翻译成 JVM 认识的底层指令：

1. **修饰代码块**：

    编译器会在代码块的入口插入 **`monitorenter`** 指令，在出口和异常处插入 **`monitorexit`** 指令。
    
    _线程执行到 `monitorenter` 时，会尝试获取对象的 Monitor（管程）的所有权；执行 `monitorexit` 时释放。_
    
2. **修饰方法**：
    
    编译器并不会生成这两个指令，而是在该方法的常量池中打上一个 **`ACC_SYNCHRONIZED`** 的标志位。
    
    _JVM 在调用方法时，看到这个标志，就会隐式地去获取锁。_
    

### 二、 物理层面：锁到底存在哪里？（对象头与 Mark Word）

很多初学者疑惑：既然万物皆可为锁，那这把锁到底存在哪里呢？

**答案是：存在对象自身的内存里。**

在 JVM 中，任何一个对象在内存中都由三部分组成：

1. **对象头（Object Header）** 👈 **锁的秘密就在这里**
    
2. 实例数据（Instance Data，你定义的字段）
    
3. 对齐填充（Padding，为了凑齐 8 字节整数倍）
    

对象头里面有一块极度核心的区域，叫做 **Mark Word（标记字段）**。它就是一个由 32 位或 64 位组成的二进制数据，里面记录了对象的哈希码、GC 年龄，最重要的就是——**锁标志位**。

---

### 三、 动态演进：锁的升级过程（面试绝杀点）

在 JDK 1.6 之后，`synchronized` 不会一上来就去向操作系统申请重量级互斥锁。它是一个**极其抠门**的动态防御系统，会根据“到底有几个人来抢”，动态地改变 Mark Word 里的状态，也就是经典的**锁升级过程：无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁**。

#### 1. 偏向锁 (Biased Lock) —— “专车接送”

- **场景**：很多时候，一把锁在整个生命周期内，其实**都只有一个线程在反复进出**。
    
- **原理**：当线程 A 第一次访问同步块时，JVM 会把对象头里的锁标志位改成“偏向锁”，并记录下**线程 A 的 ID**。
    
- **效果**：以后线程 A 每次来，只要看一眼对象头里的 ID 是自己的，连 CAS 操作都不用做，直接长驱直入！此时性能最高。
    

#### 2. 轻量级锁 (Lightweight Lock) —— “自旋抢占”

- **场景**：有一天，线程 B 突然也想进这段代码。JVM 发现偏向锁的 ID 是 A，出现了轻度竞争！
    
- **原理**：偏向锁立刻被撤销，升级为**轻量级锁**。此时，JVM 会在线程 A 和线程 B 的栈帧中各自开辟一块空间（叫 Lock Record），然后让他们俩用 CAS（比较并交换）去争抢对象头的 Mark Word 指针。
    
- **自旋等待**：如果 A 抢到了，B 没抢到，B 绝不会去睡觉（挂起），因为 B 觉得 A 很快就会出来。于是 B 会在 CPU 里执行死循环（**自旋 Spin**），疯狂尝试获取锁。
    
- **效果**：没有操作系统的介入，没有线程的上下文切换阻塞，纯靠 CPU 空转争取时间，非常适合短代码块。
    

#### 3. 重量级锁 (Heavyweight Lock) —— “操作系统的铁拳”

- **场景**：如果线程 B 自旋了好几千次，A 还没出来；或者这时候线程 C、D、E 全都跑过来抢锁了。
    
- **原理**：JVM 觉得：“竞争太激烈了，大家都在自旋，把 CPU 都烧坏了！”于是，轻量级锁彻底膨胀为**重量级锁**。
    
- **底层机制**：此时，对象的 Mark Word 会指向一个底层的 **Monitor（管程/ObjectMonitor，由 C++ 实现）**。Monitor 底层强依赖于操作系统的 **Mutex Lock（互斥量）**。
    
- **效果**：除了拿到锁的 A，剩下的 B、C、D、E 全被打包扔进 Monitor 的 `_EntryList` 阻塞队列中，**强行剥夺 CPU 执行权，让线程进入休眠（Blocked）状态**。等 A 完事了，再由操作系统去唤醒他们。此时性能损耗最大。
    

---


**老兵避坑指南：**

锁的升级过程在绝大多数情况下是**不可逆的（单向升级）**！也就是说，一旦为了应对瞬时高并发膨胀成了重量级锁，就算后来没竞争了，它也不会自动降级回偏向锁（只有在 GC 时或者极特殊极特殊的重偏向优化机制下才可能发生）。这也是为什么在长期的极高并发场景下，直接关闭偏向锁（`-XX:-UseBiasedLocking`）有时反而能提升性能，因为它省去了撤销偏向锁带来的额外 Safepoint 停顿开销。
