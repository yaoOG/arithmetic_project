为了把 AQS（AbstractQueuedSynchronizer）吃透，光看抽象的源码是极其痛苦的。

作为一个老兵，我最喜欢用**“一场三个人的抢夺战”**来给团队推演 AQS 的核心流转。

我们以最经典的 **`ReentrantLock`（非公平锁）** 为例。

假设 AQS 的初始状态是：`state = 0`（没有人拿锁），CLH 同步队列为空（`head = null`, `tail = null`）。

现在，**线程 A、线程 B、线程 C** 依次登场，我们一步步拆解 AQS 的底层做了什么：

### 第一幕：线程 A 长驱直入（无竞争）

1. **动作**：线程 A 调用 `lock()`。
    
2. **底层流转**：
    
    - 线程 A 发现 `state == 0`，直接执行 **CAS(0, 1)** 成功。
        
    - 线程 A 将自己设置为独占锁的拥有者（`setExclusiveOwnerThread(A)`）。
        
3. **当前状态**：`state = 1`。A 在执行业务代码。**注意：此时 AQS 的队列根本没有被创建，因为没有竞争，不需要排队，极其克制。**
    

### 第二幕：线程 B 撞墙与队列初始化（核心难点）

1. **动作**：线程 A 还没执行完，线程 B 也调用了 `lock()`。
    
2. **底层流转**：
    
    - 线程 B 尝试 CAS(0, 1) 失败（因为 `state` 已经是 1 了）。
        
    - B 进入 **`acquire(1)`** 主流程，调用 **`tryAcquire`** 再次失败。
        
    - B 被迫进入 **`addWaiter(Node.EXCLUSIVE)`** 准备排队。
        
    - **🔥 面试绝杀点（延迟初始化）**：AQS 发现队列是空的！它绝不会直接把 B 放进去，而是先执行一个 `enq()` 方法，**创建一个空的 Node（称为 Dummy Head / 虚拟头节点）**。这个虚拟头节点其实代表的就是当前正在拿锁的**线程 A**。
        
    - 接着，AQS 把线程 B 包装成一个 Node，挂在这个虚拟头节点的后面。此时队列变成：`[Dummy Head] <-> [Node B]`。
        
    - B 进入 **`acquireQueued()`** 死循环：B 发现自己的前驱是 Head，不死心，再尝试抢一次锁（`tryAcquire`），还是失败。
        
    - B 决定睡觉，调用 `shouldParkAfterFailedAcquire`，**把前驱节点（Dummy Head）的 `waitStatus` 改为 `-1`（SIGNAL）**。意思是：“头节点大哥，我 B 要睡了，你办完业务记得叫醒我！”
        
    - 最后，B 调用 `LockSupport.park()`，彻底挂起休眠。
        
3. **当前状态**：`state = 1`。队列结构：`[Head(waitStatus=-1)] <-> [Node B(parked)]`。
    

### 第三幕：线程 C 绝望排队

1. **动作**：线程 A 依然没释放，线程 C 来抢锁。
    
2. **底层流转**：
    
    - C CAS 失败，`tryAcquire` 失败。
        
    - 调用 `addWaiter`，发现队列已经建好了。直接用 CAS 尾插法，把自己挂在 Node B 的后面。
        
    - C 进入 `acquireQueued`，发现前驱是 B（不是 Head），连抢锁的资格都没有。
        
    - C 把 B 的 `waitStatus` 改为 `-1`，然后安心调用 `park()` 睡觉。
        
3. **当前状态**：`state = 1`。队列结构：`[Head(waitStatus=-1)] <-> [Node B(waitStatus=-1)] <-> [Node C(parked)]`。
    

### 第四幕：线程 A 释放锁与唤醒 B

1. **动作**：线程 A 办完业务，调用 `unlock()`。
    
2. **底层流转**：
    
    - A 调用 **`release(1)`**，执行 `tryRelease`，把 `state` 从 1 改回 0。并清空独占线程标记。
        
    - A 检查 AQS 队列，发现 `head != null` 且 `head.waitStatus == -1`。这说明后面有人在等！
        
    - A 调用 **`unparkSuccessor(head)`**。它顺着 Head 往下找，找到了排在最前面的有效节点 Node B。
        
    - A 调用 `LockSupport.unpark(B.thread)`，把 B 强行叫醒。
        
3. **当前状态**：`state = 0`。B 醒了，准备抢锁。C 还在睡。
    

### 第五幕：线程 B 登基当皇帝

1. **动作**：线程 B 被唤醒后，从之前 `park()` 阻塞的地方继续往下执行代码。
    
2. **底层流转**：
    
    - B 还在 `acquireQueued` 的死循环里。它醒来第一件事就是**再次调用 `tryAcquire`**。
        
    - 这次 `state == 0`，B CAS(0, 1) 成功，拿到锁！
        
    - **🔥 队列出队逻辑**：B 把自己这个 Node 设置为新的 Head（清空里面的线程引用，把自己变成了新的 Dummy Head）。然后把老的 Dummy Head 踢出链表，等待 GC 回收。
        
3. **当前状态**：`state = 1`。队列结构：`[New Head (原来是 B)] <-> [Node C(parked)]`。

**老兵连环问预警：**

看完这个流程，很多高级面试官会立刻抛出一个刁钻的问题：**“为什么唤醒 B 的时候，如果 B 节点因为超时被取消（CANCELLED = 1）了，AQS 不是从前往后找，而是要从 Tail（尾部）从后往前找有效节点呢？”**

- **答案核心**：因为在 `addWaiter` 尾插法入队时，底层代码是先执行 `node.prev = pred`，再执行 CAS 替换 Tail，最后才执行 `pred.next = node`。如果在 CAS 成功后，`pred.next = node` 还没来得及执行，此时从前向后遍历，`next` 指针可能是断的（null）！但 `prev` 指针绝对是可靠的！所以从后向前遍历是极端并发下最安全的兜底策略。
