早期版本的 `synchronized` 因为强依赖操作系统的互斥量（Mutex Lock），导致线程上下文切换开销巨大，被称为“重量级锁”。但从 JDK 1.6 开始，JVM 对它进行了极其深度的优化，引入了锁升级机制，使其性能得到了质的飞跃。

我们可以从**使用方式、底层原理、以及核心特性**三个维度来彻底拆解它：

## 一、使用方式
作为一个老兵，我通常会把 `synchronized` 的使用方式分为 **3 种**对应着不同的锁粒度：

### 1、修饰实例方法（对象锁）

这是最简单的用法，直接把关键字加在普通的成员方法上。

- **锁究竟是谁？** 锁的是 **当前对象实例（`this`）**。

- **适用场景**：当多个线程操作**同一个对象**的成员变量时。


```Java
public class BankAccount {
    private int balance = 1000;

    // 锁住的是调用这个方法的具体对象 (this)
    public synchronized void deduct(int amount) {
        balance -= amount;
    }
}
```

**老兵排雷**：如果线程 A 操作的是 `account1.deduct()`，线程 B 操作的是 `account2.deduct()`，因为它们是两个不同的实例对象（两把不同的锁），它们**绝对不会互相阻塞**，会完全并行执行！

### 2、 修饰静态方法（类锁）

把关键字加在 `static` 静态方法上。

- **锁究竟是谁？** 锁的是 **当前类的 Class 对象**（比如 `BankAccount.class`）。我们在聊 JVM 方法区时说过，每个类在内存里只有唯一的一份 Class 对象。

- **适用场景**：用来保护静态变量，或者在单例模式下保护全局唯一的资源。


```Java
public class BankFactory {
    public static int totalBanks = 0;

    // 锁住的是全局唯一的 BankFactory.class 对象
    public static synchronized void addBank() {
        totalBanks++;
    }
}
```

**老兵排雷**：这是一把“全局锁”。不管你 `new` 了多少个 `BankFactory` 的实例，只要线程调用了这个静态方法，所有线程都要在同一把锁前面排队。**它和上面的对象锁互不干扰**（一个线程拿着实例锁，另一个线程可以同时拿类锁）。

### 3、 修饰代码块（细粒度锁）

这是高级开发最推崇的用法。不要锁整个方法，只锁容易发生并发冲突的那几行代码。

- **锁究竟是谁？** 锁的是 **括号里传入的那个对象**。你可以传入任何对象。

**方式 A：传入 `this`（等同于修饰实例方法，但粒度更小）**

```Java
public void complexTask() {
    System.out.println("这里是耗时的非共享操作，不加锁，大家一起跑...");
    
    // 只锁住会发生冲突的这一小块
    synchronized (this) {
        System.out.println("修改共享变量，排队进入");
    }
}
```

**方式 B：传入 `.class`（等同于类锁）**

```Java
synchronized (BankFactory.class) {
    // 全局排队执行
}
```

**方式 C：传入一个专门的私有锁对象（极致安全）**

```Java
public class SafeTask {
    // 专门 new 一个没有实际业务意义的对象，只用来当锁
    private final Object lock = new Object();

    public void doWork() {
        synchronized (lock) {
            // 业务逻辑
        }
    }
}
```

---


### ⚠️ 老兵实战避坑指南（绝对红线）

在使用 `synchronized (obj)` 时，有几个对象是绝对不能当做锁的：

1. **绝对不要锁 `String` 常量**：比如 `synchronized ("myLock")`。因为 Java 中存在字符串常量池，如果别人在你不知道的第三方库里也写了 `synchronized ("myLock")`，你们俩就会莫名其妙地竞争同一把锁，导致极其诡异的死锁或阻塞。
    
2. **绝对不要锁基本类型的包装类（如 `Integer`, `Boolean`）**：比如 `Integer count = 1; synchronized(count)`。因为当你做 `count++` 操作时，底层其实是产生了一个全新的 `Integer` 对象，锁的引用瞬间就变了，锁直接失效！


在工程实践中，如果您想用代码块锁，**最推荐的方式就是 `private final Object lock = new Object();`**，将锁彻底私有化，不给外部任何捣乱的机会。

## 二、JDK 1.6 之后的“锁升级”机制

为了避免每次都直接向操作系统申请重量级锁，JVM 在对象头（Mark Word）中记录了锁的状态，并随着竞争的加剧而逐步升级（**只能升级，不能降级**）：

- **无锁状态：** 对象刚被创建。

- **偏向锁 (Biased Locking)：** 假设大多数情况下锁不仅不存在多线程竞争，而且总是由同一个线程多次获得。JVM 会直接在对象头里记录下这个线程的 ID。下次这个线程再来，连 CAS 操作都不用做，直接放行。_(注：由于维护成本高，JDK 15 默认废弃了此特性)_。

- **轻量级锁 (Lightweight Locking)：** 当有第二个线程来竞争锁时，偏向锁升级为轻量级锁。未抢到锁的线程不会立刻阻塞陷入内核态，而是会在用户态执行一个很短的**自旋（Spinning，通常是 CAS 操作）**，也就是写个空循环等一等，看看持有锁的线程会不会马上释放。

- **重量级锁 (Heavyweight Locking)：** 如果自旋次数超过了阈值（或者自适应自旋认为等待无望），竞争变得非常激烈，轻量级锁就会升级为重量级锁。此时，底层会调用操作系统的 Mutex Lock，把所有没抢到锁的线程全部挂起，交由操作系统去调度。
    

## 三、 核心特性总结

- **悲观与排他：** 它是一种典型的悲观锁和排他锁。

- **非公平性：** 当锁被释放时，刚来请求的线程和已经在 `_EntryList` 里排队的线程会一起去抢锁，刚来的线程大概率能利用 CPU 缓存直接抢到，这打破了先来后到的公平性，但也极大地提升了吞吐量。

- **可重入性：** 同一个线程在获取锁之后，可以继续调用该锁对象的其他 `synchronized` 方法，底层会通过一个计数器（recursions）累加，退出时递减，减为 0 时真正释放锁。
