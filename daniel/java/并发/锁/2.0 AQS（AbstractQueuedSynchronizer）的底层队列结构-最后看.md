AQS（AbstractQueuedSynchronizer）可以说是整个 Java 并发包（JUC）的“定海神针”和绝对基石。像我们日常高频使用的 `ReentrantLock`、`CountDownLatch`、`Semaphore`，底层全都是直接套用了 AQS 的这套排队与状态管理机制。

要彻底搞透 AQS，我们就得扒开它的源码，看看它底层用来管理阻塞线程的那个**同步队列**到底长什么样。

AQS 的底层架构其实非常精炼，核心就是**“一个状态变量 + 一个双向链表”**。我为您从以下四个维度详细拆解这个底层的队列结构：

### 1. 队列的拓扑结构：变种的 CLH 双向链表

AQS 内部维护了一个 FIFO（先进先出）的队列。这个队列并不是 Java 里的普通集合，而是一个基于指针关联的**双向链表**（底层实现是对 CLH 锁队列的变种）。

- **头尾指针：** AQS 内部有两个核心的指针，`head` 指向队列的头部，`tail` 指向队列的尾部。
    
- **虚拟头节点（Dummy Node）：** 这是 AQS 队列设计里极其巧妙的一点。队列的 `head` 节点实际上是一个“空节点”（或者说是当前已经拿到锁的那个线程的占位符）。真正的排队是从 `head.next`（也就是第二个节点）开始的。这种设计极大地简化了头节点释放锁时唤醒后继节点的操作边界判断。
    

### 2. 队列的组成元素：Node 节点

当一个线程去抢锁失败时，AQS 不会任由这个线程自生自灭，而是会把这个线程包装成一个 `Node` 对象，塞进队列里。这个 `Node` 类是 AQS 内部的一个核心静态内部类，它包含以下几个关键属性：

- **`thread`：** 里面真枪实弹地封装了当前被阻塞的那个线程对象。
    
- **`prev` / `next`：** 分别指向前驱节点和后继节点，由此构成了双向链表。
    
- **`waitStatus`（等待状态）：** 这是一个极其关键的 `int` 变量，它决定了当前节点在队列里的“命运”。常见的状态有：
    
    - **`0`：** 初始状态，节点刚被扔进队列时默认就是 0。
        
    - **`SIGNAL (-1)`：** 这个状态最重要！它表示**“我的后继节点已经（或即将）被阻塞了，所以当前节点释放锁的时候，有义务去唤醒（unpark）我的后继节点”**。
        
    - **`CANCELLED (1)`：** 表示这个节点对应的线程因为超时或者被中断，已经放弃抢锁了，成了废节点，稍后会被踢出队列。
        

### 3. 核心状态基准：`volatile int state`

虽然您问的是队列结构，但脱离了 `state` 谈队列是没有灵魂的。

AQS 内部有一个用 `volatile` 修饰的 `int state` 变量。

- 对于 `ReentrantLock` 来说，`state = 0` 表示锁空闲；`state > 0` 表示锁被占用了（如果是 2，表示重入了两次）。
    
- 所有的线程在入队之前，第一件事就是试图通过 CAS（Compare-And-Swap）操作去把 `state` 从 0 改为 1。只有 CAS 失败的倒霉蛋，才会被打包成 `Node` 扔进底层队列。
    

### 4. 队列的运作机制（入队与出队）

- **入队（Enqueuing）：** 当线程抢锁（修改 `state`）失败后，AQS 会创建一个 `Node`。为了保证在高并发下入队不丢失节点，AQS 会在一个**死循环（自旋）里使用 CAS 操作**，不断尝试将自己挂到 `tail` 节点的后面（也就是更新 `tail` 指针），直到挂载成功为止。挂载成功后，该线程会调用 `LockSupport.park()` 将自己真正挂起（阻塞入内核态）。
    
- **出队（Dequeuing）：** 当占有锁的线程执行完业务逻辑，调用 `unlock()` 时，它会把 `state` 还原为 0。然后，它会去看一眼自己对应的那个 `head` 节点，如果发现后继节点需要唤醒（`waitStatus == SIGNAL`），它就会调用 `LockSupport.unpark()` 唤醒 `head.next` 里的那个线程。被唤醒的线程会再次尝试去 CAS 抢锁，抢到了，它就会把自己设为新的 `head`（虚拟节点），旧的 `head` 出队被 GC 回收。
    

---

### 总结

概括来说，AQS 的底层就是一个**通过 CAS 保证线程安全入队的、由 `Node` 组成的 FIFO 双向链表**。它巧妙地利用 `state` 管理锁的归属，利用双向链表管理排队的线程，利用 `LockSupport` 控制线程的休眠与唤醒。
