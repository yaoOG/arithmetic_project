`synchronized` 和 `ReentrantLock` 可以说是 Java 并发编程中最核心的两种加锁机制，也是我们在做底层架构选型时经常需要权衡的两个组件。

如果用一句话来概括它们的区别：**`synchronized` 是 JVM 层面的“傻瓜式”内置锁，简单省心；而 `ReentrantLock` 是 API 层面的“专业级”显式锁，功能强大但需要小心驾驭。**

为了让您一目了然，我先用一张表格对比它们的核心差异，然后再为您详细拆解几个关键维度：

| **比较维度** | **synchronized**                    | **ReentrantLock**                         |
| -------- | ----------------------------------- | ----------------------------------------- |
| **实现层次** | **JVM 层面**（底层依赖 Monitor 指令）         | **API 层面**（基于 JUC 包的 AQS 实现）              |
| **释放机制** | **自动释放**（发生异常或执行完毕自动解绑）             | **手动释放**（必须显式在 `finally` 块中调用 `unlock()`） |
| **公平性**  | **只能是非公平锁**                         | **支持公平锁和非公平锁**（默认非公平）                     |
| **中断响应** | **不可中断**（死等，容易造成死锁）                 | **可响应中断**（`lockInterruptibly()`）          |
| **超时尝试** | **不支持**（只能阻塞等待）                     | **支持**（`tryLock(time, unit)`，拿不到可以撤）      |
| **条件变量** | **单一条件**（关联一个 `wait()/notify()` 队列） | **多条件**（可绑定多个 `Condition` 对象，实现精准唤醒）      |

下面我为您详细拆解这几个决定了我们在生产环境中“用哪个”的核心痛点差异：

### 1. 灵活性与容错能力（API 层面的碾压）

- **`synchronized` 的死板：** 当一个线程去抢 `synchronized` 锁时，如果锁被别人占了，它就会进入阻塞状态，**一直死等下去**。你无法打断它，也无法让它等两秒钟就放弃。这在遇到外部资源夯死时，极易导致整个系统的线程池被耗尽。
    
- **`ReentrantLock` 的破局：** 它提供了极其丰富的获取锁的姿势。
    
    - 我们可以用 `tryLock()` 尝试拿一下，拿不到直接走人去干别的业务（非阻塞）。
        
    - 也可以用 `tryLock(5, TimeUnit.SECONDS)` 设置一个超时时间，拿不到就抛异常降级。
        
    - 还可以用 `lockInterruptibly()`，在等待锁的过程中如果收到中断信号（interrupt），可以直接醒来处理异常，从而有效打破死锁。
        

### 2. 精准唤醒（Condition 机制）

- **`synchronized` 的“大锅饭”：** 它配合 `Object.wait()` 和 `notifyAll()` 使用时，所有的线程都在同一个等待队列里。一旦调用 `notifyAll()`，会把所有等待的线程全部唤醒，去竞争锁，这会造成大量的无效上下文切换（惊群效应）。
    
- **`ReentrantLock` 的“精准打击”：** 它可以调用 `newCondition()` 创建多个条件变量。比如在实现阻塞队列（ArrayBlockingQueue）时，我们可以创建一个 `notFull` 条件和一个 `notEmpty` 条件。当队列满了，生产者在 `notFull` 上等待；当消费者消费了一个元素，它可以**明确指定只唤醒在 `notFull` 上等待的生产者**，而不干扰其他等待的消费者。这种控制粒度是 `synchronized` 无法企及的。
    

### 3. 公平与非公平的选择

- `synchronized` 骨子里就是非公平的，新来的线程有机会直接插队抢锁。
    
- `ReentrantLock` 默认也是非公平的（为了高吞吐量），但如果你在开发一些对执行顺序极其敏感的调度系统，你可以通过构造函数 `new ReentrantLock(true)` 将其切换为**公平锁**，保证先排队的线程先拿到锁，彻底杜绝线程“饿死”的现象。
    

### 4. 致命的踩坑点：锁的释放

- 使用 `synchronized` 时，JVM 像个尽职的保姆，无论是正常执行完毕，还是代码中途抛出了 `RuntimeException`，JVM 都会保证把锁自动释放掉。
    
- 但是使用 `ReentrantLock` 时，**必须、绝对、一定要把 `unlock()` 写在 `finally` 块中**。如果忘记写，或者写在了可能抛出异常的代码之前，一旦发生异常，这把锁将永远不会被释放，导致极其严重的线上死锁灾难。
    

Java

```
// ReentrantLock 的标准安全写法
Lock lock = new ReentrantLock();
lock.lock(); 
try {
    // 业务逻辑
} finally {
    lock.unlock(); // 绝不能忘！
}
```

### 总结与选型建议

在早期的 JDK 版本中，`ReentrantLock` 的性能远超 `synchronized`。但自 JDK 1.6 引入了“偏向锁、轻量级锁”等优化后，两者在日常并发场景下的性能已经相差无几。

**我们的选型原则通常是：** 优先使用 `synchronized`，因为它代码更简洁，且不易出错；只有当业务场景真正需要高级特性（如可中断、超时、公平锁、多条件唤醒）时，才果断换用 `ReentrantLock`。
