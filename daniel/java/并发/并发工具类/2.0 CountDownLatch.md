
`CountDownLatch` 是 JUC 并发包里非常高频的一个同步工具类。如果用一句话来概括，它就是一个**“并发倒计时器”**。

作为一名有经验的开发者，我看待 `CountDownLatch` 通常会从**业务场景**、**底层 AQS 实现**以及**生产避坑**这三个维度来拆解。

我们可以直接从这三个方面深入探讨：

### 1. 核心实战场景 (我平时怎么用它)

在实际的业务开发中，`CountDownLatch` 主要应对两种经典的并发协同场景：

- **场景一：“一等多”（最常用，数据并行聚合）**
    
    - **业务诉求**：比如一个电商的 App 首页接口，需要同时拉取“用户信息”、“推荐商品”、“未读消息”和“优惠券”。如果串行调用 RPC，耗时太长（假设每个 100ms，总耗时 400ms）。
        
    - **用法**：主线程创建一个 `new CountDownLatch(4)`。开启 4 个异步线程并行去拉取数据，主线程调用 `await()` 阻塞等待。每个子线程拉取完毕后执行 `countDown()`。当 4 个任务全部完成，主线程被唤醒，拼接数据返回（总耗时降至 100ms 左右）。


示例

```java
import java.util.concurrent.*;

public class CountDownLatchOneWaitsMulti {

    // 生产环境中，必须使用自定义的线程池，避免资源耗尽
    private static final ExecutorService EXECUTOR = Executors.newFixedThreadPool(4);

    public static void main(String[] args) {
        long startTime = System.currentTimeMillis();
        
        // 1. 初始化倒计时器，总数为 4（对应 4 个前置依赖任务）
        CountDownLatch latch = new CountDownLatch(4);

        // 2. 提交 4 个异步任务
        submitTask("拉取用户信息", 100, latch);
        submitTask("拉取推荐商品", 120, latch);
        submitTask("拉取未读消息", 80, latch);
        submitTask("拉取优惠券", 150, latch);

        try {
            // 3. 主线程阻塞等待
            // 【老兵避坑】：绝对不要用 latch.await() 死等！如果某个子线程死锁，主线程将永远卡死。
            // 假设 App 首页接口 SLA 要求 200ms 返回，这里设置 200ms 超时兜底
            boolean isSuccess = latch.await(200, TimeUnit.MILLISECONDS);
            
            if (isSuccess) {
                System.out.println("所有数据组装完毕，返回给前端！");
            } else {
                System.err.println("警告：部分底层服务超时，触发降级逻辑，返回部分数据！");
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // 恢复中断状态
            System.err.println("主线程被意外中断");
        } finally {
            // 演示完毕，关闭线程池
            EXECUTOR.shutdown();
            System.out.println("接口总耗时: " + (System.currentTimeMillis() - startTime) + "ms");
        }
    }

    // 模拟提交异步任务的方法
    private static void submitTask(String taskName, long costTime, CountDownLatch latch) {
        EXECUTOR.execute(() -> {
            try {
                // 模拟业务耗时
                Thread.sleep(costTime);
                System.out.println(Thread.currentThread().getName() + " -> " + taskName + " 完成");
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                // 【核心红线】：必须在 finally 中扣减，确保即使业务抛出异常，计数器也能正常减 1
                latch.countDown();
            }
        });
    }
}
```



- **场景二：“多等一”（模拟高并发发车）**
    
    - **业务诉求**：我们在做系统压测时，希望 1000 个线程能在同一绝对时刻瞬间发起请求，造成并发洪峰。
        
    - **用法**：创建一个 `new CountDownLatch(1)`。1000 个测试线程启动后全部调用 `await()` 在起跑线死等。主线程准备就绪后，调用 `countDown()`，这就好比裁判鸣枪，1000 个线程瞬间同时释放。


示例：

```Java
import java.util.concurrent.*;

public class CountDownLatchMultiWaitsOne {

    public static void main(String[] args) throws InterruptedException {
        int threadCount = 1000; // 模拟 1000 个并发用户
        
        // 1. 裁判发令枪（计数器为 1）
        CountDownLatch startGun = new CountDownLatch(1);
        
        // 2. 为了演示主线程知道所有人都跑完了，我们再加一个终点线计时器
        CountDownLatch endLine = new CountDownLatch(threadCount);

        // 使用线程池创建 1000 个并发任务
        ExecutorService executor = Executors.newFixedThreadPool(thread00);

        System.out.println("选手正在入场，准备就绪中...");
        
        for (int i = 0; i < threadCount; i++) {
            final int userNo = i + 1;
            executor.execute(() -> {
                try {
                    // 【核心逻辑】：所有子线程在这里死等，等待发令枪 count == 0
                    startGun.await();
                    
                    // 以下是瞬间爆发的业务逻辑（比如：发起 HTTP 请求抢茅台）
                    // System.out.println("用户 " + userNo + " 发起秒杀请求！");
                    
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                } finally {
                    // 跑完后，终点线计数器减 1
                    endLine.countDown();
                }
            });
        }

        // 模拟主线程做一些压测前的准备工作（比如初始化连接池、加载数据等）
        Thread.sleep(2000);
        System.out.println("准备工作完毕。3, 2, 1... 砰！发车！");
        
        long startTime = System.currentTimeMillis();
        
        // 3. 裁判鸣枪！将 count 减为 0，瞬间唤醒阻塞队列里的 1000 个线程
        startGun.countDown();

        // 4. 等待所有人跑完终点线
        endLine.await();
        System.out.println("压测结束，1000个线程全部执行完毕！总耗时: " + (System.currentTimeMillis() - startTime) + "ms");
        
        executor.shutdown();
    }
}
```

### 2. 底层原理：它是如何利用 AQS 的？

作为一个工作多年的后端，如果只停留在 API 调用肯定是不够的。`CountDownLatch` 的底层其实是对 **AQS（AbstractQueuedSynchronizer）共享锁机制**的一种极其经典的运用。

- **初始化 (`new CountDownLatch(N)`)**：
    
    底层实际上是把 AQS 的核心同步状态变量 `state` 的值设置为了 `N`。
    
- **`countDown()` 做了什么？**
    
    每次调用 `countDown()`，底层就会利用 **CAS** 操作将 AQS 的 `state` 减 1。
    
    如果减 1 之后 `state` 变成了 0，它就会调用 AQS 底层的 `doReleaseShared()` 方法，去唤醒同步队列中所有处于阻塞状态的等待线程。
    
- **`await()` 做了什么？**
    
    主线程调用 `await()` 时，底层会去检查 `state` 的值。如果 `state != 0`，说明倒计时还没结束，主线程就会被包装成一个 Node 节点，加入到 AQS 的阻塞队列中，并调用 `LockSupport.park()` 将自己挂起休眠。如果 `state == 0`，则直接放行。
    

### 3. 生产环境的避坑与架构演进（老兵经验）

在实际的 Code Review 中，我经常会揪出新同学使用 `CountDownLatch` 时的一个致命隐患，以及推动一种更现代的替代方案：

- **致命隐患：未在 `finally` 中释放**
    
    如果在子线程的业务逻辑中发生了 `NullPointerException` 等未捕获异常，导致线程中断，没有执行到最后的 `countDown()`。那么 AQS 的 `state` 将永远无法清零，**导致调用 `await()` 的主线程发生死锁，永久阻塞**。
    
    **规范**：`countDown()` **必须、绝对**要放在子线程的 `try-finally` 块中的 `finally` 里执行！同时，主线程强烈建议使用带有超时的 `await(long timeout, TimeUnit unit)` 进行兜底。
    
- **架构演进：被 `CompletableFuture` 取代**
    
    在 Java 8 之后，如果是针对复杂的“任务编排”和“结果聚合”（就是上面的场景一），我目前在团队里**已经不推荐使用 `CountDownLatch` 了**。因为它只负责“等”，不负责“结果的合并”和“异常处理”。
    
    我会强制要求团队使用 `CompletableFuture.allOf().join()`。`CompletableFuture` 不仅能实现相同的并发等待效果，还能直接进行链式调用、极其优雅地处理异步返回结果和进行全局异常捕获。
