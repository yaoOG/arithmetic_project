如果说 `put` 操作体现了细粒度锁的艺术，那么 `transfer`（扩容机制）则将**“多线程协同”**发挥到了极致。在传统的 `HashMap` 中，扩容是一个单线程的、极度消耗 CPU 且容易引起卡顿的“重量级”操作；但在 CHM 中，Doug Lea 大神把它变成了一场**“全民参与的接力赛”**。

作为有经验的开发者，我通常会从**触发时机、任务分配、数据迁移、以及扩容时的读写处理**这四个关键维度来向团队剖析这个机制：

扩容时，正在迁移的槽位会被放置一个特殊的节点 ***ForwardingNode（hash 值为 MOVED）***。

### 1. 触发扩容的时机

并不是只有元素塞满了才会扩容，CHM 触发 `transfer` 主要有两个场景：

- **常规扩容**：在 `addCount` 增加元素个数后，发现当前元素总数达到了阈值（`sizeCtl`，即数组长度的 0.75 倍）。
    
- **代替树化**：在 `put` 操作中，当某个桶的链表长度达到 8，但**整个数组的长度还小于 64** 时，系统会优先选择扩容，而不是转红黑树（调用 `tryPresize`）。
    

---

### 2. 核心思想：步长（Stride）与任务认领

CHM 扩容最牛的地方在于，它允许**正在进行 `put` 操作的其他线程主动过来帮忙**。为了让多个线程不打架，它引入了“分段迁移”的概念。

- **计算步长（stride）**：系统会根据 CPU 核心数和数组总长度，计算出一个步长。规定每个线程每次最少领走 **16 个桶** 的迁移任务。
    
- **认领任务（transferIndex）**：有一个全局的游标 `transferIndex`，初始指向旧数组的末尾。线程进来帮忙时，会通过 **CAS** 操作把 `transferIndex` 减去步长，从而“圈地”成功，领走属于自己的那段区间（从右往左迁移）。
    

---

### 3. 数据迁移的底层流转（核心逻辑）

当一个线程领到自己的 16 个桶后，它会挨个遍历这些桶，执行真正的搬运动作：

1. **加锁**：使用 `synchronized` 锁住当前要迁移的桶的**头节点**，防止在这零点几毫秒内有新数据写进来。
    
2. **高低位拆分（极致的 CPU 优化）**： 它不会去重新计算每个元素的 Hash 值！而是巧妙地利用了 `hash & n`（n 为旧数组长度）进行按位与运算。
    
    - 结果为 `0` 的节点，组成一条**低位链表**（留在新数组的原索引位置 `i`）。
        
    - 结果不为 `0` 的节点，组成一条**高位链表**（搬运到新数组的 `i + n` 位置）。
        
    - 这就瞬间把一个长链表拆成了两个短链表，直接塞进新数组（`nextTable`）。
        
3. **放置“占位符”（ForwardingNode）**：
    
    这是最关键的一步！当当前桶迁移完毕后，线程会在旧数组的这个位置放上一个特殊的节点：`ForwardingNode`（它的 hash 值固定为 `-1`，也就是常说的 `MOVED` 状态）。它的内部包含一个指向新数组 `nextTable` 的指针。
    

---

### 4. 扩容期间，发生并发读写怎么办？

这是业务开发中最关心的问题：扩容会阻塞线上请求吗？答案是**几乎不阻塞**。

- **并发读（`get`）**：
    
    - 如果请求打到了还没迁移的桶，直接在旧数组里读，正常返回。
        
    - 如果请求打到了已经迁移完的桶，会碰到 `ForwardingNode`。这个节点重写了 `find` 方法，它会**顺着内部的指针去新数组（nextTable）里帮你把数据找出来**。**全程无锁，读操作丝滑顺畅！**
        
- **并发写（`put`）**：
    
    - 如果打到了还没迁移的桶，正常加锁写入旧数组（稍后迁移线程会把它搬走）。
        
    - 如果打到了已经迁移完的桶，碰到了 `ForwardingNode`（发现 hash 为 -1），**该线程立马停下手头的写操作，调用 `helpTransfer` 加入扩容大军**，领一块任务去搬数据。搬完之后，再进行自己的 `put` 操作。
        

---

### 总结

CHM 的扩容机制，本质上是用**状态机（ForwardingNode）**保证了数据的一致性，用**CAS（transferIndex）**实现了多线程的任务调度，用**高低位算法**榨干了 CPU 的计算性能。
