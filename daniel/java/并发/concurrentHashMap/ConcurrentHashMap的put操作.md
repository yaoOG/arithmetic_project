在 JDK 1.8 中，`put` 操作抛弃了 1.7 的分段锁，转而采用 **CAS + synchronized + 细粒度锁** 的设计。其核心逻辑被封装在 `putVal` 方法中。

为了讲透这个过程，我将其拆解为 **“六步演进法”**：

---

### 1. 第一步：哈希计算与非法校验

首先，CHM 不允许 `key` 或 `value` 为 `null`（这与 `HashMap` 不同，主要是为了避免在多线程环境下产生歧义，如 `get(key)` 返回 `null` 无法分辨是值真的为 `null` 还是没有这个 key）。

- **扰动函数**：通过 `(h ^ (h >>> 16)) & HASH_BITS` 计算 hash 值。
    
- **强制正数**：`HASH_BITS` 是 `0x7fffffff`，确保计算出的 hash 值永远为正数。因为在 CHM 中，负数 hash 有特殊含义（如 `-1` 代表扩容中，`-2` 代表红黑树节点）。
    

### 2. 第二步：自旋初始化 (`initTable`)

如果发现底层数组 `table` 尚未初始化，线程会进入一个 `for(;;)` 死循环（自旋）。

- **CAS 抢占**：线程会尝试利用 CAS 将变量 `sizeCtl` 设置为 `-1`。
    
- **成功者初始化**：CAS 成功的线程负责创建数组；失败的线程则调用 `Thread.yield()` 让出 CPU，等待初始化完成。这比直接加全局锁要轻量得多。
    

### 3. 第三步：无锁写入（槽位为空）

当根据 hash 定位到数组索引 `i`，且发现该槽位 `tab[i] == null` 时：

- **CAS 插入**：尝试通过 `casTabAt`（底层 `Unsafe.compareAndSwapObject`）将新节点直接放入该槽位。
    
- **零锁开销**：如果成功，说明没有冲突，整个 `put` 过程**完全没有加锁**，性能极高。
    

### 4. 第四步：协助扩容 (`helpTransfer`)

如果在自旋中发现槽位节点的 hash 值为 `MOVED`（即 `-1`）：

- **身份识别**：说明当前数组正在进行扩容，且该槽位的数据已经迁移到了新数组，留下了一个 `ForwardingNode`。
    
- **主动帮忙**：当前线程不会阻塞，而是立刻调用 `helpTransfer` 加入扩容大军，协助搬运数据。这种“全民皆兵”的设计极大缩短了扩容停顿时间。
    

### 5. 第五步：细粒度加锁（槽位冲突）

如果槽位不为空，且没有在扩容，说明发生了 Hash 冲突：

- **锁住头节点**：此时使用 `synchronized` 锁住当前槽位的**头节点（First Node）**。
    
- **逻辑处理**：
    
    - **链表**：遍历链表，若 key 相同则更新 `value`，若不同则在尾部追加。
        
    - **红黑树**：若节点类型是 `TreeBin`，则按照红黑树的规则插入。
        
- **锁粒度**：只锁住当前的“桶”，其他 99% 不冲突的桶依然可以由其他线程并发读写。
    

### 6. 第六步：树化与计数

- **Treeify**：插入完成后，若链表长度达到 8 且数组长度达到 64，将链表转为红黑树。
    
- **高性能计数 (`addCount`)**：更新 `size`。CHM 不使用全局 `AtomicInteger`，而是采用类似 `LongAdder` 的分散累加思想（`baseCount` + `CounterCell[]`），有效规避了高并发下的 CAS 热点竞争。
    

---

### 总结：put 操作的性能三阶梯

|**场景**|**技术手段**|**性能开销**|
|---|---|---|
|**无竞争/初始化**|CAS|极低|
|**桶位为空**|CAS|极低|
|**桶位冲突**|synchronized (头节点)|低 (仅阻塞当前桶)|
|**正在扩容**|多线程协同迁移|高效 (利用闲置 CPU)|
