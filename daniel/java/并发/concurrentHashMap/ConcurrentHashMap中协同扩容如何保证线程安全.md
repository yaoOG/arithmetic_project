这是一个非常深入且精彩的问题。ConcurrentHashMap 在扩容时的“多线程协同”机制（Multi-threaded Transfer），是其设计中最复杂、也是最精妙的部分。

简单来说，保证线程安全的核心在于：**分工（任务分配）是基于 CAS 的，干活（数据迁移）是基于 synchronized 的，标记（完成状态）是基于 ForwardingNode 的。**

我们可以把这个过程想象成**“搬砖”**：旧数组是旧工地，新数组是新工地，多个线程就是多个搬砖工。

以下是具体的线程安全保障机制：

**1. 任务分配的安全性：CAS + transferIndex**

首先，怎么保证两个线程不会抢同一个位置的数据去迁移？

- **全局游标 transferIndex**： Map 内部维护了一个 volatile 变量 transferIndex，初始值为旧数组的长度。它代表着“还没被认领的槽位范围的边界”。
- **步长 (Stride)**： 为了避免频繁竞争，线程不是一个一个槽位地抢，而是按“块”来抢。每个线程一次认领一个**步长**（默认是 16 个槽位）。
- **CAS 抢任务**： 当一个线程加入扩容大军时，它会尝试修改 transferIndex。 **操作：** CAS(transferIndex, oldVal, oldVal - stride)

- **成功**：代表该线程抢到了从 oldVal - stride 到 oldVal 这段范围的任务。它拥有了这段范围的“独家搬运权”。
- **失败**：说明被别人抢先了，重新读取 transferIndex，再次计算并尝试 CAS。

**结论：** 通过 CAS 原子地修改 transferIndex，将整个数组切分成互不重叠的段（Bucket Ranges），实现了**任务分配层面的线程安全**。

**2. 数据迁移的安全性：synchronized**

当线程领到了任务（比如下标 16-31），它需要逐个迁移这些槽位的数据。具体的迁移过程如何保证安全？

- **锁住槽位头节点**： 在迁移具体的某一个槽位（Bucket）时，线程会使用 **synchronized 锁住该槽位的头节点**。
- **互斥性**： 这意味着，当线程 A 正在迁移下标为 i 的槽位时：

- 线程 B 想往这个位置 put 数据？会被阻塞（因为要获取同一把锁）。
- 线程 C 也领到了任务，但它处理的是下标 j 的槽位？互不干扰，并行执行。

- **迁移逻辑**： 持有锁之后，线程构建两个链表（高位链和低位链），分别对应新数组的两个位置，然后将它们挂到新数组 nextTable 上。

**结论：** 利用 synchronized 实现了**单个槽位内部操作的原子性**，防止了迁移过程中数据被并发修改。

**3. 完成状态的可见性：ForwardingNode**

当一个槽位迁移完成后，怎么告诉其他线程“这里已经搬空了，去新家找”？

- **放置标记节点**： 原数组的槽位被处理完后，会被放置一个特殊的节点 —— **ForwardingNode** (hash 值为 MOVED, -1)。
- **作用**：

1. **读操作可见**：如果其他线程来 get 数据，发现头节点是 ForwardingNode，它会通过该节点内部的指针直接去新数组 (nextTable) 查找。
2. **写操作协作**：如果其他线程来 put 数据，发现头节点是 ForwardingNode，说明该位置正在或已经扩容。它意识到“正在扩容”，于是不仅不等待，反而通过 helpTransfer() 方法加入扩容大军（跳转到第 1 步去抢任务）。

**4. 全局状态管理：sizeCtl**

如何知道所有人都搬完了？

- sizeCtl 在扩容期间被用作一个信号量。
- 高 16 位存储扩容标识，低 16 位存储**参与扩容的线程数 + 1**。
- 每当一个线程加入扩容，对 sizeCtl 低位加 1（CAS）。
- 每当一个线程完成了它的所有任务并退出，对 sizeCtl 低位减 1。
- 当最后一个线程（减到初始值）完成时，它知道自己是最后一个人，负责将 nextTable 赋值给 table，扩容正式结束。

**总结：线程安全流程图**

1. **抢地盘**：线程通过 **CAS** 修改 transferIndex，划定自己的工作区（互斥，不冲突）。
2. **搬数据**：线程用 **synchronized** 锁住当前处理的桶（防止此时有 put 操作干扰）。
3. **插旗子**：搬完一个桶，在该位置留下 **ForwardingNode**（volatile 可见性），告知其他线程“此路已迁”。
4. **大家帮**：其他线程看到 ForwardingNode，不阻塞，转而通过 helpTransfer 加入上述第 1 步。

这就是 JDK 1.8 在高并发下实现**无阻塞、协作式扩容**的奥秘。
